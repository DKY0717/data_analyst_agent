# 第23章 测试金字塔与契约测试
> 本章预计 1～2 小时，学习如何用不同层级测试证明 Agent 工程质量。
## 23.1 学习目标
> 能为纯函数、节点、API、前端契约和真实模型评测选择合适测试层级。
## 23.2 前置知识
> 已走完整条产品链路，并会使用 pytest 与 npm build。
## 23.3 为什么需要这一模块
> LLM 输出不稳定且端到端测试昂贵；大量确定性问题应由更快、更局部的测试捕获。
## 23.4 输入、输出与依赖
> 输入是代码契约、样例与替身；输出是单元、集成、契约、构建和评测证据。
## 23.5 执行流程
```text
unit → node/integration → API contract → frontend build → real-model evaluation
```
## 23.6 当前代码地图
| 内容 | 路径 |
|---|---|
| 后端测试 | `backend/tests/` |
| CI | `.github/workflows/ci.yml` |
| 真实模型工作流 | `.github/workflows/real-qwen-evaluation.yml` |
## 23.7 关键代码理解
> 可重复的安全和解析逻辑优先用单元测试；跨模块边界用集成与契约测试；只有模型能力问题才消耗真实 API 配额。
## 23.8 最小动手运行
```bash
pytest backend/tests/test_sql_guard.py backend/tests/test_query_api.py -q
cd frontend
npm run build
```
## 23.9 故障注入实验
> 改变一份模拟响应字段，观察契约测试如何比人工打开页面更早发现漂移。
## 23.10 调试路径与常见误判
> 测试失败先判断是产品缺陷、测试假设过期、环境缺失还是模型波动，不要把所有红灯都归因于 LLM。
## 23.11 独立编码练习
> 为一个未覆盖的纯函数补正常、边界和错误三类用例。
## 23.12 测试或评测验证
> 用测试收集报告确认新增测试确实被发现，并保持测试之间无顺序依赖。
## 23.13 面试复述题
> 真实模型端到端测试为什么不能替代单元测试？
## 23.14 掌握度检查与下一章
