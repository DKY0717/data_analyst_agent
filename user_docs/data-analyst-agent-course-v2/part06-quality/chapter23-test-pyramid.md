# 第23章 测试金字塔与契约测试

> 本章预计 1～2 小时。学完后，你不只是会运行 `pytest`，还应能回答：某个缺陷应该由哪一层测试发现，为什么真实模型评测不能替代普通自动化测试。

## 23.1 学习目标

> 完成本章后，你应该能够：
>
> - 区分单元测试、模块集成测试、API 契约测试、前端组件测试、浏览器 E2E 和真实模型评测；
> - 为确定性逻辑选择快速、稳定的测试，而不是一律调用 LLM；
> - 从一次失败判断问题属于产品代码、测试替身、运行环境还是外部模型；
> - 用“测试对象、输入、断言、证据边界”四句话解释一个测试。

## 23.2 前置知识

> 你已经走过同步查询、SSE、Vue 页面和部署链路，并能看懂最基本的 `pytest` 与 Vitest 用例。不了解 mock 没关系，本章会把它理解成“受控替身”：用固定行为替代真实数据库、网络或模型。

## 23.3 为什么需要这一模块

> Agent 系统同时包含两类行为。SQL Guard、权限判断、结果比较等是确定性的；模型生成内容则可能受供应商、模型版本、网络和采样影响。如果所有验证都依赖真实模型，失败时就很难知道究竟是代码回归还是外部波动。
>
> 因此测试体系不是“测试越大越好”，而是从便宜、局部、可重复的证据逐层向外扩展。底层测试负责快速定位，上层测试负责证明模块能够协作，真实模型评测只回答模型能力与整条真实链路的问题。

## 23.4 输入、输出与依赖

| 层级 | 主要输入 | 核心输出 | 本项目中的典型依赖 |
|---|---|---|---|
| 单元测试 | 一个函数、固定参数 | 精确返回值或异常 | pytest，无网络 |
| 节点/模块集成 | 多个真实模块加少量替身 | 状态字段与分支结果 | 临时数据库、Fake LLM |
| API 契约 | HTTP 请求 | 状态码、JSON/SSE 结构 | FastAPI TestClient |
| 前端组件/状态 | 组件、store、模拟响应 | DOM、状态迁移、导出行为 | Vitest |
| 浏览器 E2E | 启动后的前后端 | 用户关键路径 | Playwright、服务进程 |
| 专项评测 | 版本化 case pack | 逐例证据与指标 | evaluator、固定数据 |
| 真实模型评测 | 真实 API、固定代码版本 | 特定供应商/模型证据 | GitHub Actions、密钥、网络 |

> 每层都依赖上一层，但不能替代上一层。E2E 通过只能说明选定路径在该环境可用，不能证明 SQL Guard 的所有边界；真实模型分数高也不能证明前端断线处理正确。

## 23.5 执行流程

```text
需求或缺陷
  → 找到最小可观察契约
  → 先写确定性单元/模块测试
  → 补跨边界的 API 或组件测试
  → 对关键用户路径运行浏览器 E2E
  → 对模型能力运行专项评测
  → 把代码 SHA、case 数、环境和结果一起保存
```

> 例如“危险删除请求必须被拦截”首先属于 Intent Guard 单元测试；“阻断后 API 返回什么业务状态”属于 API 契约测试；“页面如何展示阻断原因”属于组件或 E2E；“真实模型是否诱导出危险 SQL”才属于真实模型评测。

## 23.7 关键代码理解

> 第一原则是只 mock 不稳定边界，不 mock 被测试对象本身。测试 SQL Guard 时直接调用真实 Guard；测试 Agent 节点时可以把 LLM 换成固定输出；测试 API 时通常保留真实路由与响应模型，只替换耗时或外部依赖。
>
> 第二原则是断言业务契约，而不是实现偶然细节。比起断言某个私有函数被调用三次，更有价值的是断言危险请求没有进入 LLM/数据库、权限拒绝携带稳定规则 ID、SSE 最终事件遵守约定结构。
>
> 第三原则是失败隔离。批量评测中一个 case 异常要转换为该 case 的稳定失败，后续 case 继续运行；单元测试则应尽量让异常直接暴露，以便快速看到堆栈。两种策略服务的目标不同。
>
> 第四原则是证据不能越级。`npm run build` 证明前端可以生产编译，不证明交互正确；Vitest 证明受控输入下的组件行为，不证明浏览器与真实服务协作；Playwright E2E 证明关键路径，不证明真实 LLM 的准确率。

### 一个实用的选层方法

> 遇到需求时连续问四个问题：
>
> 1. 能否把规则写成纯输入输出？能，就从单元测试开始。
> 2. 是否跨越状态、数据库或节点边界？是，就补模块集成测试。
> 3. 客户端是否依赖响应字段或事件顺序？是，就补契约测试。
> 4. 风险是否只会在真实网络、浏览器或模型出现？是，才升级到 E2E 或真实评测。

## 23.8 最小动手运行

> 先在项目根目录运行两组后端测试，观察一个安全纯逻辑模块和一个 API 边界模块的速度与输出差异。

```powershell
pytest backend/tests/test_sql_guard.py backend/tests/test_query_api.py -q
```

> 再运行前端确定性测试与构建。这里使用 `--prefix frontend`，因此不需要先切换目录。

```powershell
npm run test --prefix frontend
npm run build --prefix frontend
```

> 如果本机已经安装 Playwright 浏览器和项目依赖，可再运行浏览器关键路径；它比前两组更慢，也更依赖端口和进程环境。

```powershell
npm run test:e2e --prefix frontend
```

## 23.9 故障注入实验

> 选择一个 API 或 store 测试使用的模拟响应，把其中一个稳定字段名临时改错。例如把 `execution_success` 改成不存在的名字，然后只运行对应测试。预期现象是契约测试立即失败，而不需要人工打开页面猜测。
>
> 实验完成后恢复改动。记录以下四项：失败测试名、首个有用断言、受影响契约、如果没有该测试用户会看到什么症状。

```text
故障：模拟响应字段漂移
最先报警层：API/前端契约测试
未受影响层：SQL Guard 单元测试
结论：字段契约问题不需要调用真实 LLM 才能发现
```

## 23.10 调试路径与常见误判

> 推荐按“最小失败 → 相关模块 → 更大链路”的顺序调试。先单独运行失败文件，再查看完整堆栈和第一个业务断言，最后才扩大到整个测试集。直接反复运行全量测试会让反馈变慢，也可能被后续连锁错误干扰。
>
> 常见误判一：看到 Fake LLM 就认为测试不真实。Fake LLM 的目标是固定模型边界，从而验证解析、状态流转和错误处理；模型能力由另一套真实评测负责。
>
> 常见误判二：E2E 通过等于项目正确。E2E 通常只覆盖少量主路径，而且可能没有覆盖危险 SQL、权限角色、浮点结果比较和超时分支。
>
> 常见误判三：真实模型评测失败就是代码 bug。还要核对 HTTP 状态、空 content、超时、provider/model、case 分片和代码 SHA。第 28 章会分析一次真实的长尾超时事故。
>
> 常见误判四：安全请求没执行就是普通执行失败。危险请求的正确结果本来就是被 Intent Guard 或 SQL Guard 阻断；它必须进入安全指标，而不是稀释安全查询执行成功率。

## 23.11 独立编码练习

> 找到一个尚未覆盖的确定性辅助函数，只新增测试，不修改产品行为。至少设计三类用例：正常输入、边界输入、畸形输入。每个测试名要体现条件和结果。

```python
def test_<对象>_<条件>_<预期结果>():
    # Arrange：准备最小输入
    # Act：只执行一个目标行为
    # Assert：验证公开契约
    ...
```

> 完成后口头解释：为什么这条测试属于当前层，为什么不需要数据库或真实模型，以及失败时首先检查哪个模块。

## 23.12 测试或评测验证

> 用收集模式确认测试真正被发现，而不是“命令成功但没有运行目标用例”。

```powershell
pytest backend/tests/test_sql_guard.py --collect-only -q
```

> 再运行课程自身的结构契约，理解文档也可以像 API 一样被验证：章节数量、固定栏目、路径、敏感信息和旧课程依赖都有自动检查。

```powershell
pytest backend/tests/test_learning_course_v2_docs.py -q
```

> 验收标准不是只有绿色退出码，还要确认收集数量非零、没有意外跳过、命令运行目录正确，并把 warning 与 failure 分开判断。

## 23.13 面试复述题

> **问题：为什么真实模型端到端评测不能替代单元测试？**
>
> 合格回答应包含：真实模型慢、贵、受外部波动影响；确定性规则需要快速精确定位；E2E 只覆盖有限路径；本项目把安全、权限、Grounding、Repair、结果正确性拆成不同证据层。
>
> **追问：你如何证明一次测试结果可复现？**
>
> 应回答：固定 case 与数据快照、记录样本数和分母、记录 provider/model 与 HEAD SHA、隔离外部依赖、保存逐例结果，而不是只展示一个汇总百分比。
