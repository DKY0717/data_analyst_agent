# 第1章 项目目标与完整架构

> 本章预计 1～2 小时，先建立全局地图，再进入局部代码。

## 1.1 学习目标
> 能区分 Data Analyst、NL2SQL 和受控 Agent，并画出用户问题到数据库再返回答案的路径。

## 1.2 前置知识
> 只要求理解 Python 函数、HTTP 请求和 SQL 查询的基本用途。

## 1.3 为什么需要这一模块
> 没有全局地图时，逐文件阅读会失去因果关系，也无法判断某次失败属于前端、Agent、LLM 还是数据库。

## 1.4 输入、输出与依赖
> 输入是自然语言问题和身份上下文；输出包含状态、SQL、结果、答案、优化建议与审计报告。系统依赖前端、FastAPI、AgentGraph、LLM 和数据库。

## 1.5 执行流程
```text
Vue → Query API → AgentGraph → Guard/权限 → DuckDB → Answer → Vue
```

## 1.6 当前代码地图
| 入口 | 路径 |
|---|---|
| 应用入口 | `backend/app/main.py` |
| 查询入口 | `backend/app/api/query.py` |
| 工作流 | `backend/app/agents/graph.py` |
| 前端 | `frontend/src/views/Home.vue` |

## 1.7 关键代码理解
> 先阅读对外入口和测试，再沿调用进入实现。重点观察 AgentGraph 如何把不确定的 LLM 节点放在确定性安全边界之间。

## 1.8 最小动手运行
```bash
pytest backend/tests/test_health.py -q
```
> 该命令无需网络或真实模型。

## 1.9 故障注入实验
> 把命令故意放到错误目录执行，记录导入失败，再回到项目根目录验证。不要修改源码掩盖工作目录问题。

## 1.10 调试路径与常见误判
> 首页可打开只证明静态页面可用；健康检查通过也不能证明数据库、LLM 和完整业务链路都可用。

## 1.11 独立编码练习
> 用自己的语言写一份不超过十行的模块职责清单，并标出哪些环节允许调用 LLM。

## 1.12 测试或评测验证
> 查看 `backend/tests/test_health.py` 的输入和断言，解释它能证明什么、不能证明什么。

## 1.13 面试复述题
> 为什么这个项目不是一个“把问题交给大模型写 SQL”的简单套壳？

## 1.14 掌握度检查与下一章
> 不看本章画出主链路，并能指出三层安全治理的大致位置。下一章学习阅读这些 Python 代码所需的语言能力。
