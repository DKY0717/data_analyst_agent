# 第9章 结构化分析意图
> 本章预计 1～2 小时，学习把自然语言转换为可合并的业务槽位。
## 9.1 学习目标
> 能解释指标、维度、筛选、排序、排名、置信度和缺失槽位。
## 9.2 前置知识
> 需要理解自然语言问题与 SQL 子句的基本对应关系。
## 9.3 为什么需要这一模块
> 直接从问题跳到 SQL 难以定位错误；结构化意图提供可检查的中间层。
## 9.4 输入、输出与依赖
> 输入是问题；输出是 AnalysisIntent；依赖规则解析、LLM 解析和合并器。
## 9.5 执行流程
```text
Question → Rule Parser + LLM Parser → Merger → AnalysisIntent
```
## 9.6 当前代码地图
| 模块 | 路径 |
|---|---|
| 意图包 | `backend/app/analysis_intent/` |
| 集成测试 | `backend/tests/test_analysis_intent_integration.py` |
## 9.7 关键代码理解
> 重点观察两个候选冲突时如何保留证据，以及低置信度为何不能被静默当成确定答案。
## 9.8 最小动手运行
```bash
pytest backend/tests/test_analysis_intent_integration.py -q
```
## 9.9 故障注入实验
> 输入“帮我分析一下”，观察缺失指标和维度，而不是强迫系统生成 SQL。
## 9.10 调试路径与常见误判
> 规则命中不等于业务口径已经 Grounding；LLM 候选也不是数据库字段事实。
## 9.11 独立编码练习
> 为“销售额最高的三个地区”手写期望意图对象。
## 9.12 测试或评测验证
> 对照 intent cases 检查槽位和缺失项，不调用真实模型。
## 9.13 面试复述题
> 为什么需要规则和 LLM 双路解析？
## 9.14 掌握度检查与下一章
> 能从问题提取主要槽位。下一章把业务槽位连接到语义和元数据。
