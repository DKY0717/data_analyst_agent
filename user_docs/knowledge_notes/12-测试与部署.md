## 4. 核心测试示例

### 4.1 SQL Guard 测试

> 测试 SQL 安全校验器是否正确拦截危险 SQL：

```python
# tests/test_sql_guard.py
from app.security.sql_guard import sql_guard

def test_block_drop_table():
    """DROP TABLE 应该被拦截"""
    result = sql_guard.validate("DROP TABLE orders")
    assert result["is_safe"] is False
    assert "禁止的语句类型" in result["reason"]

def test_allow_select():
    """SELECT 查询应该放行"""
    result = sql_guard.validate("SELECT * FROM orders")
    assert result["is_safe"] is True

def test_auto_inject_limit():
    """没有 LIMIT 的 SQL 应该自动注入"""
    result = sql_guard.validate("SELECT * FROM orders")
    assert "LIMIT" in result["sanitized_sql"]

def test_block_multi_statement():
    """多语句应该被拦截"""
    result = sql_guard.validate("SELECT 1; DROP TABLE orders")
    assert result["is_safe"] is False

def test_block_system_table():
    """系统表访问应该被拦截"""
    result = sql_guard.validate("SELECT * FROM information_schema.tables")
    assert result["is_safe"] is False
```

## 5. 测试数据管理

> 测试使用独立的数据库，避免影响开发数据：

```python
# 测试夹具（fixture）
@pytest.fixture
def test_db():
    """创建临时测试数据库"""
    conn = duckdb.connect(":memory:")  # 内存数据库，测试结束后自动销毁
    conn.execute(open("database/init.sql").read())
    seed_database(connection=conn)
    yield conn
    conn.close()
```

> 使用内存数据库（`:memory:`）的好处：
> - 测试速度快（不需要磁盘 I/O）
> - 每次测试都是全新的数据库
> - 测试结束后自动清理

## 6. Docker 部署

### 6.1 docker-compose.yml

```yaml
version: '3.8'

services:
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    ports:
      - "8000:8000"
    environment:
      - QWEN_API_KEY=${QWEN_API_KEY}
      - QWEN_MODEL=${QWEN_MODEL}
    volumes:
      - ./data:/app/data
      - ./logs:/app/logs

  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    ports:
      - "3000:80"
    depends_on:
      - backend
```

### 6.2 后端 Dockerfile

```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### 6.3 前端 Dockerfile

```dockerfile
FROM node:18-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
```

> 前端使用多阶段构建：先用 Node.js 构建静态文件，再用 Nginx 提供服务。

### 6.4 启动命令

```bash
# 复制环境变量
cp .env.example .env
# 编辑 .env 填入 QWEN_API_KEY

# 启动所有服务
docker-compose up -d

# 查看日志
docker-compose logs -f

# 停止服务
docker-compose down
```

## 7. 生产环境注意事项

### 7.1 安全配置

> 生产环境需要修改以下配置：

| 配置 | 开发环境 | 生产环境 |
|------|----------|----------|
| CORS | `allow_origins=["*"]` | 限制为具体域名 |
| DEBUG | `true` | `false` |
| LOG_LEVEL | `DEBUG` | `INFO` 或 `WARNING` |
| API Key | `.env` 文件 | 环境变量或密钥管理服务 |

### 7.2 性能优化

> - **数据库**：DuckDB 适合分析型查询，如果并发量大考虑 PostgreSQL
> - **LLM 调用**：API 调用有延迟，可以考虑缓存常见问题的结果
> - **前端**：Vite 构建后是静态文件，用 Nginx 或 CDN 提供服务

### 7.3 监控

> - 健康检查端点：`GET /health`
> - 日志文件：`logs/` 目录
> - 审计报告：每次查询都包含 `audit_report`，可以记录到日志系统

## 8. 常见问题

### 8.1 QWEN_API_KEY 未设置

```text
QWEN_API_KEY is not set. LLM calls will fail.
```

> 解决：在 `.env` 文件中填入正确的 API Key。

### 8.2 端口被占用

```text
Error: [Errno 10048] Only one usage of each socket address
```

> 解决：修改 `.env` 中的 `APP_PORT` 或关闭占用端口的程序。

### 8.3 DuckDB 文件锁定

```text
Could not open file: database.duckdb
```

> 解决：确保没有其他进程在使用数据库文件。DuckDB 同一时间只允许一个写连接。
