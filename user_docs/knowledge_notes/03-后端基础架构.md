## 2. FastAPI 应用入口

> 后端的入口文件是 `backend/app/main.py`，它做了三件事：
>
> 1. 创建 FastAPI 应用实例
> 2. 注册中间件（CORS 跨域）
> 3. 注册路由（API 接口）

### 2.1 应用创建与生命周期

```python
from fastapi import FastAPI
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app: FastAPI):
    """应用生命周期管理：启动时初始化，关闭时清理"""
    ensure_directories()  # 确保 data/ 和 logs/ 目录存在
    yield
    # 应用关闭时的清理逻辑

app = FastAPI(
    title="Data Analyst Agent",
    description="自然语言驱动的数据库分析与 SQL 优化系统",
    version="1.0.0",
    lifespan=lifespan,
)
```

> `lifespan` 是 FastAPI 的生命周期管理器。应用启动时会执行 `yield` 之前的代码，关闭时执行之后的代码。这里用它来确保数据目录和日志目录存在。

### 2.2 CORS 中间件

```python
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],      # 允许所有来源
    allow_credentials=True,
    allow_methods=["*"],      # 允许所有 HTTP 方法
    allow_headers=["*"],      # 允许所有请求头
)
```

> **什么是 CORS？**
>
> 当前端（运行在 `localhost:3000`）向后端（运行在 `localhost:8000`）发请求时，浏览器会因为"跨域"而阻止请求。CORS 中间件告诉浏览器："我允许来自其他域名的请求。"
>
> `allow_origins=["*"]` 表示允许所有域名，在开发阶段很方便，但生产环境应该限制为具体域名。

### 2.3 路由注册

```python
from .api import health, schema, query

app.include_router(health.router)   # /health 健康检查
app.include_router(schema.router)   # /api/schema 获取表结构
app.include_router(query.router)    # /api/chat/query 核心查询接口
```

> 每个路由模块定义了一组相关的 API 接口。拆分成不同文件是为了代码清晰。

## 3. 配置管理

> 配置文件在 `backend/app/config.py`，使用 `python-dotenv` 从 `.env` 文件读取环境变量。

### 3.1 Settings 类

```python
from dotenv import load_dotenv
import os

load_dotenv()  # 加载 .env 文件中的环境变量

class Settings:
    # 应用配置
    APP_HOST: str = os.getenv("APP_HOST", "0.0.0.0")
    APP_PORT: int = _get_int("APP_PORT", 8000)
    DEBUG: bool = _get_bool("DEBUG", False)

    # 数据库配置
    DATABASE_URL: str = os.getenv("DATABASE_URL", "duckdb:///./data/database.duckdb")

    # Qwen API 配置
    QWEN_API_KEY: str = os.getenv("QWEN_API_KEY", "")
    QWEN_MODEL: str = os.getenv("QWEN_MODEL", "qwen-turbo")

    # SQL 配置
    SQL_TIMEOUT: int = _get_int("SQL_TIMEOUT", 30)      # 查询超时秒数
    SQL_MAX_ROWS: int = _get_int("SQL_MAX_ROWS", 1000)   # 最大返回行数
    SQL_MAX_RETRIES: int = _get_int("SQL_MAX_RETRIES", 3) # SQL 修复最大重试次数

settings = Settings()
```

> **设计要点：**
>
> - 所有配置都从环境变量读取，有合理的默认值
> - 敏感信息（API Key）不写死在代码里，而是放在 `.env` 文件中
> - `.env` 文件在 `.gitignore` 中，不会被提交到 Git 仓库

### 3.2 辅助函数

```python
def _get_int(env_key: str, default: int) -> int:
    """读取整数类型的环境变量，转换失败则报错"""
    val = os.getenv(env_key, str(default))
    try:
        return int(val)
    except ValueError:
        raise ValueError(f"Environment variable {env_key} must be an integer, got: {val}")

def _get_bool(env_key: str, default: bool) -> bool:
    """读取布尔类型的环境变量"""
    val = os.getenv(env_key, str(default)).lower()
    return val in ("true", "1", "yes", "on")
```

> 这些辅助函数确保环境变量的类型正确。如果 `.env` 中写了 `APP_PORT=abc`，程序会立刻报错，而不是运行时出问题。

## 4. Pydantic 数据模型

> 模型定义在 `backend/app/models/schemas.py`，使用 Pydantic v2 来定义请求和响应的数据结构。
>
> **Pydantic 的作用：** 自动校验数据类型和约束，序列化/反序列化 JSON。

### 4.1 请求模型

```python
from pydantic import BaseModel, Field

class QueryRequest(BaseModel):
    """查询请求模型"""
    question: str = Field(
        ..., min_length=1, max_length=1000,
        description="自然语言问题"
    )
    session_id: Optional[str] = Field(
        None, max_length=128,
        description="多轮分析会话ID"
    )
```

> - `question` 是必填字段（`...` 表示必填），长度 1-1000 字符
> - `session_id` 是可选字段，用于多轮对话
> - FastAPI 会自动校验请求体，如果 `question` 为空或超长，会返回 422 错误

### 4.2 响应模型

```python
class SuccessResponse(BaseModel):
    """统一成功响应格式"""
    code: int = 200
    message: str = "success"
    data: Any  # 响应数据，可以是任意类型

class QueryResponse(BaseModel):
    """查询结果响应"""
    question: str                    # 用户问题
    sql: str                         # 生成的 SQL
    is_sql_safe: bool                # SQL 是否安全
    columns: List[str]               # 查询结果列名
    rows: List[List[Any]]            # 查询结果数据行
    answer: str                      # 自然语言回答
    execution_time_ms: int           # 执行耗时（毫秒）
    retry_count: int                 # SQL 修复重试次数
    optimization_suggestions: List[str]  # 优化建议
```

> **为什么用统一的 `SuccessResponse` 包装？**
>
> 让所有 API 的响应格式一致：`{ "code": 200, "message": "success", "data": {...} }`。前端可以统一处理。

### 4.3 Agent 内部模型

```python
class SQLGeneratorOutput(BaseModel):
    """SQL 生成器的输出结构"""
    sql: str            # 生成的 SQL
    tables: List[str]   # 用到的表名
    columns: List[str]  # 用到的字段名
    explanation: str    # SQL 逻辑说明

class SQLRepairOutput(BaseModel):
    """SQL 修复器的输出结构"""
    repaired_sql: str    # 修复后的 SQL
    repair_reason: str   # 修复原因
```

> 这些模型用于约束 LLM 的输出格式。LLM 被要求返回 JSON，这些模型定义了 JSON 的结构。

## 5. API 路由详解

### 5.1 健康检查（`/health`）

```python
# backend/app/api/health.py
@router.get("/health", response_model=SuccessResponse)
async def health_check():
    return SuccessResponse(
        code=200,
        message="success",
        data={"status": "healthy"}
    )
```

> 最简单的接口，用于监控服务是否正常运行。Docker 健康检查和负载均衡器都会调用它。

### 5.2 获取 Schema（`/api/schema`）

```python
# backend/app/api/schema.py
@router.get("/api/schema", response_model=SuccessResponse)
async def get_schema():
    schema = schema_loader.get_full_schema()
    return SuccessResponse(code=200, message="success", data=SchemaResponse(**schema))
```

> 返回数据库中所有表的结构信息（列名、类型、主键、外键），前端的 Schema 面板会用到。

### 5.3 核心查询接口（`/api/chat/query`）

```python
# backend/app/api/query.py
@router.post("/api/chat/query", response_model=SuccessResponse)
async def query(request: QueryRequest):
    result = await get_agent_graph().run(
        request.question,
        session_id=request.session_id,
    )
    response = QueryResponse(
        question=result["question"],
        sql=result.get("validated_sql") or result.get("generated_sql") or "",
        columns=query_result.get("columns", []),
        rows=query_result.get("rows", []),
        answer=result.get("answer") or "抱歉，处理您的问题时遇到困难。",
        ...
    )
    return SuccessResponse(code=200, message="success", data=response)
```

> 这是整个系统的核心入口。接收自然语言问题，调用 AgentGraph 工作流，返回完整结果。
>
> 注意 `await get_agent_graph().run(...)` — 这里会触发后面章节学习的整个 LangGraph 工作流。

### 6.2 异常定义

> `backend/app/utils/exceptions.py` 定义了项目专用的异常类：

```python
class DatabaseError(Exception): ...     # 数据库操作异常
class SchemaLoadError(Exception): ...   # Schema 加载异常
class SQLExecutionError(Exception): ... # SQL 执行异常
class SQLGuardError(Exception): ...     # SQL 安全校验异常
class LLMError(Exception): ...          # LLM 调用异常
class LLMTimeoutError(LLMError): ...   # LLM 超时

```

> 使用自定义异常类而不是通用 `Exception`，方便在不同层捕获不同类型的错误。

## 8. 启动流程总结

> 当你运行 `uvicorn app.main:app --reload` 时：
>
> 1. Python 加载 `main.py`
> 2. `load_dotenv()` 读取 `.env` 文件
> 3. `Settings()` 创建全局配置对象
> 4. FastAPI 创建应用实例
> 5. CORS 中间件注册
> 6. 路由注册（health、schema、query）
> 7. `lifespan` 执行，创建 data/ 和 logs/ 目录
> 8. Uvicorn 开始监听 8000 端口
> 9. 等待 HTTP 请求
