# 意图解析系统

## 1. 学习目标

> - 理解为什么要进行意图解析
> - 了解规则解析器如何快速提取结构化信息
> - 了解 LLM 解析器如何处理复杂场景
> - 理解两个解析器的结果如何合并

## 2. 为什么需要意图解析

> 用户的问题是自然语言，SQL 生成器需要结构化的信息才能生成正确的 SQL。
>
> 例如："2024年销售额最高的前5个商品类别"
>
> 人能理解这是：
> - 时间范围：2024年
> - 指标：销售额
> - 排序：从高到低
> - 限制：前5个
> - 维度：商品类别
>
> 但 LLM 需要更明确的提示才能准确理解。意图解析系统先把用户问题转换为结构化数据，再注入到 SQL 生成的 prompt 中。

## 3. 双层解析架构

```text
用户问题
  ├── 规则解析器（Rule Parser）→ 快速、确定性、零成本
  └── LLM 解析器（LLM Parser）→ 灵活、能处理复杂场景
         ↓
      合并器（Merger）→ 规则优先，LLM 补充
         ↓
      结构化意图（AnalysisIntent）
```

> **为什么不用纯 LLM？**
>
> - LLM 有延迟和成本
> - 简单信息（年份、Top-N）用正则更快更准
> - LLM 可能"幻觉"出不存在的指标
>
> **为什么不用纯规则？**
>
> - 规则无法理解隐式意图（"看看趋势"→时间维度）
> - 规则无法处理多意图（"对比华东和华南的销售额和利润率"）

## 4. 意图数据模型

> 代码在 `backend/app/analysis_intent/models.py`。

```python
from pydantic import BaseModel
from typing import List, Optional

class MetricConcept(BaseModel):
    """指标概念"""
    concept: str        # 指标英文 key，如 "sales_amount"
    confidence: float   # 置信度 0-1
    evidence: str       # 命中的原文片段，如 "销售额"

class DimensionConcept(BaseModel):
    """维度概念"""
    concept: str        # 维度英文 key，如 "region"
    confidence: float
    evidence: str

class AnalysisIntent(BaseModel):
    """完整的分析意图"""
    task_types: List[str]                # 任务类型：aggregation, ranking, trend...
    metrics: List[MetricConcept]         # 指标列表
    dimensions: List[DimensionConcept]   # 维度列表
    filters: List[dict]                  # 过滤条件
    time_granularity: Optional[str]      # 时间粒度：month, quarter, year
    ranking: Optional[dict]              # 排序信息：{order, limit}
    missing_slots: List[str]             # 缺失的槽位
    conflicts: List[str]                 # 冲突信息
    overall_confidence: float            # 整体置信度
```

## 5. 规则解析器（Rule Parser）

> 代码在 `backend/app/analysis_intent/rule_parser.py`。
>
> 使用正则表达式快速提取明确的信息。

### 5.1 提取 Top-N

```python
def _extract_ranking(self, question: str) -> Optional[dict]:
    patterns = [
        r"(?:前|top)\s*(\d+)",        # "前5个" 或 "top 5"
        r"最高的\s*(\d+)\s*个",        # "最高的5个"
        r"排名前\s*(\d+)",             # "排名前10"
    ]
    for pattern in patterns:
        match = re.search(pattern, question, re.IGNORECASE)
        if match:
            return {"order": "desc", "limit": int(match.group(1))}
    return None
```

### 5.2 提取时间信息

```python
def _extract_time(self, question: str) -> dict:
    # 提取年份
    year_match = re.search(r"(20\d{2})年?", question)
    if year_match:
        return {"year": int(year_match.group(1))}

    # 提取时间粒度
    if re.search(r"(按月|每月|月度|月趋势)", question):
        return {"granularity": "month"}
    if re.search(r"(按季度|每季度|季度)", question):
        return {"granularity": "quarter"}
    return {}
```

### 5.3 匹配业务别名

```python
def _match_metrics(self, question: str) -> List[MetricConcept]:
    # 从业务语义层加载指标别名映射
    ALIAS_MAP = {
        "sales_amount": ["销售额", "销售金额", "营收"],
        "profit": ["利润", "毛利", "净利润"],
        "order_count": ["订单数", "订单量", "下单数"],
    }
    matched = []
    for concept, aliases in ALIAS_MAP.items():
        for alias in aliases:
            if alias in question:
                matched.append(MetricConcept(
                    concept=concept, confidence=0.95, evidence=alias
                ))
                break
    return matched
```

> 规则解析器的优点：**零成本、零延迟、结果确定性**。相同输入永远产生相同输出。

## 6. LLM 解析器

> 代码在 `backend/app/analysis_intent/llm_parser.py`。
>
> 对于规则无法处理的复杂场景，交给 LLM 理解。

```python
class AnalysisIntentLLMParser:
    async def parse(self, question: str) -> AnalysisIntent:
        result = await llm_client.parse_analysis_intent(
            question=question,
            semantic_summary=self._get_semantic_summary(),
        )
        return AnalysisIntent(**result)
```

> LLM 的 prompt 要求它：
> 1. 只输出业务概念，不输出 SQL
> 2. 不输出物理表名或字段名
> 3. 对不确定的概念保留合理置信度
> 4. 返回严格的 JSON 格式

## 7. 合并器（Merger）

> 代码在 `backend/app/analysis_intent/merger.py`。
>
> 当规则和 LLM 都给出结果时，需要合并。

### 7.1 合并策略

```python
class AnalysisIntentMerger:
    def merge(self, rule_intent: AnalysisIntent, llm_intent: AnalysisIntent) -> AnalysisIntent:
        # 规则优先，LLM 补充
        merged_metrics = self._merge_concepts(
            rule_intent.metrics,        # 规则提取的指标
            llm_intent.metrics,         # LLM 提取的指标
        )
        merged_dimensions = self._merge_concepts(
            rule_intent.dimensions,
            llm_intent.dimensions,
        )

        # 排序信息：规则提取的更可靠
        ranking = rule_intent.ranking or llm_intent.ranking

        # 时间粒度：规则提取的更可靠
        time_granularity = rule_intent.time_granularity or llm_intent.time_granularity

        return AnalysisIntent(
            task_types=rule_intent.task_types or llm_intent.task_types,
            metrics=merged_metrics,
            dimensions=merged_dimensions,
            ranking=ranking,
            time_granularity=time_granularity,
            ...
        )
```

### 7.2 概念去重

```python
def _merge_concepts(self, rule_concepts, llm_concepts):
    # 规则提取的概念作为基础
    merged = {c.concept: c for c in rule_concepts}

    # LLM 提供的概念：如果规则已有相同 concept，保留规则的（更高置信度）
    # 如果规则没有，补充进来
    for llm_concept in llm_concepts:
        if llm_concept.concept not in merged:
            merged[llm_concept.concept] = llm_concept

    return list(merged.values())
```

## 8. Schema Grounding

> 代码在 `backend/app/agents/grounding.py`。
>
> 意图解析产出的是业务概念（如 `sales_amount`、`region`），还需要映射到物理数据库字段。这个过程叫 Grounding。

```text
业务概念          →    物理字段
sales_amount     →    order_items.quantity * order_items.unit_price
region           →    regions.region_name
product_category →    categories.category_name
```

> Grounding 会生成：
> - **候选列表**：每个业务概念可能对应的物理字段
> - **表达式**：指标的计算公式
> - **JOIN 路由**：需要连接哪些表

## 9. 主动澄清

> 代码在 `backend/app/agents/clarification.py`。
>
> 当用户的问题太模糊时，系统会主动追问，而不是生成一个可能错误的 SQL。

### 9.1 什么时候需要澄清

```python
def check(self, intent: AnalysisIntent) -> Optional[ClarificationRequest]:
    # 1. 指标缺失：用户说"分析一下"，但没说分析什么
    if not intent.metrics and "metric" in intent.missing_slots:
        return ClarificationRequest(
            question="您想分析什么指标？比如销售额、订单数、利润等？",
            candidates=[...]
        )

    # 2. 意图冲突：用户同时要求"按月"和"按年"
    if intent.conflicts:
        return ClarificationRequest(
            question="检测到时间粒度冲突，您想按月还是按年查看？",
            candidates=[...]
        )
```

### 9.2 澄清流程

```text
用户: "分析一下"
  → 意图解析: 指标缺失
  → 澄清引擎: 返回澄清请求
  → 工作流暂停，返回前端

用户: "销售额"（回答澄清）
  → 恢复工作流
  → 继续生成 SQL
```

## 10. 端到端示例

> 用户问题："2024年华东地区销售额最高的前5个商品类别"

```text
规则解析器输出：
  - year: 2024
  - ranking: {order: "desc", limit: 5}
  - metrics: [{concept: "sales_amount", confidence: 0.95, evidence: "销售额"}]
  - dimensions: [{concept: "category", confidence: 0.9, evidence: "商品类别"}]
  - filters: [{field: "region", value: "华东"}]

LLM 解析器输出：
  - metrics: [{concept: "sales_amount", confidence: 0.9, evidence: "销售额"}]
  - dimensions: [{concept: "category", confidence: 0.85, evidence: "商品类别"}, {concept: "region", confidence: 0.8, evidence: "华东地区"}]
  - time_granularity: null

合并后：
  - year: 2024（规则）
  - ranking: {order: "desc", limit: 5}（规则）
  - metrics: [sales_amount]（规则 + LLM 一致，置信度提升）
  - dimensions: [category, region]（规则提供 category，LLM 补充 region）
  - filters: [{field: "region", value: "华东"}]（规则）
```
