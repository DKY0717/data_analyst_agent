# 第九章 解析结构化分析意图

> 本章对应教学基线 `4d71b3c`。本章最后核对日期为 2026-07-11。

## 9.1 本章目标

> 完成本章后，你应该能够：
>
> 1. 说明“用户问题”为什么需要先转成结构化意图；
> 2. 解释指标、维度、过滤、时间粒度和排序等槽位；
> 3. 读懂规则解析器如何处理常见中文表达；
> 4. 理解 LLM 解析器为什么不能直接决定物理表字段；
> 5. 说明规则结果、LLM 结果、置信度和冲突如何合并；
> 6. 使用集成测试观察意图解析的确定性边界。

## 9.2 问题场景

> “统计华东地区 2024 年每个月销售额最高的 5 个商品”同时包含指标、地区过滤、时间范围、时间维度、排名和排名对象。如果直接把整句交给 SQL 生成器，模型可能遗漏其中一个条件，而且系统很难知道遗漏发生在哪里。
>
> 结构化意图把自然语言转换为中间表示：先描述用户要分析什么，再由语义层把业务概念映射到物理 Schema。这样问题理解和 SQL 生成可以分别测试，也可以在中间结果不确定时主动向用户提问。

## 9.3 AnalysisIntent 的核心槽位

> `backend/app/analysis_intent/models.py` 定义了 Pydantic 模型，用来表达分析任务而不是 SQL。核心概念可以先用下表理解：

| 槽位 | 例子 | 作用 |
|---|---|---|
| `task_type` | `aggregation`、`ranking` | 区分汇总、明细、趋势或排名 |
| `metrics` | 销售额、订单数 | 要计算的指标 |
| `dimensions` | 月份、地区、商品类别 | 如何分组或展开 |
| `filters` | 2024 年、华东 | 对数据范围的约束 |
| `time_granularity` | month、quarter | 时间维度粒度 |
| `ranking` | top 5、降序 | 排名方向和数量 |
| `missing_slots` | `metric` | 还缺少什么信息 |
| `overall_confidence` | `0.85` | 当前解析的可信程度 |
| `conflicts` | 两个可能指标 | 需要澄清的矛盾 |

> 这些槽位使用业务概念，不直接使用 `orders.total_amount` 或 `regions.region_name`。物理字段属于下一章的 Grounding 阶段，这个边界可以避免 LLM 解析阶段凭空创造表名。

## 9.4 规则解析器

### 9.4.1 为什么先有规则

> 规则解析器适合处理稳定、频繁、可解释的表达，例如“销售额”“订单数”“按月”“前 10 名”。它不需要网络和 API Key，速度稳定，也便于用单元测试锁定行为。

```python
metric_patterns = {
    "sales_amount": ["销售额", "销售金额", "GMV"],
    "order_count": ["订单数", "订单量", "订单数量"],
}
```

> 真实代码中规则还会结合时间表达式、比较运算和排名词。阅读时先关注“概念如何被识别”，不要把正则表达式本身当成完整中文理解能力。

### 9.4.2 规则的优点与边界

| 优点 | 边界 |
|---|---|
| 可解释、便于测试 | 覆盖范围依赖手工维护 |
| 不调用 LLM | 难以处理隐式和复杂表达 |
| 低延迟、无模型费用 | 同义词、歧义和上下文需要额外规则 |

> 规则解析器不应该为了“尽量猜”而把所有模糊问题强行映射到一个指标。没有足够证据时保留缺失槽位，后续可以让 LLM 补充或请求澄清。

## 9.5 LLM 解析器

> `llm_parser.py` 让模型把更复杂的自然语言转换成同一套 `AnalysisIntent` 结构。Prompt 明确要求只输出业务概念、任务类型、筛选和缺失槽位，并禁止输出 SQL、物理表名和物理字段名。

```text
用户问题 → LLM 解析器 → 结构化业务意图
                         ├─ 指标概念
                         ├─ 维度概念
                         ├─ 过滤条件
                         └─ 缺失槽位与置信度
```

> 让 LLM 只处理业务概念，可以降低它越过语义层自行拼接 SQL 的风险。LLM 输出仍然需要 Pydantic 校验，并且会与规则解析结果合并，不会直接决定最终执行语句。

## 9.6 合并两路结果

### 9.6.1 合并的目标

> `merger.py` 的职责不是简单“谁的字段多就选谁”，而是把规则和 LLM 的证据组合起来，去重同一指标，合并过滤和维度，保留冲突，计算整体置信度，并填充缺失槽位。

```text
规则解析结果 ─┐
              ├─ 合并、去重、冲突分析 → AnalysisIntent
LLM 解析结果 ─┘
```

> 规则命中通常是强证据；LLM 可以补充复杂表达，但如果两者对指标或维度给出不同概念，结果应该记录 `conflicts`，而不是静默覆盖。这样后续 Clarification Engine 才能知道为什么要提问。

### 9.6.2 置信度不是准确率

> `overall_confidence` 是当前解析器对自己结果的估计，不等于真实正确率。它可以作为是否澄清的一个信号，但不能替代 Grounding、SQL Guard 或执行结果验证。

## 9.7 一个解析示例

> 对问题“统计 2024 年每个月的销售额”，可以得到类似以下中间表示：

```json
{
  "task_type": "aggregation",
  "metrics": [{"concept": "sales_amount"}],
  "dimensions": [{"concept": "month"}],
  "filters": [{"concept": "order_date", "operator": "between", "value": "2024"}],
  "time_granularity": "month",
  "ranking": null,
  "missing_slots": [],
  "overall_confidence": 0.9,
  "conflicts": []
}
```

> 这里的 `sales_amount` 和 `month` 是稳定的业务 key，不是数据库列名。下一章会根据语义配置把它们映射到表达式和 JOIN 路径。

## 9.8 缺失槽位和主动澄清的入口

> 当用户只说“帮我分析一下数据”时，解析器可能无法识别指标。结果可以包含 `missing_slots: ["metric"]`，而不是猜测销售额。Clarification Engine 会根据缺失槽位生成候选，例如销售额、订单数或客户数。

> 多轮对话中，用户可能只说“按地区拆一下”。如果已有会话上下文，SQL Generator 可以继承最近一轮指标；没有上下文时，系统应保持不确定并请求补充。这就是结构化意图与会话状态协作的地方。

## 9.11 常见错误

### 把业务概念直接写成物理字段

> “销售额”可能来自订单总额，也可能来自订单明细金额；如果规则解析阶段直接写死某个字段，语义层就失去了选择口径的机会。解析阶段只保留稳定业务 key。

### 只使用 LLM 解析结果

> 这样会让同一个问题因模型版本、温度或网络变化而产生不同意图，也会增加费用。规则和 LLM 各自负责擅长的范围，再通过合并器统一输出。

### 对缺失指标强行猜测

> 猜测会把错误带入 SQL 和答案。发现 `missing_slots` 时，应进入澄清或明确提示用户补充信息。

### 用置信度掩盖冲突

> 置信度高不代表两个候选没有冲突。只要冲突可能改变查询口径，就应该保留冲突并决定是否澄清。

## 9.12 本章小结

> 结构化意图是自然语言与数据库之间的中间层：它描述用户想分析什么，但不假装已经知道物理表字段。规则提供可解释稳定性，LLM 补充复杂表达，合并器保留冲突和不确定性。下一章会把这些业务概念接到语义配置、真实 Schema 和 JOIN 图上。
