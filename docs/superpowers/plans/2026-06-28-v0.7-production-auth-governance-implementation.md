# v0.7 Production Auth Governance Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add production-style authentication and role-based table/column authorization to the query pipeline, with stable audit evidence and no regression to local unauthenticated development mode.
**Architecture:** Keep the existing LangGraph pipeline intact and insert a deterministic `authorize_sql` node after SQL Guard and before QueryRunner. API endpoints inject `AuthUser | None`, AgentState carries a sanitized auth summary, and AuditReport exposes only identity/role evidence needed for governance.
**Tech Stack:** FastAPI dependencies, existing JWT/API Key auth module, LangGraph `StateGraph`, SQLGlot AST inspection, Pydantic response models, pytest, existing Vue/Vitest frontend verification.

---

## Context

The approved design is `docs/superpowers/specs/2026-06-28-v0.7-production-auth-governance-design.md`.

Current state:

- `backend/app/security/auth.py` already supports JWT and API Key extraction, but `/api/chat/query` and `/api/chat/query/stream` do not call `get_current_user`.
- `AuthUser.roles` exists. JWT users can carry roles, API Key users currently default to `["user"]` and must be normalized to `analyst` for query authorization.
- `AgentGraph` currently routes `generate_sql -> validate_sql -> execute_sql`; v0.7 must become `generate_sql -> validate_sql -> authorize_sql -> execute_sql`.
- Query cache currently keys only by question. Once auth is present, cached data must not be reused across identities or roles.
- Audit reports are already centralized through `AuditReportBuilder`.

- [ ] Implement `backend/app/security/data_permission.py`.

  Required public API:

  ```python
  from dataclasses import dataclass, field
  from typing import Any


  @dataclass(frozen=True)
  class PermissionUser:
      user_id: str
      auth_method: str
      roles: list[str] = field(default_factory=list)


  @dataclass(frozen=True)
  class DataPermissionResult:
      is_allowed: bool
      reason: str
      blocked_rule: str | None
      audit_events: list[dict[str, Any]]
      referenced_tables: list[str]
      referenced_columns: list[str]


  class DataPermissionGuard:
      def authorize(
          self,
          sql: str,
          user: dict[str, Any] | PermissionUser | None,
          schema: dict[str, Any] | None = None,
      ) -> DataPermissionResult:
          ...
  ```

  Implementation requirements:

  - Use SQLGlot to parse DuckDB SQL; fail closed on parse errors with `block_permission_parse_error`.
  - Define built-in role policies for `admin`, `analyst`, and `support`.
  - Treat `None` user as local development user: `user_id="dev:anonymous"`, `auth_method="disabled"`, `roles=["admin"]`.
  - Normalize API Key identities with missing or generic roles (`[]`, `["user"]`) to `["analyst"]`.
  - `admin` allows all tables and all columns.
  - Unknown roles allow no business tables.
  - Table denial rule: `block_unauthorized_table`.
  - Column denial rule: `block_unauthorized_column`.
  - Ambiguous multi-table naked column rule: `block_ambiguous_column`.
  - Events use `stage="authorization"`, `action="authorize_sql"`, `status="success"` or `"blocked"`.
  - Event details may include `user_id`, `auth_method`, `roles`, `tables`, and checked column names, but must not include the full role policy.

  Suggested schema handling:

  - Support current loader shape where `schema_context["tables"]` is a mapping.
  - Build a `dict[str, set[str]]` from table names to column names.
  - For `Column` nodes with explicit table or alias, resolve aliases from `Table` nodes.
  - For unqualified columns:
    - If exactly one table is referenced, assign to that table.
    - If multiple tables are referenced and exactly one referenced table contains the column in schema metadata, assign to that table.
    - Otherwise fail closed with `block_ambiguous_column`.

  Required graph changes:

  - Import `data_permission_guard`.
  - Add node registration: `workflow.add_node("authorize_sql", self._authorize_sql)`.
  - Change SQL Guard conditional target from `execute_sql` to `authorize_sql`.
  - Add conditional edge after `authorize_sql`:

    ```python

        "authorize_sql",
        self._should_execute_authorized_sql,
        {
            "execute": "execute_sql",
            "end": END,
        },
    )
    ```

  - Keep `repair_sql -> validate_sql`, so repaired SQL is rechecked by both SQL Guard and Permission Guard before execution.
  - Add `auth_user` parameter to `run(...)` and store sanitized identity in initial state.
  - Populate default auth fields in `_build_expired_clarification_state`.

  Required node behavior:

  ```python
  @trace_node("authorize_sql")
  async def _authorize_sql(self, state: AgentState) -> Dict[str, Any]:
      await self._emit_progress(state, "校验数据权限...", 80)
      result = data_permission_guard.authorize(
          state["validated_sql"],
          state.get("auth_user"),
          state.get("schema_context"),
      )
      return {
          "status": "completed" if result.is_allowed else "blocked",
          "permission_allowed": result.is_allowed,
          "permission_error": None if result.is_allowed else result.reason,
          "audit_events": self._extend_audit_events(state, result.audit_events),
          "answer": None if result.is_allowed else f"请求已被数据权限策略阻断：{result.reason}",
      }
  ```

  Required routing behavior:

  - If `is_sql_safe` is false, end before authorization.
  - If `permission_allowed` is false, end before QueryRunner.
  - Permission block must never enter SQL Repair.

- [ ] Update `backend/app/api/query.py`.

  Required changes:

  - Import `Depends`, `AuthUser`, and `get_current_user`.
  - Inject `current_user: AuthUser | None = Depends(get_current_user)` in both query endpoints.
  - Convert `AuthUser | None` to a safe dict before passing into graph:

    ```python
    def _auth_user_payload(user: AuthUser | None) -> dict[str, Any] | None:
        if user is None:
            return None
        roles = user.roles or []
        if user.auth_method == "api_key" and (not roles or roles == ["user"]):
            roles = ["analyst"]
        return {
            "user_id": user.user_id,
            "auth_method": user.auth_method,
            "roles": roles,
        }
    ```

  - Pass `auth_user=_auth_user_payload(current_user)` to `AgentGraph.run(...)`.
  - Do not log raw credentials, tokens, API keys, or raw questions.
  - Avoid shared question-only cache for authenticated requests:
    - If `current_user is None`, keep the existing no-auth cache behavior.
    - If `current_user is not None`, skip cache read and skip cache write.

  Required normal query call shape:

  ```python
  result = await get_agent_graph().run(
      body.question,
      session_id=body.session_id,
      clarification_response=clarification_response,
      auth_user=_auth_user_payload(current_user),
  )
  ```

  Required stream query call shape:

  ```python
  result = await get_agent_graph().run(
      body.question,
      session_id=body.session_id,
      clarification_response=clarification_response,
      auth_user=_auth_user_payload(current_user),
      on_progress=on_progress_callback,
  )
  ```

## Acceptance Criteria

- [ ] When auth is enabled, `/api/chat/query` without credentials returns HTTP 401 and AgentGraph is not called.
- [ ] When auth is enabled, `/api/chat/query/stream` without credentials returns HTTP 401-equivalent FastAPI dependency behavior before graph execution.
- [ ] A valid JWT user reaches AgentGraph with sanitized `user_id`, `auth_method`, and `roles`.
- [ ] A valid API Key user reaches AgentGraph as an `analyst`.
- [ ] `analyst` can query approved analytical tables and non-sensitive customer fields.
- [ ] `analyst` cannot query `customers.customer_name`.
- [ ] `support` cannot query `payments.paid_amount`.
- [ ] Permission blocks return stable business response, no SQL execution, no SQL Repair.
- [ ] Permission decisions appear in `audit_report.events`.
- [ ] `audit_report.blocked_rules` includes permission rule IDs when blocked.
- [ ] Unauthenticated local development mode preserves existing behavior.
- [ ] Full backend tests, frontend tests, frontend build, secret scan, and `git diff --check` pass.
