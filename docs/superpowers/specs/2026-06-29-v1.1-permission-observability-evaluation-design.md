# v1.1 Permission Observability and Evaluation Design

## Goal

v1.1 turns the permission system from “implemented and tested in unit tests” into “observable in every response and regression-tested as a security capability.” The feature adds a stable permission summary to `audit_report` and a deterministic permission evaluation runner that produces JSON and Chinese Markdown reports.

## Context

v1.0 externalized data permissions into YAML, added analyst row-level SQL filters, and made `AgentGraph` execute `authorized_sql`. Today the evidence exists inside individual audit events, but consumers must scan `events[*].details` to answer basic questions:

- Was data permission checked?
- Was the query allowed or blocked?
- Did row-level filtering change the SQL?
- Which tables, columns, and policy rules were involved?

The project already has an `AuditReport` response model and multiple evaluation report writers that output both JSON and Markdown. v1.1 should follow those patterns instead of adding a new UI or external service.

## Design Choice

I considered three approaches:

1. **Event-only documentation**
   - Keep code unchanged and document how to read permission events.
   - Lowest effort, but weak for interviews and automation because every caller must re-derive the same summary.

2. **Audit summary plus deterministic evaluator**
   - Add `permission_observability` to `audit_report`.
   - Add a small evaluator that runs representative SQL/role cases through `DataPermissionGuard`.
   - Recommended because it improves API clarity, regression safety, and demo value without involving LLM or database execution.

3. **Full frontend security dashboard**
   - Build UI charts and drill-downs for permission events.
   - Visually richer, but too broad for the next backend-focused step and risks distracting from core agent/security depth.

v1.1 will use approach 2.

## Audit Report Shape

Add a new nested model:

```python
PermissionObservability
```

Fields:

- `permission_checked: bool`
- `allowed: bool | None`
- `blocked_rule: str | None`
- `referenced_tables: list[str]`
- `referenced_columns: list[str]`
- `row_filters_applied: list[dict[str, str]]`
- `authorized_sql_changed: bool`

`AuditReportBuilder.build_report()` will derive this summary from authorization events only. It will not expose row-filter expressions, YAML file paths, raw policy content, credentials, or database errors. If no authorization event exists, `permission_checked` is `False` and other fields remain empty or `None`.

## Evaluator

Add a deterministic backend evaluator:

```text
backend/evaluation/permission_evaluator.py
backend/evaluation/permission_report_writer.py
backend/tests/test_permission_evaluator.py
backend/tests/test_permission_report_writer.py
```

The evaluator uses `DataPermissionGuard.authorize()` directly with the e-commerce schema fixture style already used in tests. It does not call Qwen, does not execute SQL, and does not require a live database. This keeps the permission regression suite fast and stable.

Initial cases:

- analyst can query non-sensitive order metrics and receives row filter evidence.
- admin can query the same order metrics without SQL rewrite.
- analyst is blocked from `customers.customer_name`.
- support is blocked from `payments`.
- missing configured policy fails closed with `block_permission_policy_error`.

Summary metrics:

- `total_cases`
- `allowed_decision_accuracy`
- `blocked_rule_accuracy`
- `row_filter_expectation_accuracy`
- `authorized_sql_change_accuracy`
- `passed`

Quality target for this evaluator is strict: every metric must be `1.0` for `passed=True`.

## Report Writer

`PermissionReportWriter` writes:

- `permission-evaluation-<timestamp>.json`
- `permission-evaluation-<timestamp>.md`

The Markdown report is interview-friendly:

- top-line pass/fail and accuracy metrics
- compact case table
- row-filter rule IDs and tables
- no raw row-filter expressions and no sensitive policy dump

Output directory follows existing evaluation convention:

1. explicit `output_dir`
2. `EVALUATION_REPORT_DIR`
3. `backend/evaluation/reports`

## Data Flow

```text
SQL + role + schema
  -> DataPermissionGuard.authorize()
  -> DataPermissionResult
  -> evaluator case result
  -> summary metrics
  -> JSON + Markdown report
```

For normal API calls:

```text
AgentGraph audit events
  -> AuditReportBuilder.build_report()
  -> permission_observability
  -> QueryResponse.audit_report
```

## Error Handling

- Policy loading errors remain fail-closed and are summarized as `block_permission_policy_error`.
- Evaluator cases catch unexpected exceptions and mark the case as failed with a stable `error_type`.
- Reports do not include exception messages that may contain environment paths or secrets.

## Testing

TDD coverage will include:

- audit report builds an empty permission summary when no authorization occurred.
- audit report summarizes allowed row-filtered authorization.
- audit report summarizes blocked authorization.
- Pydantic `AuditReport` accepts `permission_observability`.
- evaluator produces all-pass metrics for default cases.
- evaluator records stable failure output for an unexpected case mismatch.
- report writer emits JSON and Markdown without dumping row-filter expressions.

Final verification:

- targeted backend tests for audit, query API, permission guard, evaluator, and report writer
- full `pytest -q`
- unresolved-marker scan
- `git diff --check`
- secret scan

## Non-Goals

- No frontend dashboard in v1.1.
- No database-backed policy admin.
- No external IAM integration.
- No LLM-based permission evaluation.
- No raw policy expression exposure in API or Markdown reports.

## Acceptance Criteria

- `audit_report.permission_observability` is present and stable for API responses.
- Row-filtered analyst queries summarize rule IDs and table names without leaking filter expressions.
- Blocked permission queries summarize the blocking rule.
- Permission evaluator can be run from Python and returns strict pass/fail metrics.
- Permission report writer outputs JSON and Markdown.
- README documents the new permission evaluation command and audit summary.
- Work diary records v1.1 implementation and verification evidence.
