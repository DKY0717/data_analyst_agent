# v0.9 权限演示 E2E 闭环设计

## 背景

v0.7 已经完成后端生产化权限治理：最终 SQL 在执行前经过角色级表/字段授权，权限阻断不执行、不修复、不写入多轮上下文，审计报告包含身份摘要、authorization 事件和阻断规则。

v0.8 已经完成前端权限演示工作台：顶部身份条支持 `admin`、`analyst`、`support` 三种演示身份，普通查询和 SSE 查询都会携带认证头，AuditPanel 能展示身份摘要和 authorization 事件。

当前缺口是“可重复验收的演示闭环”。现在已有基础 Playwright E2E 只覆盖页面结构、输入框、主题切换和基础查询状态，还不能证明 v0.8 权限演示路径能稳定跑通。v0.9 的目标是把权限演示变成稳定、快速、不依赖 LLM 的 E2E 门禁，并补一段 README 面试演示脚本。

## 目标

1. 新增一条稳定的 Playwright 权限演示 E2E。
2. E2E 不依赖真实 Qwen/MiMo API，不依赖真实后端权限执行结果。
3. E2E 覆盖完整演示路径：
   - 初始状态显示未认证。
   - 点击 Analyst 后显示 `demo:analyst`。
   - analyst 查询销售额成功。
   - analyst 查询客户姓名被权限阻断。
   - AuditPanel 展示 `demo:analyst`、`jwt`、`analyst`、`authorization` 和阻断规则。
   - 点击 Admin 后显示 `demo:admin`。
   - admin 查询同一客户姓名问题成功。
4. README 增加“30 秒面试演示路径”，让项目展示方式更直接。
5. 不改 v0.7/v0.8 的后端权限治理链路，只增加演示验收能力。

## 非目标

1. 不新增真实权限规则。
2. 不重构 DataPermissionGuard。
3. 不把 E2E 变成真实 LLM 验收。
4. 不录制 GIF 或截图文件；第一版先补文字脚本和自动化验收。
5. 不新增 CI workflow；先让本地 `npm run test:e2e -- permission-demo.spec.js` 可稳定运行。

## 推荐方案

采用“Playwright Mock 后端响应 + 前端真实交互”的方案。

原因：

- v0.9 的目的不是证明后端 SQL 权限逻辑正确，后端已有 pytest 和评测体系兜底。
- 面试演示 E2E 应该稳定、快速、可重复，不应被 API Key、网络、模型输出波动影响。
- Mock 接口仍然能验证前端真实交互链路：登录按钮、auth store、SSE/标准查询调用、结果展示、AuditPanel 证据展示。
- 将来如果需要真实端到端验收，可以单独做 `real-llm` E2E 或 GitHub Actions 手动 workflow。

## E2E 设计

新增文件：

```text
frontend/e2e/permission-demo.spec.js
```

### Mock 范围

E2E 在 `beforeEach` 中注册路由拦截：

- `GET /api/schema`
  - 返回最小 schema，让 SchemaPanel 不报错。
- `POST /api/auth/demo-login`
  - 根据请求 body 中的 role 返回对应 token 和 user。
- `GET /api/auth/me`
  - 返回当前 mock user。
- `POST /api/chat/query/stream`
  - 根据当前角色和问题返回 SSE 格式结果。
- `POST /api/chat/query`
  - 同样返回普通 JSON 结果，避免关闭 SSE 后路径失效。

### 交互路径

1. 打开首页。
2. 断言顶部显示 `未认证`。
3. 点击 `Analyst`。
4. 断言顶部显示 `demo:analyst`、`jwt`、`analyst`。
5. 在输入框提交“统计 2024 年每个月的销售额”。
6. 断言结果解释显示销售额成功答案。
7. 提交“列出客户姓名和注册日期”。
8. 断言结果解释显示权限阻断答案。
9. 断言 AuditPanel 显示：
   - `demo:analyst`
   - `jwt`
   - `analyst`
   - `authorization`
   - `block_unauthorized_column`
   - `block_unauthorized_column:customers.customer_name`
10. 点击 `Admin`。
11. 断言顶部显示 `demo:admin`。
12. 再次提交“列出客户姓名和注册日期”。
13. 断言结果解释显示 admin 成功答案，并且 AuditPanel 不再显示阻断规则数量大于 0。

## 前端选择器策略

第一版优先使用现有 class 和可见文本：

- `.auth-bar`
- `[data-test="role-analyst"]`
- `[data-test="role-admin"]`
- `.query-card textarea`
- `.submit-button`
- `.answer-panel`
- `.audit-panel`

- `data-test="auth-current-user"`
- `data-test="answer-panel"`
- `data-test="audit-panel"`

这些属性只服务测试稳定性，不改变视觉和业务行为。

## 风险与处理

### 风险 1：Mock E2E 被误解为真实后端验收

处理：README 明确说明 v0.9 E2E 是“演示闭环验收”，后端权限正确性由 pytest 和评测体系验证。

### 风险 2：SSE Mock 格式不符合前端解析

处理：Mock 返回标准 SSE 文本：

```text
data: {"type":"progress","stage":"权限检查","progress":60}

data: {"type":"result","data":{...}}

data: [DONE]
```

并设置 `Content-Type: text/event-stream`。

### 风险 3：现有 E2E 统计示例问题数量失败

v0.8 已新增第二组权限演示问题，当前 `app.spec.js` 中“示例问题展示 12 条”可能需要收紧选择器到第一个示例问题面板，或改为“普通示例问题 12 条，权限演示问题 4 条”。v0.9 允许顺手修这个 E2E 断言，但不改业务 UI。

## 验收标准

1. 新增 `permission-demo.spec.js` 可稳定通过。
2. 测试覆盖 analyst 成功、analyst 阻断、admin 同题成功。
3. AuditPanel 的身份和 authorization 证据被 E2E 断言覆盖。
4. README 能指导面试者 30 秒讲清楚权限演示路径。
5. 不引入真实 LLM/API Key 依赖。
6. 不改后端核心权限逻辑。
