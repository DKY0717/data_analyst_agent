# v1.6 核心路径打磨设计

## 背景

项目已经有完整的 Data Analyst Agent 主链路、32 条 NL2SQL 评测、结果正确性黄金基准、权限评测、安全审计导出、面试证据包和演示预检脚本。当前缺口不是能力不足，而是核心演示路径还分散在多处：前端推荐问题、评测 case、结果正确性 case、面试脚本和预检脚本各自存在，但没有一份面向“用户从提问到可信答案”的黄金路径清单和自动化回归入口。

v1.6 的目标是把核心路径打磨成可重复演示、可回归验证、可解释失败边界的闭环。

## 目标

- 建立一份核心路径黄金问题包，覆盖最能展示项目价值的业务分析、安全阻断、权限治理和多轮追问场景。
- 让黄金问题包成为前端推荐问题、面试演示脚本和自动化回归的共同事实来源，减少文档与代码漂移。
- 新增核心路径回归入口，优先验证问题配置、期望能力、前端展示文案和已有评测 case 的一致性。
- 强化演示预检输出，让面试或复盘时能按固定顺序展示“成功分析、可信 SQL、图表、审计、失败处理”。

## 非目标

- 不重写 LangGraph 工作流。
- 不新增新的 LLM 供应商或模型调用方式。
- 不扩大数据库 schema。
- 不把前端改成完整 BI 配置器。
- 不在普通 CI 中强制调用真实 LLM。

## 方案对比

### 方案 A：只更新文档和演示脚本

成本最低，能快速改善面试讲述，但黄金问题仍然无法被测试锁定，后续前端推荐问题和评测 case 可能继续漂移。

### 方案 B：新增统一黄金问题包并做轻量回归

把核心问题定义为结构化 YAML，前端推荐问题、演示脚本和测试都围绕它对齐。实现范围较小，但能把“可演示”变成“可验证”。这是推荐方案。

### 方案 C：端到端真实 LLM 演示质量门禁

最接近生产质量，但会引入网络、密钥、模型波动和执行耗时，不适合普通开发节奏。可以作为后续真实评测 workflow 的扩展，不作为 v1.6 首要目标。

## 推荐设计

采用方案 B。v1.6 新增一个核心路径黄金问题包，保留已有 `backend/evaluation/cases/ecommerce_nl2sql_cases.yaml` 和 `backend/evaluation/cases/golden_result_cases.yaml` 的职责不变：前者评测 NL2SQL 生成，后者评测结果正确性。新文件只负责“演示和核心路径选择”，通过 case id 复用已有评测资产。

核心路径分为五类：

- `business_success`：能稳定展示表格、图表、SQL 和自然语言解释的业务问题。
- `business_metric`：突出业务口径，例如退款率、客单价、复购率。
- `follow_up`：展示 `session_id` 多轮追问能力。
- `permission`：展示 analyst/admin 权限差异、行级过滤和字段阻断。
- `safety_failure`：展示危险意图或危险 SQL 被清楚拦截。

## 文件设计

### `backend/evaluation/cases/core_path_cases.yaml`

新增结构化配置，字段如下：

- `id`：稳定 case id。
- `question`：用户可直接输入的问题。
- `category`：核心路径分类。
- `demo_role`：推荐演示身份，可选 `analyst`、`admin`、`support`。
- `linked_cases`：关联已有 NL2SQL、golden result、permission 或 safety case id。
- `expected_surfaces`：演示时应该观察到的界面或响应要素，例如 `sql`、`chart`、`audit_report`、`permission_observability`、`clarification`。
- `success_criteria`：面向人类的验收说明，用于脚本和文档展示。

第一版选择 12 到 16 条问题，不追求覆盖所有评测，只覆盖最适合演示的路径。

### `backend/evaluation/core_path.py`

新增轻量加载器和校验器，不调用 LLM，不连接真实数据库。职责包括：

- 读取 `core_path_cases.yaml`。
- 校验 id 唯一、分类合法、问题非空。
- 校验 `linked_cases` 指向的已有评测 case 存在。
- 输出按分类分组的演示问题列表，供脚本或文档生成使用。

### `backend/tests/test_core_path_cases.py`

新增测试锁定核心路径配置质量：

- 至少包含业务成功、业务指标、权限、安全失败四类。
- 前端推荐问题必须包含核心业务成功问题的一个稳定子集。
- `scripts/interview_demo_preflight.py` 的演示顺序必须与核心路径配置中的主线 case 对齐。
- 所有关联的评测 case id 必须存在。

### `frontend/src/api/agent.js`

保留静态 `exampleQuestions`，但调整顺序和内容，使前 6 个问题与核心路径黄金问题一致。这样前端首页的第一屏就是可演示主线。

### `scripts/interview_demo_preflight.py`

扩展演示顺序，输出核心路径问题和每一步观察点。预检仍然不联网、不调用 LLM；它只帮助用户确认环境和演示脚本，不负责证明模型实时结果。

### 文档

更新 README、`docs/interview_guide.md` 和 `docs/resume_project_packet.md` 的演示段落，明确 v1.6 的核心路径入口：

- 黄金问题包路径。
- 核心路径回归测试命令。
- 面试前推荐演示顺序。

## 数据流

```text
core_path_cases.yaml
  -> CorePathCaseLoader
  -> tests validate linked case ids and frontend/demo alignment
  -> frontend exampleQuestions use stable subset
  -> interview_demo_preflight prints ordered demo script
  -> docs reference same command and case file
```

## 错误处理

- YAML 结构错误：加载器抛出明确的 `CorePathCaseError`，指出文件路径和 case id。
- 关联 case 不存在：测试失败并列出缺失 id。
- 分类不合法：测试失败，避免前端或文档出现无法解释的类别。
- 演示脚本与核心路径不一致：测试失败，提示更新脚本或配置。

## 测试策略

- 单元测试加载器和配置校验。
- 文档一致性测试锁定 README、面试稿和简历包中的核心路径入口。
- 前端静态问题测试锁定推荐问题顺序。
- 不在 v1.6 普通测试中调用真实 LLM。

## 验收标准

- 新增核心路径黄金问题包，至少 12 条 case。
- 核心路径配置能被测试加载并校验通过。
- 前端推荐问题前 6 条与核心业务演示问题一致。
- 面试演示预检输出核心路径演示顺序和观察点。
- README、面试稿、简历包记录同一个核心路径入口和测试命令。
- 后端 focused 测试、前端相关测试和 `git diff --check` 通过。
