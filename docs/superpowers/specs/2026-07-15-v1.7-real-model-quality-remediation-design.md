# v1.7 真实模型质量门禁修复设计

## 背景

分支 `codex/v1.7-final-hardening` 的基础 GitHub CI 已全部通过。相同 HEAD 使用 Qwen `qwen-plus` 完成真实模型全链路评测后，100% 质量门禁暴露出两类实现缺陷：合法 CTE/派生子查询被数据权限 Guard 误判，以及“地区”语义同时展开为地区、省份、城市三个字段。评测报告还没有记录权限阻断原因，导致失败项呈现为 `error: null`。

本轮只修复上述真实模型证据直接证明的问题，不降低质量门槛，不放宽敏感字段权限，也不改动 LangGraph 节点顺序、SQL Guard 或 Repair 重试语义。

## 目标

1. 数据权限 Guard 能按查询作用域区分物理表、CTE 和派生子查询。
2. CTE/派生结果字段不再被误认为物理字段，同时底层敏感字段仍必须接受权限检查。
3. “地区”“省份”“城市”分别映射到最小且明确的物理字段。
4. NL2SQL 评测明确记录权限阻断阶段、规则和脱敏错误原因。
5. 当前 HEAD 重新通过本地门禁、远端基础 CI 和强制阈值真实模型评测。

## 方案比较

### 方案 A：作用域解析 + 语义拆分（采用）

使用 SQLGlot 的查询作用域解析每个 `SELECT` 的数据源。物理表字段直接授权；CTE 和派生子查询的外层字段不重复授权，其底层查询仍逐字段检查。将地区语义拆成三个独立维度，并补齐评测可观测性。该方案修复根因且保持 fail-closed。

### 方案 B：别名白名单

识别当前失败 SQL 中的少量 CTE/子查询别名并跳过。改动较小，但无法覆盖嵌套、相关子查询和不同模型生成形态，容易形成安全缺口。

### 方案 C：降低门槛或修改评测预期

通过降低 100% 阈值或把失败 case 排除出安全执行分母使工作流变绿。该方案会掩盖实现缺陷，不能形成可信的应聘展示证据，因此不采用。

## 设计

### 1. 查询作用域感知的字段权限解析

`DataPermissionGuard` 继续先由 SQL Guard 保证只处理安全 SQL，再使用 SQLGlot scope 遍历每个查询层：

- 当前 source 为物理 `Table` 时，将列解析为真实的 `table.column` 并按策略授权。
- 当前 source 为 CTE 或派生 `Scope` 时，外层对派生字段的引用不重复映射到伪物理表；子 scope 中的真实表字段仍会独立遍历并授权。
- 无限定列只在当前 scope 内解析。唯一物理来源或唯一 Schema 命中时允许归属；无法唯一判断时继续 `block_ambiguous_column`，保持 fail-closed。
- `SELECT *` 对物理表仍按 `table.*` 检查；派生 source 的外层星号不重复授权，但其内层物理星号必须检查。
- 相关子查询引用外层物理表时，沿父 scope 解析；仍无法确定时阻断。

表级授权、字段级授权、行级过滤注入和审计事件格式保持不变。管理员通配策略不会绕过解析失败；Analyst 通过子查询读取 `customers.customer_name` 仍应被 `block_unauthorized_column` 阻断。

### 2. 地域维度最小映射

语义层改为三个独立维度：

- `region`：别名“地区、区域”，字段仅为 `regions.region_name`。
- `province`：别名“省份、省”，字段仅为 `regions.province`。
- `city`：别名“城市、市”，字段仅为 `regions.city`。

三者复用相同的订单到客户再到地区 JOIN 路由。Grounding 继续选择单个候选表达式，不新增模型后处理或硬编码 SQL。这样“按地区”只输出 `region_name`，明确提出省份或城市时仍能得到相应粒度。

### 3. 评测可观测性

`EvaluationRunner` 增加权限阶段识别：SQL Guard 通过但 `permission_allowed` 为 false 时，`blocked_stage` 记录为 `permission_guard`。结果增加 `permission_allowed` 和脱敏的权限规则 ID；`error` 按意图、SQL 校验、权限、执行顺序选择原因。

安全 case 的成功语义保持严格：标记为 `safe` 的 case 必须执行成功；权限误阻断不能被计为成功。标记为 `unsafe` 的 case 仍只接受 Intent Guard 或 SQL Guard 阻断，不能用权限 Guard 替代 SQL 安全边界。

### 4. 测试设计

新增或扩展以下回归测试：

- 管理员可执行真实评测中合法的多层 CTE SQL。
- 管理员可执行派生子查询 SQL。
- Analyst 不能通过 CTE 或派生子查询泄露 `customers.customer_name`。
- 多表裸列和无法解析的相关引用继续 fail-closed。
- 地区、省份、城市分别得到唯一字段和稳定 candidate ID。
- SQL Generator 的 Grounding prompt 对“地区”只包含 `regions.region_name`。
- NL2SQL 评测正确记录 `permission_guard`、规则和权限错误，不再输出无原因失败。

## 错误处理与安全不变量

- 不通过捕获异常后默认允许来提高通过率。
- 不把 CTE/派生别名加入物理表白名单。
- 不在报告中输出策略表达式、身份凭据或数据库诊断原文。
- 解析异常、重复别名、列归属不唯一继续阻断。
- SQL 必须保持 SQL Guard → Permission Guard → Query Runner 的执行顺序。
