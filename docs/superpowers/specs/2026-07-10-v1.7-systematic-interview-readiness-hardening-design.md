# v1.7 系统性求职项目加固设计

## 1. 背景

当前项目已经具备 LangGraph 工作流、三层安全治理、结构化评测、前端工作台、Docker 和 CI，但代码审查发现若干“单元测试通过、真实闭环仍可能失败”的问题：

- 前端读取旧响应字段，真实后端返回后 SQL、意图和优化建议面板可能为空。
- 主动澄清事件参数不一致，用户点击候选后无法恢复原问题。
- Schema Grounding 通过字符串切割表达式提取表和字段，产生 `SUM(orders` 等伪表名，并且没有正确生成 JOIN 边。
- SQL Guard 只补缺失 LIMIT，不限制已有超大 LIMIT。
- 行级权限过滤在无别名 JOIN 中可能注入裸列，造成歧义错误。
- 直接执行模式没有真实查询超时；Docker 默认配置也没有把认证、沙箱和代理超时形成一致部署边界。
- LLM 意图解析调用未进入最终调用统计，成本和耗时被低估。
- E2E Mock 同时提供新旧字段，掩盖真实 API 契约漂移；端口占用时测试会连接到错误服务。
- v1.6 核心路径只有静态关联校验，不能证明当前主流程可执行。
- 真实模型工作流的 provider、URL、模型和 Secret 没有形成统一配置，远端真实评测证据早于当前版本。
- Alembic 只有脚手架，没有真实 revision；CI 缺少统一 lint、类型/静态检查、覆盖率和依赖安全门禁。

v1.7 的目标不是重写项目，而是修复这些边界，使代码、测试、部署和求职材料描述同一套真实能力。

## 2. 目标

1. 真实浏览器通过当前 FastAPI `QueryResponse` 后能完整展示回答、SQL、意图、优化建议、表格、图表和审计报告。
2. 主动澄清候选可以恢复原问题并继续执行，不依赖旧事件结构。
3. Grounding 输出只包含合法物理表/字段，并正确提供语义层声明的 JOIN 边。
4. SQL 结果规模、执行超时和行级权限改写形成可验证的安全边界。
5. 全部 LLM 调用都进入请求级调用次数、Token、耗时和成本统计。
6. 核心路径至少具备确定性全链路回归，真实模型工作流具备少量 smoke 和可下载报告。
7. 本地开发、面试演示和受保护部署的配置边界明确且可验证。
8. 数据库迁移、代码质量和依赖安全不再只停留在文档声明。

## 3. 非目标

- 不重写 LangGraph 主流程。
- 不更换 Vue、FastAPI、DuckDB/PostgreSQL 或现有 LLM OpenAI-compatible 协议。
- 不引入复杂多租户管理后台、外部策略服务或完整 Kubernetes 基础设施。
- 不把真实 LLM 调用加入普通 PR 的强制门禁，避免成本和随机性影响日常 CI。
- 不为了追求测试数而复制已有 case；核心路径继续引用已有业务事实来源。

## 4. 总体架构

```text
Vue Query Store
  ↕ QueryResponse 唯一契约
FastAPI Query API
  ↓
Intent Guard → Intent Parser → Schema Grounding → Clarification
  ↓
SQL Generator → SQL Guard → Permission Guard → Query Runner
  ↓                         ↖ Repair 后重新校验
Optimizer → Answer Generator → Audit Report
```

### 4.1 契约原则

- FastAPI `QueryResponse` 是前端唯一事实来源。
- 前端不再读取 `generated_sql`、`optimized_sql`、`intent_parsed`、`optimizations` 等历史字段。
- 后端对外保留一个最终安全 SQL 字段；生成前后差异通过审计事件表达，不再让多个相似字段漂移。
- E2E Mock 必须严格匹配 Pydantic 响应模型，禁止额外添加旧字段来“兼容”组件。

### 4.2 Agent 原则

- Intent Guard、SQL Guard 和 Permission Guard 继续 fail-closed。
- Guard 或权限拒绝不进入 Repair。
- 执行失败可以 Repair，但 Repair 输出必须重新经过 SQL Guard 和 Permission Guard。
- Grounding 负责业务概念到物理 Schema 的结构化映射，不承担 SQL 字符串解析职责。

### 4.3 执行原则

- SQL Guard 负责语句类型、危险对象和结果行数上限。
- Permission Guard 负责表、字段和行级授权，改写后的 SQL 必须可被目标数据库实际执行。
- Query Runner 负责真实超时和连接清理；超时不是仅用于日志统计的配置值。
- DuckDB 沙箱连接保持只读；PostgreSQL 生产账号应使用只读权限并设置 statement timeout。

## 5. 分阶段设计

### 5.1 阶段一：前端真实闭环

修改 `Home.vue` 和相关组件，统一消费：

- `result.sql`
- `result.analysis_intent`
- `result.optimization_suggestions`
- `result.execution_time_ms`
- `result.retry_count`
- `result.audit_report`

澄清事件统一直接传递候选对象，Store 从候选读取 `candidate_id` 和 `label`。新增前端契约 fixture，由测试根据后端公开响应字段构造，不再维护两套字段。

验收：真实契约 fixture 下 SQL、意图、优化建议和澄清操作均可见、可交互。

### 5.2 阶段二：Grounding 与 LLM 可观测性

Grounding 不再解析聚合表达式中的表名。表和字段来源按优先级确定：

1. 语义配置显式 `default_table`、`required_joins` 和维度字段。
2. 对确需解析的 `table.column` 表达式使用 SQLGlot 包装成可解析 SELECT 表达式后遍历 Column AST。
3. 无法解析时返回空候选并降低置信度，不生成伪表名。

JOIN 边直接来自 metric/dimension 的 `required_joins`，统一解析成稳定 `(left, right)` 二元组并去重。

LLM 轨迹从请求开始持续到答案生成。意图解析、SQL 生成、Repair 和答案生成都在同一 ContextVar 轨迹中追加，节点不得用空 state 重置已有调用。

验收：Grounding precision/recall、JOIN edge accuracy 和完整 LLM call count 都有回归测试。

### 5.3 阶段三：SQL、权限和执行安全

SQL Guard 对顶层 LIMIT 执行三态处理：

- 缺失：注入 `SQL_MAX_ROWS`。
- 常量大于上限：钳制为 `SQL_MAX_ROWS` 并记录审计事件。
- 非常量、负数或无法安全判断：fail-closed。

Permission Guard 在行过滤表达式中始终给目标表顶层字段加物理表名或别名；对子查询内部字段保持原作用域。新增“授权改写 → SQL Guard → 真实 DuckDB 执行”的集成测试。

Query Runner：

- 沙箱模式继续由父进程硬超时。
- PostgreSQL 直接模式在事务内设置本地 `statement_timeout`。
- DuckDB 直接模式使用可中断执行边界；如果当前驱动无法可靠跨平台中断，则生产/容器强制使用沙箱，本地直接模式明确标记为开发模式。

验收：超大 LIMIT、动态 LIMIT、权限 JOIN、超时和修复回流均有测试。

### 5.4 阶段四：核心路径、E2E、真实模型和部署

新增确定性核心路径 Runner：

- 使用可注入 Fake LLM 返回固定结构化输出。
- 其余环节使用真实 LangGraph、Guard、Permission Guard 和隔离 DuckDB。
- 覆盖业务成功、业务指标、多轮追问、澄清、权限允许/阻断和危险意图阻断。

真实模型工作流只选择 3～5 条 smoke，provider、API URL、model 和 Secret 显式输入；完整 32 条评测仍保留手动触发。报告写入当前提交 SHA、provider、model、case 版本和生成时间。

E2E 托管服务使用显式独占端口和 `strictPort`，Playwright baseURL 从环境变量读取。端口被占用时快速失败，不能自动漂移后继续访问旧服务。

Docker：

- 代理读取超时大于单次 LLM 超时，并通过 SSE 心跳避免长节点期间断流。
- 提供 demo compose 默认和受保护 compose/profile 的明确说明。
- 受保护模式要求认证配置、沙箱和非默认管理凭据；缺失时 readiness 不通过。

### 5.5 阶段五：迁移与工程治理

- 合并重复的 `alembic/`、`migrations/` 概念，只保留 Alembic 实际使用目录。
- 创建可重复执行的初始 revision，并在临时 PostgreSQL 上验证 upgrade/downgrade/upgrade。
- 增加 Ruff，优先覆盖新增和核心后端目录；前端增加 ESLint。
- 增加覆盖率报告和合理阈值，阈值以当前基线为起点，不通过排除核心代码来虚增。
- 增加 `pip-audit`/`npm audit` 或等价依赖扫描；普通依赖风险告警与高危阻断规则分开。
- 更新依赖安装方式时保持现有启动命令兼容。

## 6. 错误处理与安全审计

- 对外不返回 LLM 原始响应、数据库异常全文、凭据、Prompt 或权限表达式。
- 审计报告可记录最终 SQL 和规则 ID；OpenTelemetry 默认不记录包含用户字面量的完整 SQL，只记录 hash、表名、节点状态和耗时。
- SSE 错误使用稳定错误事件，并定期发送注释心跳；取消请求后取消后台 Agent task。
- 配置错误在启动或 readiness 阶段暴露，不允许静默退回不安全生产默认值。

## 7. 测试矩阵

| 层级 | 重点 |
|---|---|
| 后端单元测试 | Grounding、LIMIT、权限、LLM 轨迹、配置校验 |
| 后端集成测试 | 权限改写后真实执行、超时、修复重新过 Guard |
| API 契约测试 | Pydantic QueryResponse 与前端 fixture 字段一致 |
| 前端单元测试 | SQL/意图/优化建议展示、澄清提交 |
| Playwright E2E | 干净托管端口、权限演示、核心响应面板 |
| 确定性核心路径 | Fake LLM + 真实工作流/数据库 |
| 真实模型 smoke | 当前 provider/model 的少量代表性问题 |
| CI/部署 | DuckDB、PostgreSQL、Docker readiness、迁移、lint、安全扫描 |

## 8. 提交与回滚

每个阶段独立提交：

1. `fix: align frontend query response contract`
2. `fix: make schema grounding routes precise`
3. `fix: enforce sql execution safety boundaries`
4. `test: add executable core path regression`
5. `ci: align real model and deployment quality gates`
6. `chore: add migrations and engineering quality gates`
7. `docs: sync interview readiness evidence`

阶段提交必须通过对应 focused tests；最终提交前运行全量后端、前端单测、构建、E2E、Secret Scan 和文档一致性检查。任一阶段可按提交独立回滚，不允许用兼容旧字段等方式掩盖失败。

## 9. 完成标准

- 审查中列出的功能性、Agent 正确性、安全执行、证据、部署和工程治理问题都有对应代码修复或明确受保护配置。
- 所有新增测试验证真实行为，而不只验证文件中存在某个字符串。
- 当前 main 的真实 API 契约可被浏览器完整消费。
- 当前 provider/model 至少有一份与 HEAD 对齐的真实 smoke 报告；若本地网络或 Secret 不可用，工作流配置和确定性证据必须完整，真实运行明确列为唯一外部执行项，不能伪造通过。
- README、面试稿、简历材料不再把静态清单、脚手架或历史报告包装成当前端到端证据。
