# 前端工作台设计方案

## 背景

Data Analyst Agent 的核心不是固定报表或普通聊天，而是把自然语言问题转成安全 SQL，并把 SQL 生成、校验、执行、修复、解释和优化建议形成一个可追踪闭环。

前端需要同时满足两个目标：

1. 给普通用户一个好用、清爽、容易理解的问数入口。
2. 给项目展示和开发调试保留足够透明的 SQL、执行状态和优化细节。

## 设计方向

第一版采用“现代 AI 数据分析工作台”界面。

整体风格应清爽、专业、轻量、可信。页面不做营销落地页，也不做炫技型 BI 大屏；首页直接进入可使用的分析工作台。视觉上采用浅色背景、白色内容面板、低饱和强调色和克制的状态标签，让界面符合当前主流 SaaS / AI 工具审美，同时保持数据工具的扫描效率。

## 页面结构

首页采用三栏工作台结构。

### 左侧栏：提问与上下文

左侧栏承载用户开始分析前最常用的入口：

- 自然语言问题输入框。
- 示例问题快捷按钮。
- 查询历史。
- Schema 简览入口。

输入框需要始终容易找到。示例问题用于降低首次使用门槛，建议覆盖销售趋势、商品排行、地区表现、退款率和复购率等电商分析场景。

### 中间主区：分析结果

中间主区优先展示用户最关心的结果：

- 自然语言回答摘要。
- 图表展示。
- 关键指标。
- 查询结果表格。

结果返回后，用户应该先看到“答案是什么”，再看到支撑答案的数据。表格和图表应在同一视野内形成互补：图表适合看趋势和对比，表格适合核对明细。

### 右侧详情栏：过程透明

右侧详情栏展示技术过程：

- 生成 SQL。
- SQL 安全校验状态。
- 执行耗时。
- 修复次数。
- 优化建议。
- 错误信息。

详情栏应支持折叠。普通用户可以专注答案，开发者和面试展示时可以展开查看系统闭环。

## 组件划分

第一版建议按以下组件拆分：

```text
App.vue
└── Home.vue
    ├── QueryInput.vue
    ├── ExampleQuestions.vue
    ├── HistoryPanel.vue
    ├── SchemaPanel.vue
    ├── AnswerPanel.vue
    ├── ChartPanel.vue
    ├── ResultTable.vue
    ├── SQLPanel.vue
    └── OptimizationPanel.vue
```

组件职责：

- `QueryInput.vue`：输入问题、提交问题、展示加载状态。
- `ExampleQuestions.vue`：展示可点击的示例分析问题。
- `HistoryPanel.vue`：展示最近查询记录，第一版可先使用前端内存状态。
- `SchemaPanel.vue`：展示数据库 8 张电商业务表的简览。
- `AnswerPanel.vue`：展示自然语言解释和状态提示。
- `ChartPanel.vue`：根据结果数据展示 ECharts 图表。
- `ResultTable.vue`：展示查询结果表格。
- `SQLPanel.vue`：展示 SQL、安全状态和复制按钮。
- `OptimizationPanel.vue`：展示执行耗时、retry 次数和优化建议。

## 数据流

前端通过 `POST /api/chat/query` 提交问题。

请求：

```json
{
  "question": "统计 2024 年每个月的销售额"
}
```

响应字段按开发文档第一版约定消费：

```json
{
  "question": "统计 2024 年每个月的销售额",
  "sql": "SELECT ...",
  "is_sql_safe": true,
  "columns": ["month", "sales"],
  "rows": [],
  "answer": "2024 年每个月销售额如下...",
  "execution_time_ms": 42,
  "retry_count": 0,
  "optimization_suggestions": []
}
```

`stores/query.js` 负责保存：

- 当前问题。
- 加载状态。
- 当前结果。
- 当前错误。
- 查询历史。

`api/agent.js` 负责 API 请求封装。第一版可以保留 mock 数据兜底，方便后端 Agent 尚未全部完成时预览前端。

## 交互状态

页面至少需要覆盖以下状态：

1. 初始状态：展示输入框、示例问题和空结果引导。
2. 加载状态：展示分析流程进度，包括读取 Schema、生成 SQL、安全校验、执行查询、生成解释。
3. 成功状态：展示回答、图表、表格、SQL、安全状态和执行信息。
4. 错误状态：展示失败原因、最后一次 SQL、是否经过修复、可重试入口。
5. 空数据状态：SQL 执行成功但没有返回行时，清楚说明没有匹配数据。

错误提示不能只依赖弹窗。主区需要给出用户可理解的失败原因，右侧详情栏保留技术错误信息。

## 图表策略

第一版图表不需要复杂配置面板，采用自动推断：

- 如果第一列是日期、月份或时间字段，且第二列是数值，默认折线图。
- 如果第一列是类别字段，第二列是数值，默认柱状图。
- 如果结果行数较少且是类别占比类问题，可使用饼图。
- 无法推断时隐藏图表，仅展示表格。

图表颜色应清晰但克制，避免高饱和大面积颜色。图表要服务分析，不做装饰。

## 视觉规范

建议视觉规则：

- 背景：浅灰白。
- 面板：白色或接近白色。
- 圆角：约 8px，保持专业。
- 主色：蓝绿色或清爽蓝色，小面积用于按钮、状态和重点图表。
- 状态色：安全通过用绿色，警告用琥珀色，错误用红色。
- SQL 区：浅色代码块，避免大面积黑底终端风。
- 字体层级：工作台内标题克制，不使用营销页大标题。

布局要优先保证信息密度和可读性。不要做大面积装饰性渐变、悬浮大卡片或与数据分析无关的视觉元素。

## 第一版范围

第一版实现一个完整首页即可：

- Vue 3 + Vite + Element Plus + Pinia + ECharts 工程骨架。
- 首页三栏工作台布局。
- 输入问题并调用 `/api/chat/query`。
- 展示 `answer`、`sql`、`is_sql_safe`、`columns`、`rows`、`execution_time_ms`、`retry_count`、`optimization_suggestions`。
- 提供示例问题。
- 展示查询历史，先使用前端内存保存。
- 支持 mock 数据兜底。
- 支持基本响应式布局：桌面三栏，窄屏时左侧和右侧可折叠或上下堆叠。

暂不做：

- 完整用户系统。
- 多数据源连接管理。
- 复杂图表编辑器。
- 持久化查询历史。
- BI 仪表盘拖拽布局。

这些能力可以在后续增强阶段加入。

## 验收标准

设计实现后应满足：

1. 用户打开首页即可输入自然语言问题。
2. 用户能清楚看到分析答案、图表和结果表格。
3. 用户能查看生成 SQL 与安全校验状态。
4. 接口失败时页面能展示友好的失败说明。
5. 后端接口未完成时，mock 数据仍能支撑页面预览。
6. 页面整体符合现代 SaaS / AI 工具审美，清爽、专业、易用。
