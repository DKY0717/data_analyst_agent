# 前端工作台设计方案

## 文档对齐

本设计方案是 `docs/data_analyst_agent_开发文档_v_0_2.md` 的前端展示补充，不替代主开发文档。后续实现以前端设计文档 v0.2 为主线，尤其对齐以下章节：

- 第 11.1 节：`POST /api/chat/query` API 契约。
- 第 14 节：前端页面模块、组件划分、交互流程和 Pinia 状态管理。
- 第 18 节：阶段七“前端展示”的目标和验收标准。
- 第 23 节：当前推荐开发顺序，Vue 前端展示排在 LangGraph 工作流之后。
- 第 25 节：版本规划，v0.3 对应 Vue 前端页面、SQL 展示、查询结果表格和图表展示。

因此，本文只定义前端展示层应该如何呈现，不改变后端 Agent、SQL Guard、SQL Repair、SQL Optimizer 的实现顺序和职责。

## 设计目标

前端第一版的目标是完成一个可展示、可联调、用户体验清晰的数据分析页面。它需要让用户完成以下流程：

```text
输入自然语言问题
  ↓
点击“开始分析”
  ↓
调用 POST /api/chat/query
  ↓
展示生成 SQL
  ↓
展示查询结果表格
  ↓
展示图表和自然语言解释
  ↓
展示优化建议
```

界面视觉可以现代化，但功能范围必须收敛在 v0.2 文档定义的前端展示能力内。第一版不做完整 BI 平台，不做复杂权限系统，不做多数据源管理，也不做持久化查询历史。

## 界面方向

推荐采用“现代 AI 数据分析工作台”界面。

这个方向不是营销落地页，也不是固定 BI 大屏，而是一个围绕自然语言问数结果展开的工作界面。它既符合当前大众对 AI 工具和 SaaS 工具的审美，也能突出本项目区别于普通 Text-to-SQL Demo 的核心能力：SQL 可见、安全状态可见、执行结果可见、错误和优化信息可追踪。

视觉原则：

- 清爽、专业、可信。
- 浅色背景，白色内容面板。
- 小面积蓝绿色或蓝色作为主色。
- 使用 8px 左右圆角，保持工具型界面的克制感。
- SQL 区使用浅色代码块，不使用大面积黑底终端风。
- 避免炫技大屏、装饰性渐变和过度卡片堆叠。

## 页面模块

前端首页必须覆盖 v0.2 第 14.1 节列出的模块：

1. 自然语言输入框。
2. 生成 SQL 展示区。
3. SQL 安全状态展示区。
4. 查询结果表格。
5. 图表展示区。
6. 自然语言解释区。
7. SQL 优化建议区。
8. 查询历史区，可选。

为了兼顾可用性和展示效果，页面可以组织为三栏工作台：

### 左侧：输入与历史

左侧放置用户发起查询的入口：

- 自然语言输入框。
- “开始分析”按钮。
- 示例问题快捷入口。
- 查询历史区，可选，第一版可只做前端内存状态。

示例问题应来自开发文档第 20 节，例如：

- 统计 2024 年每个月的销售额。
- 找出销售额最高的前 10 个商品。
- 分析各商品类别的退款率。
- 统计不同地区的复购率。

### 中间：结果展示

中间是用户最关注的结果区：

- 自然语言解释。
- 图表展示。
- 查询结果表格。

结果展示顺序建议是“解释优先、图表辅助、表格核对”。这样普通用户先理解答案，开发者或面试官也能继续查看数据明细。

### 右侧：SQL 与执行细节

右侧展示系统闭环中的技术信息：

- 生成 SQL。
- SQL 安全状态。
- 执行耗时。
- retry 修复次数。
- SQL 优化建议。
- 错误信息。

这些内容是本项目区别于普通问答 Demo 的关键展示点。右侧区域可以做成固定栏或折叠栏，避免普通用户被技术细节干扰。

## 组件划分

组件命名优先对齐 v0.2 第 14.2 节：

```text
Home.vue
  ├── QueryInput.vue
  ├── SQLPanel.vue
  ├── ResultTable.vue
  ├── ChartPanel.vue
  ├── AnswerPanel.vue
  ├── ExplainPanel.vue
  └── OptimizationPanel.vue
```

建议职责如下：

- `Home.vue`：首页容器，负责整体布局和组合各模块。
- `QueryInput.vue`：自然语言输入框、提交按钮、输入校验和 loading 状态。
- `SQLPanel.vue`：展示生成 SQL，并展示 SQL 安全状态。
- `ResultTable.vue`：根据 `columns` 和 `rows` 展示查询结果表格。
- `ChartPanel.vue`：根据查询结果展示 ECharts 图表。
- `AnswerPanel.vue`：展示自然语言解释和整体执行状态。
- `ExplainPanel.vue`：展示 SQL 查询逻辑、错误说明或修复说明；如果第一版后端暂未返回独立 explain 字段，可先合并到 `AnswerPanel.vue` 或保留为空状态。
- `OptimizationPanel.vue`：展示执行耗时、retry 次数和优化建议。

可选增强组件：

- `ExampleQuestions.vue`：示例问题快捷入口。
- `HistoryPanel.vue`：查询历史区。第一版仅内存保存，后续历史服务完成后再持久化。

可选组件不能阻塞 v0.2 必需组件。

## API 契约

前端通过 `POST /api/chat/query` 提交问题。

请求：

```json
{
  "question": "统计 2024 年每个月的销售额"
}
```

响应字段必须对齐 v0.2 第 11.1 节：

```json
{
  "question": "统计 2024 年每个月的销售额",
  "sql": "SELECT ...",
  "is_sql_safe": true,
  "columns": ["month", "sales"],
  "rows": [],
  "answer": "2024 年每个月销售额如下...",
  "execution_time_ms": 42,
  "retry_count": 0,
  "optimization_suggestions": []
}
```

前端内部可以在 `api/agent.js` 中做适配层，但页面组件应消费统一后的字段名。这样即使后端后续模型有小调整，也只需要改 API 适配层。

## Pinia 状态

Pinia 状态管理对齐 v0.2 第 14.4 节：

- `question`：当前输入问题。
- `loading`：是否正在请求。
- `result`：当前查询结果。
- `error`：当前错误。
- `history`：查询历史，可选。

核心方法：

- `submitQuestion()`：提交当前问题并调用 `queryAgent`。
- `clearResult()`：清空结果和错误。

如果实现示例问题选择，可额外提供 `setQuestion(question)`，但不改变主状态模型。

## 交互状态

页面至少覆盖以下状态：

1. 初始状态：展示自然语言输入框和空结果提示。
2. 加载状态：用户点击“开始分析”后，按钮 loading，结果区显示分析中。
3. 成功状态：展示 SQL、SQL 安全状态、结果表格、图表、解释和优化建议。
4. 错误状态：展示用户可理解的错误说明，并保留技术错误信息入口。
5. 空结果状态：SQL 执行成功但 `rows` 为空时，提示没有匹配数据。

错误提示不能只依赖弹窗。页面主体需要显示错误原因，方便用户理解和截图展示。

## 图表策略

v0.3 的前端目标包含图表展示，因此第一版需要提供基础 ECharts 展示能力。

建议采用简单自动推断：

- 第一列像日期或月份，第二列是数值：折线图。
- 第一列是类别，第二列是数值：柱状图。
- 结果无法推断：隐藏图表，仅展示表格。

图表不需要复杂配置面板。第一版重点是展示查询结果，而不是建设完整 BI 图表编辑器。

## 与开发顺序的关系

根据 v0.2 第 23 节，推荐顺序是：

```text
SQL Generator
  → SQL Repair
  → LangGraph 工作流
  → Vue 前端展示
  → EXPLAIN ANALYZE 与 SQL Optimizer
```

因此，前端实现有两种合理方式：

1. 按推荐顺序，在后端 `/api/chat/query` 基本可用后实现并联调。
2. 先搭建前端页面，但必须使用 mock 数据兜底，不假设后端 Agent 已经完成。

如果提前做前端，mock 数据只能用于 UI 预览，不能替代后端功能验收。最终验收仍需要接入真实 `/api/chat/query`。

## 第一版范围

第一版前端实现范围：

- Vue 3 + Vite 工程。
- Element Plus 基础控件。
- Pinia 查询状态管理。
- Axios API 请求封装。
- `POST /api/chat/query` 调用。
- 生成 SQL 展示。
- SQL 安全状态展示。
- 查询结果表格。
- ECharts 图表展示。
- 自然语言解释展示。
- SQL 优化建议展示。
- 错误信息展示。
- 查询历史区可选。
- mock 数据兜底可选，仅用于后端接口未完成时预览 UI。

第一版不做：

- 用户登录和权限系统。
- 多租户系统。
- 多数据源管理。
- 完整 BI 平台。
- 图表拖拽编辑器。
- 查询历史持久化。
- ChromaDB Schema Retriever UI。

## 验收标准

对齐 v0.2 阶段七验收标准：

1. 用户可以在页面提交自然语言问题。
2. 页面可以展示生成 SQL。
3. 页面可以展示 SQL 安全状态。
4. 页面可以展示查询结果表格。
5. 页面可以展示自然语言解释。
6. 页面可以展示错误信息。
7. 页面可以展示图表。
8. 页面可以展示 SQL 优化建议。

补充体验验收：

1. 页面整体符合现代 SaaS / AI 工具审美，清爽、专业、易用。
2. 文本在桌面和窄屏下不重叠、不溢出。
3. 后端接口暂不可用时，mock 数据只用于演示页面状态，并在 UI 中清楚标识。
4. 真实后端接口完成后，前端能通过 `/api/chat/query` 展示真实结果。
