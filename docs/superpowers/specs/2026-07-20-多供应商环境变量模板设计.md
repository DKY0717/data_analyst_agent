# Qwen、DeepSeek、MiMo 环境变量模板设计

## 背景

项目通过 OpenAI Chat Completions 兼容协议调用大模型，运行时代码历史上统一读取 `QWEN_API_KEY`、`QWEN_API_URL` 和 `QWEN_MODEL`。当前 `.env.example` 默认写入 MiMo 配置，容易让用户误以为项目只能使用 MiMo。

本次只改善环境变量模板，让用户根据自己已有的模型资源，在 Qwen、DeepSeek 和 MiMo 中任选一家。它不是运行时供应商路由功能，不新增 `LLM_PROVIDER`，也不修改 LLM 调用链。

## 方案

在 `.env.example` 的 LLM 配置区提供三组默认注释的示例：

```dotenv
# 方案一：Qwen / 阿里云百炼
# QWEN_API_KEY=your_api_key_here
# QWEN_API_URL=https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions
# QWEN_MODEL=qwen-plus

# 方案二：DeepSeek 官方 API
# QWEN_API_KEY=your_api_key_here
# QWEN_API_URL=https://api.deepseek.com/chat/completions
# QWEN_MODEL=deepseek-chat

# 方案三：小米 MiMo 按量付费
# QWEN_API_KEY=your_api_key_here
# QWEN_API_URL=https://api.xiaomimimo.com/v1/chat/completions
# QWEN_MODEL=mimo-v2.5-pro
```

MiMo Token Plan 的地址由用户套餐决定，因此模板额外保留一条注释说明：Token Plan 用户应使用控制台提供的专属完整 Chat Completions 地址，不把某个地区或账号的专属地址当作通用默认值。

用户复制 `.env.example` 为 `.env` 后，只取消一组的三行注释并填写 API Key。三组继续使用相同的 `QWEN_*` 变量名，以保持后端、Docker Compose、评测脚本和现有私有 `.env` 完全兼容。

## 官方地址依据

- Qwen：阿里云百炼 OpenAI 兼容 HTTP 地址为 `https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions`。
- DeepSeek：官方 API 使用 `https://api.deepseek.com` 作为兼容 Base URL，当前项目直接发送 HTTP 请求，因此使用完整地址 `https://api.deepseek.com/chat/completions`。
- MiMo：官方按量付费 OpenAI 兼容地址为 `https://api.xiaomimimo.com/v1/chat/completions`；Token Plan 使用控制台提供的专属 Base URL。

## 文档调整

README 的快速开始和环境变量表同步说明：

1. 三家供应商任选其一。
2. 只启用一组配置，避免重复变量由后出现的值覆盖前面的值。
3. `QWEN_*` 是历史兼容变量名，不代表只能调用 Qwen。
4. 不把真实 API Key 写入 `.env.example`、README 或 Git。

## 验证

- 更新文档一致性测试，检查三组模型名和三个完整 Chat Completions 地址都出现在 `.env.example`。
- 检查三组 API Key 均使用允许的占位值。
- 运行配置/文档专项测试、Secret Scan 和 `git diff --check`。
- 不发起真实模型请求，避免消耗用户额度。

## 不在范围内

- 不修改本地 `.env`，避免覆盖用户现有密钥和模型选择。
- 不新增供应商自动探测、故障切换或多供应商并发。
- 不修改后端 `Settings`、LLM 请求格式、超时和重试策略。
- 不验证用户账户余额、套餐地域或模型权限。
