# v0.8 权限演示工作台设计

## 背景

v0.7 已经把后端权限治理第一阶段打通：查询接口在认证启用时强制 JWT/API Key，AgentGraph 在 SQL Guard 之后、QueryRunner 之前新增 `authorize_sql`，`DataPermissionGuard` 基于最终 SQL AST 做角色级表/字段授权，审计报告包含身份摘要、权限事件和阻断规则。

当前缺口在前端演示闭环：工作台还没有登录态、角色切换、认证请求头、401 展示、权限阻断高亮，也没有把 `audit_report.user_id/auth_method/roles` 和 authorization 事件明确展示出来。结果是 v0.7 后端能力虽然真实存在，但面试演示时不够直观。

v0.8 的目标是做“权限演示工作台”，不是做完整 SaaS 用户系统。

## 目标

1. 前端能展示当前身份状态：未认证、admin、analyst、support。
2. 前端支持一键演示登录和退出，便于面试时快速切换角色。
3. 普通查询和 SSE 查询都会自动携带认证信息。
4. 认证启用且未登录时，前端能展示稳定的“需要登录”提示。
5. 权限阻断时，前端能明确展示阻断原因、阻断规则和 authorization 审计事件。
6. AuditPanel 展示 `user_id`、`auth_method`、`roles`，让权限证据完整闭环。
7. 提供一组权限演示问题，覆盖同一问题在不同角色下通过或阻断的效果。

## 非目标

1. 不做注册、密码找回、用户管理后台。
2. 不做权限策略编辑器。
3. 不做多租户或行级权限。
4. 不把 JWT_SECRET 暴露到前端。
5. 不把 demo 登录当作生产身份系统。
6. 不改变 v0.7 的核心授权链路，前端只消费后端现有结果。

## 推荐方案

采用“受开关保护的后端 demo-login + 前端 Auth Store + 工作台身份条”。

原因：

- 现有 `/api/auth/login` 只稳定返回 admin 角色，不足以演示 analyst/support。
- 前端不能自己生成 JWT，否则会暴露或复制后端签名密钥。
- API Key 当前默认 analyst，也不能演示 support/admin 三种角色差异。
- 新增 demo-login 可以让后端用现有 `create_jwt_token` 生成短期演示 JWT，前端只保存 token，不接触密钥。

## 后端设计

### 新增配置

在 `backend/app/config.py` 增加：

```python
AUTH_DEMO_ENABLED: bool = _get_bool("AUTH_DEMO_ENABLED", False)
```

`.env.example` 增加：

```bash
# Demo Auth（仅本地演示使用；生产环境保持 false）
# AUTH_DEMO_ENABLED=false
```

设计约束：

- 默认关闭。
- 只有 `AUTH_DEMO_ENABLED=true` 且配置了 `JWT_SECRET` 时，demo-login 才能返回 token。
- 如果未配置 `JWT_SECRET`，返回稳定错误，提示需要设置本地演示密钥。

### 新增接口

新增接口：

```text
POST /api/auth/demo-login
body: { "role": "admin" | "analyst" | "support" }
```

返回：

```json
{
  "code": 200,
  "message": "demo login success",
  "data": {
    "access_token": "...",
    "token_type": "bearer",
    "expires_in": 86400,
    "user": {
      "user_id": "demo:analyst",
      "auth_method": "jwt",
      "roles": ["analyst"]
    }
  }
}
```

错误行为：

- `AUTH_DEMO_ENABLED=false`：返回 404 或 403，避免生产环境暴露演示入口。
- role 不在白名单：返回 400。
- `JWT_SECRET` 未配置：返回 500 或 503，错误信息不包含环境变量值。

### 保留现有接口

- `/api/auth/status` 保留。
- `/api/auth/login` 保留 admin/password 演示入口，不作为 v0.8 前端主路径。
- `/api/auth/me` 保留，前端用于刷新当前身份。

## 前端设计

### 新增 Auth Store

新增：

```text
frontend/src/stores/auth.js
```

职责：

- 保存 `token`、`user`、`selectedRole`、`loading`、`error`。
- 从 `localStorage` 恢复演示身份。
- 提供：
  - `demoLogin(role)`
  - `logout()`
  - `loadMe()`
  - `authHeaders`
  - `isAuthenticated`
  - `currentRole`

存储策略：

- 使用 `localStorage` 保存 token 和 user，便于刷新页面后继续演示。
- 不保存密码。
- 不保存 JWT_SECRET。
- logout 时清理 token、user 和错误状态。

### 更新 API Client

更新：

```text
frontend/src/api/agent.js
```

新增能力：

- axios client 请求拦截器自动读取 auth store token，设置：

```text
Authorization: Bearer <token>
```

- `queryAgentSSE` 使用 `fetch`，也要显式带同样的 Authorization header。
- 新增 API 函数：
  - `demoLogin(role)`
  - `fetchCurrentUser()`
  - `fetchAuthStatus()`

错误处理：

- 401 统一转换为前端可读错误：`需要登录后才能查询`。
- 权限阻断不作为 HTTP 错误，而是从业务响应的 `status/audit_report.blocked_rules/events` 中展示。

### 新增 AuthBar 组件

新增：

```text
frontend/src/components/AuthBar.vue
```

放置位置：

- `Home.vue` 顶部 `status-group` 内，靠近 Session 和 SSE 开关。

展示内容：

- 未登录：
  - 标签：`未认证`
  - 角色按钮：`Admin`、`Analyst`、`Support`
- 已登录：
  - 用户 ID：如 `demo:analyst`
  - 认证方式：`jwt`
  - 角色标签：`analyst`
  - `退出` 按钮

交互：

- 点击角色按钮调用 `auth.demoLogin(role)`。
- 登录成功后提示“已切换到 analyst”。
- 退出后提示“已退出演示身份”。

### 更新 AuditPanel

更新：

```text
frontend/src/components/AuditPanel.vue
```

新增展示：

- 用户 ID
- 认证方式
- 角色列表
- authorization 阶段事件单独突出
- 阻断规则数量和具体 rule id

展示原则：

- 不展示完整权限策略。
- 不展示 token。
- authorization blocked 使用 danger 状态。
- authorization success 使用 success 状态。

### 权限演示问题

在 `frontend/src/api/agent.js` 或单独常量中新增一组权限演示问题：

```js
export const permissionDemoQuestions = [
  {
    role: 'analyst',
    question: '统计 2024 年每个月的销售额',
    expected: '允许：分析师可访问 orders.total_amount',
  },
  {
    role: 'analyst',
    question: '列出客户姓名和注册日期',
    expected: '阻断：分析师不能访问 customers.customer_name',
  },
  {
    role: 'support',
    question: '查看支付金额最高的订单',
    expected: '阻断：客服不能访问 payments 或 paid_amount',
  },
  {
    role: 'admin',
    question: '列出客户姓名和注册日期',
    expected: '允许：管理员拥有全字段权限',
  },
]
```

前端可以在 ExampleQuestions 下增加一个小分组“权限演示”，或者复用现有问题列表但增加标签。第一版推荐新增小分组，避免干扰普通分析问题。

## 数据流

### 登录流

```text
AuthBar 选择角色
  -> authStore.demoLogin(role)
  -> POST /api/auth/demo-login
  -> 保存 token + user
  -> Header 展示当前身份
```

### 查询流

```text
用户提交问题
  -> queryStore.submitQuestion
  -> queryAgent / queryAgentSSE
  -> 自动带 Authorization
  -> 后端 get_current_user
  -> AgentGraph auth_user
  -> DataPermissionGuard
  -> QueryResponse + AuditReport
  -> AuditPanel 展示身份和权限事件
```

### 401 流

```text
认证启用且未登录
  -> 后端 401
  -> API client 转换错误
  -> AnswerPanel 显示“需要登录后才能查询”
  -> History 记录失败
```

### 权限阻断流

```text
角色无权访问表/字段
  -> HTTP 200 业务响应
  -> status=blocked
  -> answer=请求已被数据权限策略阻断...
  -> audit_report.blocked_rules 包含 block_unauthorized_column/table
  -> AuditPanel 高亮 authorization blocked
```

## 验收标准

1. 未登录且认证启用时，查询展示登录提示，不进入业务结果展示。
2. admin、analyst、support 三种演示身份可以在前端快速切换。
3. 普通查询和 SSE 查询都会携带 Authorization。
4. analyst 查询允许字段成功，查询 `customers.customer_name` 被权限阻断。
5. support 查询支付金额被权限阻断。
6. admin 查询客户姓名成功。
7. AuditPanel 能展示身份摘要和 authorization 事件。
8. token 不出现在页面、日志、审计报告或测试快照中。
9. `pytest -q`、`npm run test`、`npm run build`、Secret Scan、`git diff --check` 全部通过。

## 风险与处理

### 风险 1：demo-login 被误用于生产

处理：默认关闭，必须显式设置 `AUTH_DEMO_ENABLED=true`。生产部署文档明确保持关闭。

### 风险 2：没有 JWT_SECRET 导致演示失败

处理：前端显示后端返回的稳定错误，并在 README/.env.example 说明本地演示需要设置 `JWT_SECRET=dev-demo-secret` 和 `AUTH_DEMO_ENABLED=true`。

### 风险 3：SSE 与 axios 认证行为不一致

处理：前端测试必须分别覆盖 axios 和 fetch；`queryAgentSSE` 不复用 axios，所以不能只测普通查询。

### 风险 4：权限阻断被当作网络错误

处理：权限阻断保持 HTTP 200 业务响应，前端按 `status=blocked` 和 `audit_report.blocked_rules` 展示，而不是进入通用错误态。

## 交付边界

v0.8 完成后，项目可以在面试中演示完整链路：

```text
选择 analyst
  -> 查询销售额成功
  -> 查询客户姓名被阻断
  -> AuditPanel 展示 analyst + authorization blocked
切换 admin
  -> 同一客户姓名问题成功
```

这能直接证明 v0.7 的后端权限治理不是文档能力，而是前端可交互、可审计、可复现的 Agent 权限闭环。
