# Data Analyst Agent 零基础学习教程设计

## 1. 背景

当前项目主要通过 Vibe Coding 逐步完成，已经具备数据库、FastAPI、OpenAI-compatible LLM、LangGraph、NL2SQL、安全治理、权限、评测、Vue 前端和 Docker/CI 等完整能力。项目现有两套中文材料：

- `user_docs/dev_guide/`：7 篇偏开发步骤的指南；
- `user_docs/knowledge_notes/`：12 篇偏模块说明的学习笔记。

两套材料可以作为重要素材，但还没有形成一套让零基础读者按顺序学习、动手验证并最终理解当前完整项目的系统课程。新教程参考 Hello-Agents 的在线教材形式，以“部分 → 章节 → 小节”组织内容，不把全部知识堆进一篇长文，也不拆成大量失去上下文的微型页面。

第一版教程固定以当前代码提交 `4d71b3ce84cffe175fffaffa252a9072d6e79d18` 为教学基线。后续项目优化不阻塞第一版教程；代码升级后通过版本说明、代码映射表和更新日志增量修订。

## 2. 教程目标

1. 让没有 AI Agent 项目经验的读者理解系统解决的问题、核心概念和完整数据流。
2. 让读者能够按照章节运行、验证和修改当前项目，而不是只阅读源码摘要。
3. 让读者理解每个核心模块为什么存在、输入输出是什么、如何与上下游协作。
4. 让读者掌握 NL2SQL、Agent 工作流、安全治理、评测和前后端产品化的通用方法。
5. 让读者在最后一章完成一次从环境准备到核心路径验收的完整复现。
6. 让教程可以随着项目版本持续维护，而不依赖复制多份完整源码快照。

## 3. 目标读者与前置条件

### 3.1 目标读者

- Python、数据库或前端基础薄弱的初学者；
- 使用 AI 编程完成项目，但希望真正理解代码的开发者；
- 希望学习完整 AI 应用工程链路的学生或求职者；
- 希望基于当前项目继续二次开发的维护者。

### 3.2 教学边界

教程会补充完成项目所需的 Python、HTTP、SQL、异步编程和前端基础，但不会替代完整的 Python、数据库或 Vue 入门课程。基础知识以“够理解当前代码、够完成本节实践”为标准。

## 4. 方案比较与选择

### 4.1 方案一：按现有源码模块逐文件讲解

优点是代码映射直接、编写速度快。缺点是学习顺序会被目录结构支配，初学者容易知道某个文件做什么，却不理解系统为什么逐步演进成现在的结构。

### 4.2 方案二：拆成约 76 个独立微课文件

优点是单节短、任务颗粒度小。缺点是左侧导航过长，跨文件上下文频繁中断，后续维护成本高，也不符合用户指定的 Hello-Agents 教材观感。

### 4.3 方案三：五部分、十九章、章内分节

每章对应一个完整主题和一个 Markdown 文件，内部使用 `1.1`、`1.1.1` 等编号继续拆分。每章包含理论、真实代码、运行验证、常见错误和练习；章节之间按照系统演进顺序组织。

本教程采用方案三。它既能保持教材的连续叙事，也能避免把所有知识堆进单章。配套资源只保存必要图片、示例输入和辅助文件，当前项目源码始终是实现事实来源。

## 6. 单章内容模板

每章根据主题调整细节，但必须覆盖以下内容：

1. **本章目标**：学习完成后能够解释和完成什么。
2. **学习前检查**：前置章节、环境和依赖。
3. **问题场景**：本章功能解决什么真实问题。
4. **基础概念**：只解释理解当前实现所需的知识。
5. **设计思路**：为什么采用当前方案，以及主要取舍。
6. **代码地图**：列出本章涉及的真实文件及职责。
7. **关键实现**：使用小段代码解释输入、输出和核心逻辑，禁止整文件照抄。
8. **执行流程**：使用 Mermaid、文本流程或表格连接上下游关系。
9. **动手验证**：提供可复制命令、输入、预期现象和判断标准。
10. **常见错误**：说明症状、原因、检查方式和处理办法。
11. **本章小结**：用读者应该能回答的问题收束知识。
12. **练习与扩展**：包含观察题、修改题或验证题。
13. **下一章衔接**：解释本章成果为什么需要下一章能力。

解释性文字使用清晰的中文重新组织，不大段复制旧文档。概念解释、流程说明、代码说明、注意事项和总结默认使用 Markdown 引用块；标题、表格、图片与代码块保持普通 Markdown 结构。代码、命令、SQL、JSON、YAML 和日志必须使用带语言标识的 Markdown 代码块。

## 7. 可复现策略

### 7.1 当前仓库作为事实来源

教程不创建 19 份完整项目副本。每章引用当前仓库中的真实文件，并在“代码地图”中说明路径。这样可以避免重复源码随着项目优化迅速漂移。

### 7.2 每章设置可运行检查点

每章至少提供一个不依赖猜测的验收方式，例如：

- 导入模块并观察输出；
- 运行指定 pytest；
- 请求一个 API 并核对响应字段；
- 执行数据库查询并核对结果；
- 运行前端单测或构建；
- 运行确定性评测或安全检查。

需要真实 LLM、网络、Secret 或浏览器的步骤必须明确标记条件，同时提供不依赖真实费用的替代学习路径。不能把外部依赖未运行写成“已经验证通过”。

### 7.3 最终复现方式

第十九章从干净环境出发，串联以下过程：

```text
获取代码
  → 配置 Python 与 Node.js
  → 初始化数据库
  → 配置 LLM 环境变量
  → 启动 FastAPI
  → 启动 Vue
  → 完成业务查询
  → 完成安全阻断
  → 运行测试与评测
  → 使用 Docker 验收
```

## 8. 首页与导航设计

`README.md` 作为课程首页，包含：

- 教程定位与适合人群；
- 教学基线提交和更新时间；
- 学习后能够获得的能力；
- 五部分十九章内容导航；
- 推荐学习顺序；
- 环境和费用提示；
- 当前完成状态；
- 教程更新策略。

`_sidebar.md` 只展示“部分 → 章”，不把章内所有三级标题铺到主导航。页面内部目录由 Markdown 标题自动生成。

`index.html` 提供轻量 Docsify 阅读站，支持：

- 左侧章节导航；
- 全文搜索；
- 代码高亮和复制；
- 上一章、下一章；
- Mermaid 渲染；
- 移动端阅读；
- 本地静态服务器启动。

在线站点只是阅读层，Markdown 文件即使不启动 Docsify 也必须能在 GitHub 和编辑器中正常阅读。

## 9. 代码与文档映射

课程根目录维护一份代码映射表，至少包含：

| 章节 | 核心源码 | 核心测试 | 验收命令 | 基线提交 |
|---|---|---|---|---|
| 第八章 SQL 安全 | `backend/app/security/sql_guard.py` | `backend/tests/test_sql_guard.py` | 指定 pytest | `4d71b3c` |

后续项目代码移动或重构时，先更新映射表，再更新受影响章节，避免只能依靠人工全文搜索判断文档是否过期。

## 10. 版本与更新策略

第一版课程首页和每章页尾记录：

- 教学基线版本；
- 本章最后核对日期；
- 对应源码路径；
- 对应测试路径。

课程根目录增加 `CHANGELOG.md`，按项目版本记录：

- 新增或删除的章节；
- 代码路径变化；
- 行为或接口变化；
- 尚未同步的已知差异。

后续项目优化时优先保证核心路径章节与代码一致，再更新外围工程章节。旧版教程不复制归档，重要行为变化通过 Git 历史和 Changelog 追踪。

## 11. 真实性与内容质量要求

1. 只根据当前代码、测试、配置、运行结果和已验证文档描述事实。
2. 明确区分“当前实现”“设计目标”“可选扩展”和“尚未验证”。
3. 不使用测试数量替代测试质量分析。
4. 不把确定性 Mock 测试描述成真实模型结果。
5. 不把具备生产意识描述成已经经过企业生产验证。
6. 不暴露 API Key、JWT Secret、Authorization、真实用户问题或敏感 SQL。
7. 外部概念需要引用时优先使用官方文档或原始论文，并控制引用范围。
8. 每章必须能够回答“是什么、为什么、如何实现、如何验证、哪里容易错”。

## 12. 验证与审查

### 12.1 结构验证

- 所有 `_sidebar.md` 链接存在；
- 所有章节均在首页导航和侧边栏出现一次；
- 章节编号、文件名和标题一致；
- 没有空文件、占位符或未闭合代码块；
- 本地 Docsify 可以打开首页和任意章节。

### 12.2 内容验证

- 命令在当前平台或明确声明的平台上可执行；
- 引用的源码和测试路径存在；
- 关键代码片段与当前源码一致；
- 章节预期结果与测试或真实运行证据一致；
- 每章包含目标、代码地图、验证、小结和练习。

### 12.3 两阶段复核

每批章节完成后依次进行：

1. 规格符合性复核：章节、范围、模板和可复现要求是否满足本设计；
2. 内容质量复核：技术准确性、学习顺序、中文表达、代码一致性和安全边界是否可靠。

## 13. 实施分批

为降低一次性生成大量低质量内容的风险，按以下批次实施并分别验证、提交：

1. 课程骨架：首页、侧边栏、Docsify、Changelog、代码映射表；
2. 第一部分：第 1～3 章；
3. 第二部分：第 4～7 章；
4. 第三部分上：第 8～10 章；
5. 第三部分下：第 11～12 章；
6. 第四部分：第 13～15 章；
7. 第五部分：第 16～19 章；
8. 全课程交叉链接、站点、代码路径和验收命令总复核。

每批次只提交已经完成并验证的内容，不创建空章节占位文件来制造完成假象。

## 14. 完成标准

- 课程目录、首页、侧边栏和十九章正文全部存在且内容完整；
- 每章至少包含一个可执行或可观察的验收步骤；
- 每章引用的核心源码和测试路径均能在教学基线中找到；
- 五部分形成从基础、最小系统、完整 Agent、产品化到质量与部署的连续学习路径；
- 本地阅读站可正常导航、搜索和渲染核心 Markdown 内容；
- 结构检查、链接检查、占位符检查和 Markdown 基础检查通过；
- 规格符合性复核和内容质量复核均通过；
- 工作日记记录课程基线、完成批次、提交和后续维护方式；
- 教程变更已提交，工作树中不遗留未提交的教程文件。
