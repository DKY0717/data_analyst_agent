# LLM 调用可观测性与成本评测设计

## 1. 背景

当前系统已经能够量化 NL2SQL 生成、安全拦截和 SQL Repair 能力，但仍无法回答以下生产化问题：

1. 一次分析请求调用了几次 Qwen。
2. SQL 生成、SQL Repair 和答案生成分别消耗多少 Token 与时间。
3. 修复重试增加了多少 LLM 成本。
4. 离线评测集整体需要多少 Token、耗时和估算费用。

现有 `QwenAPIClient._call_api()` 能读取 DashScope 原始响应，但只向上层返回文本内容，响应中的 usage、调用耗时和重试信息会丢失。因此需要在不改变业务结果和安全路由的前提下，建立结构化 LLM 调用可观测性。

## 2. 目标

1. 记录每次 Qwen API 调用的节点、模型、Token、耗时和 API 尝试次数。
2. 根据可配置的输入、输出 Token 单价估算调用成本。
3. 在一次 Agent 请求内汇总全部 LLM 调用，并写入 `audit_report`。
4. 在 NL2SQL 与 SQL Repair 离线评测中汇总 Token、LLM 耗时和估算成本。
5. 保证并发请求之间的调用指标互相隔离。
6. 禁止可观测性数据包含 API Key、请求头、完整 prompt 或完整原始响应。

## 3. 非目标

1. 不修改 SQL Guard、安全路由或 Repair 重试上限。
2. 不把 Qwen 价格硬编码进代码。
3. 不实现外部 APM、Prometheus、OpenTelemetry 或持久化监控平台。
4. 不把估算成本作为计费凭证。
5. 第一版不统计网络 DNS、连接建立等更细粒度阶段。

## 4. 方案选择

### 4.1 采用方案

采用“LLM Service 采集，异步上下文隔离，AgentGraph 与评测器消费汇总”的方案：

```text
Qwen API Response
  -> LLM Service 解析 usage、耗时、尝试次数
  -> ContextVar 保存当前异步请求的调用轨迹
  -> AgentGraph / Repair Evaluator 获取调用明细
  -> AuditReport / Evaluation Report 汇总展示
```

`llm_client` 是全局单例，不能使用普通实例字段保存“最后一次调用指标”，否则并发请求可能串数据。第一版使用 Python `ContextVar` 保存当前异步任务的指标列表，并在每次 Agent 请求或独立 Repair case 开始时创建新轨迹。

### 4.2 未采用方案

1. **全局进程级列表**：实现简单，但并发不安全，无法区分请求。
2. **各 Agent 节点自行计时**：只能统计节点耗时，无法获得 DashScope 返回的真实 Token usage。
3. **改变所有 LLM 方法返回类型**：类型更显式，但会大范围修改 SQL Generator、Repair 和 Answer Generator 的已有接口，第一版收益不足。

## 5. 数据结构

### 5.1 单次调用指标

每次 Qwen API 调用记录：

```python
{
    "stage": "generate_sql",
    "model": "qwen-plus",
    "input_tokens": 1200,
    "output_tokens": 180,
    "total_tokens": 1380,
    "latency_ms": 1250,
    "attempt_count": 1,
    "estimated_cost": 0.00123,
    "success": True,
    "error_type": None,
}
```

字段说明：

- `stage`：`generate_sql`、`repair_sql` 或 `generate_answer`。
- `model`：实际请求使用的模型名称。
- `input_tokens`、`output_tokens`、`total_tokens`：优先使用 DashScope usage；字段缺失时为 `0`。
- `latency_ms`：包含该次逻辑调用内的失败尝试与退避等待总时间。
- `attempt_count`：实际向 API 发起请求的次数。
- `estimated_cost`：依据环境变量单价计算；任一必要价格未配置时为 `null`。
- `success`：调用是否最终成功。
- `error_type`：最终失败时记录异常类型，不记录可能包含敏感内容的完整错误响应。

### 5.2 请求级汇总

`audit_report.llm_observability` 结构：

```python
{
    "call_count": 2,
    "input_tokens": 1800,
    "output_tokens": 300,
    "total_tokens": 2100,
    "total_latency_ms": 2300,
    "total_attempt_count": 2,
    "estimated_cost": 0.00210,
    "cost_available": True,
    "calls": [...]
}
```

若任意调用无法估算成本：

- `estimated_cost` 为 `null`。
- `cost_available` 为 `false`。
- Token、耗时和尝试次数仍正常汇总。

## 6. 组件设计

### 6.1 LLM Observability 模块

新增：

```text
backend/app/services/llm_observability.py
backend/tests/test_llm_observability.py
```

职责：

1. 使用 `ContextVar` 管理当前异步请求的调用列表。
2. 提供开始新轨迹、追加调用、读取调用和汇总指标的方法。
3. 根据环境变量中的单价计算单次调用成本。
4. 保证返回值为副本，避免调用方原地修改共享数据。

### 6.2 Qwen API Client

修改：

```text
backend/app/services/llm_service.py
backend/tests/test_llm_service.py
```

职责：

1. `generate_sql`、`repair_sql`、`generate_answer` 调用 `_call_api()` 时传入明确 `stage`。
2. `_call_api()` 使用单调时钟记录逻辑调用总耗时。
3. 成功时解析 DashScope `usage` 并记录成功指标。
4. 最终失败时记录失败指标，Token 默认为 `0`。
5. 重试过程只生成一条逻辑调用记录，`attempt_count` 反映真实 API 尝试次数。

可观测性记录不得包含：

- API Key。
- Authorization 请求头。
- 完整 messages 或 prompt。
- 完整原始响应。

### 6.3 AgentGraph 与 AuditReport

修改：

```text
backend/app/agents/graph.py
backend/app/agents/state.py
backend/app/agents/audit.py
backend/app/models/schemas.py
backend/tests/test_agent_graph.py
backend/tests/test_audit_report.py
backend/tests/test_query_api.py
```

流程：

1. `AgentGraph.run()` 开始时创建新的 LLM 指标轨迹。
2. SQL 生成、Repair 和答案生成调用完成后，从当前上下文获取最新轨迹并写入 `llm_calls` state。
3. 工作流完成后，`AuditReportBuilder` 汇总 `llm_calls`。
4. API 通过 Pydantic 模型返回 `llm_observability`。

即使 SQL 被 Guard 阻断或执行失败，已发生的 LLM 调用仍应出现在报告中。

### 6.4 NL2SQL 评测

修改：

```text
backend/evaluation/evaluator.py
backend/evaluation/report_writer.py
backend/tests/test_evaluator.py
backend/tests/test_report_writer.py
```

每条 case 从最终 `audit_report.llm_observability` 获取：

- `llm_call_count`
- `llm_total_tokens`
- `llm_latency_ms`
- `llm_estimated_cost`

汇总报告增加：

- `average_llm_call_count`
- `average_llm_total_tokens`
- `average_llm_latency_ms`
- `total_llm_estimated_cost`
- `cost_available`

若任一 case 成本不可用，报告总成本为 `null`，不使用 `0` 冒充免费。

### 6.5 SQL Repair 评测

修改：

```text
backend/evaluation/repair_evaluator.py
backend/evaluation/repair_report_writer.py
backend/tests/test_repair_evaluator.py
backend/tests/test_repair_report_writer.py
```

每条 Repair case 开始时创建独立 LLM 指标轨迹，避免多条 case 互相累计。每条结果和汇总报告增加与 NL2SQL 评测一致的 LLM Token、耗时和成本指标。

## 7. 配置设计

新增可选环境变量：

```text
QWEN_INPUT_PRICE_PER_MILLION_TOKENS=
QWEN_OUTPUT_PRICE_PER_MILLION_TOKENS=
```

规则：

1. 两个价格都配置且可解析为非负数字时，计算估算成本。
2. 任一价格缺失时，成本为 `null`。
3. 配置非法或为负数时，应用启动阶段抛出明确配置错误。
4. 成本公式：

```text
estimated_cost =
  input_tokens / 1_000_000 * input_price
  + output_tokens / 1_000_000 * output_price
```

金额保留足够精度用于汇总，展示层再格式化。

## 8. DashScope Usage 兼容

第一版优先读取常见字段：

```python
result["usage"]["input_tokens"]
result["usage"]["output_tokens"]
result["usage"]["total_tokens"]
```

兼容策略：

1. `usage` 缺失时 Token 记为 `0`，调用本身仍可成功。
2. `total_tokens` 缺失时使用输入与输出 Token 相加。
3. usage 字段无法转换为整数时按 `0` 处理，不影响业务主流程。
4. usage 解析错误不得让已成功的 SQL 生成或答案生成失败。

## 9. 错误处理

1. API 最终失败时记录 `success=false`、`error_type`、耗时和尝试次数。
2. API 重试后成功时只记录一条成功调用，`attempt_count > 1`。
3. 成本配置缺失时只影响成本字段，不影响 Token 和耗时。
4. usage 缺失或异常时 Token 为 `0`，不影响业务结果。
5. 可观测性模块自身异常必须降级处理，不能阻断 SQL 生成、Repair 或答案生成。

## 10. 测试策略

### 10.1 单元测试

1. 成功响应能够解析 input/output/total Token。
2. usage 缺失时 Token 为 `0`。
3. 重试后成功时 `attempt_count` 正确。
4. 最终失败时记录失败指标。
5. 价格完整时成本计算正确。
6. 价格缺失时成本为 `null`。
7. 两个并发异步任务的调用轨迹互不污染。
8. AuditReport 正确汇总多次调用。
9. NL2SQL 与 Repair 评测正确汇总 LLM 指标。
10. Markdown/JSON 报告正确展示成本可用和不可用两种状态。

### 10.2 回归验证

1. 完整后端 pytest 通过。
2. 现有 AgentGraph 安全路由不变。
3. 现有 32 条 NL2SQL case 和 6 条 Repair case 结构兼容。
4. `git diff --check` 通过。
5. 对生成报告扫描 API Key、Authorization 和 Bearer。

### 10.3 真实基线

使用本地 `qwen-plus` 运行：

```powershell
$env:PYTHONPATH='E:\Code\PythonCode\data_analyst_agent\backend'
python -m evaluation.evaluator
python -m evaluation.repair_evaluator
```

真实报告应展示 Token、LLM 耗时与调用次数。只有本地配置了价格时才展示估算成本。

## 11. 安全与隐私

1. 指标只保存数量、时间、模型名、节点名和错误类型。
2. 不保存 prompt、数据库 Schema 内容、查询结果、完整 LLM 响应和请求头。
3. 不在报告中写入环境变量原值或 API Key。
4. `error_type` 只保存异常类名，不保存可能包含服务端响应正文的完整异常消息。

## 12. 验收标准

1. 每次 Qwen 调用都有独立结构化指标。
2. 并发请求指标互相隔离。
3. Agent API 的 `audit_report` 包含请求级 LLM 可观测性汇总与调用明细。
4. NL2SQL 与 SQL Repair 报告包含平均 Token、平均 LLM 耗时、调用次数和成本状态。
5. 未配置价格时成本明确为 `null`，不展示为 `0`。
6. 可观测性故障不影响正常业务结果。
7. 完整后端测试通过，真实报告不包含敏感信息。
