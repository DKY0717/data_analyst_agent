# v0.7 生产化权限治理设计

## 背景

当前项目已经具备完整 NL2SQL Agent 链路：Intent Guard、分层意图解析、Schema Grounding、主动澄清、SQL Guard、执行、Repair、优化、答案生成和审计报告。v0.6 之后后端验证已稳定到 `459 passed`，但核心查询接口仍处于“认证模块存在、查询入口未强制接入”的作品级状态。

v0.7 的目标不是继续横向堆功能，而是把自然语言数据分析 Agent 往真实企业数据访问场景推进：用户必须带身份，身份映射到角色，角色决定可访问的表和字段，最终审计报告能说明这次查询是由哪个身份、按哪条权限策略放行或阻断。

## 目标

1. 查询接口支持强制认证：`/api/chat/query` 和 `/api/chat/query/stream` 在启用认证时必须验证 JWT 或 API Key。
2. 角色级数据权限：不同角色拥有不同的表级和字段级访问范围。
3. SQL 权限 Guard：在 SQL Guard 安全校验之后、SQL 执行之前，基于 SQLGlot AST 检查最终 SQL 引用的表和字段。
4. 审计闭环：权限检查结果进入 `audit_report.events`，阻断规则进入 `blocked_rules`。
5. 保持开发体验：未配置 `JWT_SECRET` 和 `API_KEYS` 时仍保持本地放行模式，方便学习和演示。
6. 保持现有主链路稳定：不重写 LangGraph，不改变 SQL 生成、Repair 和结果正确性评测的核心逻辑。

## 非目标

1. 不实现多租户数据隔离。
2. 不实现行级权限，如“只能看华东地区”。
3. 不实现管理员后台或用户管理界面。
4. 不引入外部 IAM、OAuth、Casbin 或 OPA。
5. 不把查询历史持久化纳入本次第一阶段实现。查询历史可作为 v0.7 后续阶段。

## 推荐方案

采用轻量 RBAC + SQL AST 权限 Guard。

现有 `AuthUser` 已包含 `roles`，因此 v0.7 只需要补齐三层能力：

1. API 层调用 `get_current_user`，拿到当前身份。
2. 权限策略模块根据角色返回允许访问的表和字段。
3. AgentGraph 在 `validate_sql` 之后新增 `authorize_sql` 节点，对 `validated_sql` 做权限检查。

这种方案最适合当前项目：它不打断现有 Agent 节点，不依赖模型输出可信，也能把权限作为安全治理的一部分被测试和审计。

## 角色与权限模型

第一版内置三类角色：

| 角色 | 用途 | 权限 |
|---|---|---|
| `admin` | 管理员和面试演示默认满权限用户 | 可访问全部业务表和字段 |
| `analyst` | 普通数据分析师 | 可访问订单、商品、支付、退款、地区、类目等分析表；客户表只允许非敏感字段 |
| `support` | 客服或运营辅助角色 | 可访问订单、订单明细、商品、类目和地区；禁止访问支付金额、退款金额和客户个人字段 |

字段级权限第一版按表配置，不做动态列脱敏。策略示例：

```python
ROLE_POLICIES = {
    "analyst": {
        "tables": {
            "orders": {"*"},
            "order_items": {"*"},
            "products": {"*"},
            "categories": {"*"},
            "payments": {"payment_id", "order_id", "payment_method", "payment_status", "paid_at"},
            "refunds": {"refund_id", "order_id", "refund_reason", "refund_date"},
            "regions": {"*"},
            "customers": {"customer_id", "gender", "age", "region_id", "register_date"},
        }
    }
}
```

说明：

- `*` 表示该表全部字段可访问。
- 如果 SQL 引用某张表但角色没有该表，直接阻断。
- 如果 SQL 引用 `table.column` 且字段不在允许列表，直接阻断。
- 对无法归属到具体表的裸列，第一版使用保守策略：当 SQL 只涉及一张表时按该表判断；涉及多张表且字段名无法唯一归属时阻断，避免误放行。

## 权限检查位置

权限检查放在 SQL Guard 之后、QueryRunner 执行之前：

```text
generate_sql
  -> validate_sql
  -> authorize_sql
  -> execute_sql
```

原因：

1. SQL Guard 已经保证 SQL 是单条安全查询，权限 Guard 不需要处理 DDL/DML 等危险语句。
2. 权限必须检查最终 `validated_sql`，因为 LIMIT 注入、Repair 输出和 SQL 规范化都发生在执行前。
3. Guard 拒绝的 SQL 不能进入 Repair，避免模型把无权限请求改写成其他可访问查询。

## API 行为

### 未启用认证

当 `JWT_SECRET` 和 `API_KEYS` 都为空时：

- `/api/chat/query` 保持现有行为。
- 权限 Guard 使用匿名开发用户，角色为 `admin`。
- 审计事件记录 `auth.disabled`，方便演示时解释“本地开发模式”。

### 启用认证且凭证有效

- JWT 用户使用 token 中的 `sub` 和 `roles`。
- API Key 用户默认角色为 `analyst`，除非后续通过环境变量扩展映射。
- 查询成功或阻断时，响应仍保持 HTTP 200 的业务响应风格，除非认证本身失败。

### 启用认证但缺少或无效凭证

- FastAPI 返回 HTTP 401。
- 不进入 AgentGraph，不调用 LLM，不访问数据库。
- 日志不记录 token、API Key 或原始问题。

## 审计事件

新增权限事件：

```json
{
  "stage": "authorization",
  "action": "authorize_sql",
  "status": "success",
  "message": "SQL 通过数据权限检查",
  "rule_id": null,
  "details": {
    "user_id": "user:demo",
    "roles": ["analyst"],
    "tables": ["orders", "customers"],
    "columns_checked": ["orders.total_amount", "customers.region_id"]
  }
}
```

阻断时：

```json
{
  "stage": "authorization",
  "action": "authorize_sql",
  "status": "blocked",
  "message": "当前角色无权访问字段: customers.customer_name",
  "rule_id": "block_unauthorized_column",
  "details": {
    "roles": ["analyst"],
    "table": "customers",
    "column": "customer_name"
  }
}
```

阻断状态进入 `audit_report.blocked_rules`，并且不执行 SQL Repair。

## 文件设计

### 新增 `backend/app/security/data_permission.py`

职责：

- 定义角色权限策略。
- 提供 `DataPermissionGuard.authorize(sql, user, schema)`。
- 使用 SQLGlot 提取表和列引用。
- 返回稳定结构：`is_allowed`、`reason`、`blocked_rule`、`audit_events`、`referenced_tables`、`referenced_columns`。

### 修改 `backend/app/agents/state.py`

新增字段：

- `auth_user`: 当前用户摘要，包含 `user_id`、`roles` 和 `auth_method`。
- `permission_allowed`: SQL 权限检查结果。
- `permission_error`: 权限阻断原因。

### 修改 `backend/app/agents/graph.py`

新增节点：

- `authorize_sql`：在 `validate_sql` 之后执行。

新增条件：

- `validate_sql` 安全后进入 `authorize_sql`。
- `authorize_sql` 通过后进入 `execute_sql`。
- `authorize_sql` 阻断后直接结束，不进入 Repair。

### 修改 `backend/app/api/query.py`

变化：

- 查询接口注入 `AuthUser | None`。
- 将用户摘要传给 `AgentGraph.run(...)`。
- SSE 和普通查询保持一致的认证行为。

### 修改 `backend/app/models/schemas.py`

变化：

- `AuditReport` 增加可选 `user_id`、`auth_method`、`roles` 字段。
- `QueryResponse` 不直接暴露完整权限策略，只通过审计事件展示必要证据。

### 新增测试

新增或扩展：

- `backend/tests/test_data_permission_guard.py`
- `backend/tests/test_agent_graph.py`
- `backend/tests/test_query_api.py`
- `backend/tests/test_audit_report.py`

## 测试策略

1. 单元测试权限策略：
   - `admin` 可访问全部表字段。
   - `analyst` 可查订单销售额。
   - `analyst` 查询 `customers.customer_name` 被阻断。
   - `support` 查询 `payments.paid_amount` 被阻断。
   - 多表裸列无法唯一归属时 fail-closed。

2. AgentGraph 测试：
   - SQL Guard 通过但权限 Guard 阻断时，不调用 QueryRunner。
   - 权限阻断不进入 SQL Repair。
   - 权限阻断写入 audit event 和 blocked rule。

3. API 测试：
   - 未启用认证时保持兼容。
   - 启用认证且缺少凭证时返回 401。
   - 有效 JWT 能进入查询链路。
   - SSE 查询与普通查询认证行为一致。

4. 回归验证：
   - `pytest -q`
   - `npm run test`
   - `npm run build`
   - `git ls-files -z | python scripts/check_secrets.py`

## 风险与处理

### 风险 1：LLM 生成裸列导致误阻断

处理：第一版只在多表且字段无法唯一归属时阻断。单表裸列可以根据唯一表归属判断。测试覆盖这一行为。

### 风险 2：权限策略过细影响现有评测

处理：未启用认证时默认匿名 `admin`，现有评测和本地演示不受影响。启用认证的权限行为由新增测试覆盖。

### 风险 3：API Key 用户没有角色来源

处理：第一版 API Key 用户默认 `analyst`。后续可以通过 `API_KEY_ROLES` 环境变量扩展，但不纳入本阶段。

### 风险 4：权限阻断被 SQL Repair 改写

处理：权限阻断与 SQL Guard 阻断一样直接结束，禁止进入 Repair。

## 验收标准

1. 查询接口在认证启用时缺少凭证返回 401。
2. 有效 JWT 用户能按角色权限执行查询。
3. 无权访问表或字段时，SQL 不执行、不修复、返回稳定阻断结果。
4. 权限检查事件出现在 `audit_report.events`。
5. 本地未配置认证时，现有查询、评测和测试不受影响。
6. 全量后端测试通过，前端单测和构建通过，Secret Scan 通过。
