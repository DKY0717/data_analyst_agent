# v1.2 Permission CI Quality Gate Design

## Goal

v1.2 promotes the deterministic permission evaluation added in v1.1 from a local command into a CI-enforced quality signal. Every pull request and push should prove that role permissions, row filters, and fail-closed policy behavior still work. Manual real-model evaluations should also include permission metrics in the consolidated quality gate summary.

## Context

Current CI already runs backend tests, frontend build, secret scan, deterministic intent evaluation, and deterministic intent grounding evaluation. The manual `real-qwen-evaluation.yml` workflow runs several deterministic and real-Qwen evaluations, then combines their summaries through `evaluation.quality_gate`.

v1.1 added:

- `evaluation.permission_evaluator`
- `evaluation.permission_report_writer`
- `audit_report.permission_observability`

The missing piece is enforcement. A future change could break `DataPermissionGuard` or row-filter behavior while still passing model-related evaluation. v1.2 closes that gap.

## Recommended Approach

Use a quality-gate integration rather than a standalone script-only step.

Alternatives considered:

1. **CI step only**
   - Add `python -m evaluation.permission_evaluator --json` to `ci.yml`.
   - Simple, but manual quality reports would still omit permission posture.

2. **Quality gate integration**
   - Add report-writing CLI support to `permission_evaluator`.
   - Run permission evaluation in both base CI and manual real-Qwen workflow.
   - Extend `quality_gate.py` with permission summary metrics.
   - Recommended because it makes the permission system visible and enforceable.

3. **Frontend audit dashboard**
   - More visual, but not necessary for backend safety enforcement and larger than this iteration.

v1.2 will implement approach 2.

## CLI Contract

`python -m evaluation.permission_evaluator` will keep its current behavior:

- without flags: print summary JSON and exit `0` only when passed
- `--json`: print full JSON and exit `0` only when passed

New flags:

- `--write-report`: write JSON and Markdown reports
- `--output-dir <path>`: optional report directory override
- `--timestamp <value>`: optional deterministic filename timestamp for tests

When `--write-report` is used, the evaluator writes:

- `permission-evaluation-<timestamp>.json`
- `permission-evaluation-<timestamp>.md`

## CI Design

### `.github/workflows/ci.yml`

Add a deterministic permission evaluation step to the DuckDB backend job after deterministic intent grounding evaluation:

```bash
python -m evaluation.permission_evaluator --json
```

This keeps PR safety fast and independent of secrets, LLM calls, or real databases.

### `.github/workflows/real-qwen-evaluation.yml`

Add a permission evaluation report step:

```bash
python -m evaluation.permission_evaluator --write-report
```

The step uses `EVALUATION_REPORT_DIR: ${{ runner.temp }}/qwen-evaluation`, matching existing deterministic evaluation report steps.

The quality gate step discovers:

```bash
PERMISSION_REPORT="$(find "$EVALUATION_REPORT_DIR" -name 'permission-evaluation-*.json' | sort | tail -n 1)"
```

and passes:

```bash
--permission-report "$PERMISSION_REPORT"
```

## Quality Gate Design

`evaluation.quality_gate` will require a permission summary and add checks for:

- `allowed_decision_accuracy >= 1.0`
- `blocked_rule_accuracy >= 1.0`
- `row_filter_expectation_accuracy >= 1.0`
- `authorized_sql_change_accuracy >= 1.0`

The Markdown summary will add a “Data Permission” observation row. The gate should fail when any permission metric drops below `1.0` if `--enforce` is supplied. Without `--enforce`, it writes warnings but exits `0`, matching current behavior.

## Testing

TDD coverage:

- `permission_evaluator` writes reports when `--write-report` is passed.
- `quality_gate.evaluate_quality()` fails when a permission metric is below threshold.
- `quality_gate.main()` requires and reads `--permission-report`.
- workflow tests assert both CI workflows run permission evaluation.
- workflow tests assert manual quality gate passes `--permission-report`.

Verification:

- targeted backend tests for evaluator, quality gate, workflow files
- full backend `pytest -q`
- unresolved-marker scan
- `git diff --check`
- secret scan

## Non-Goals

- No frontend UI changes.
- No new GitHub secrets.
- No real LLM calls in PR CI.
- No database-backed policy management.
- No change to permission policy semantics.

## Acceptance Criteria

- Base CI runs deterministic permission evaluation.
- Manual real-Qwen workflow writes permission evaluation reports.
- Manual quality gate includes permission metrics.
- Permission gate requires all permission metrics to be `1.0`.
- Permission evaluator can write report files from CLI.
- Tests cover workflow wiring and quality gate behavior.
- README and work diary record the new CI/quality-gate capability.
