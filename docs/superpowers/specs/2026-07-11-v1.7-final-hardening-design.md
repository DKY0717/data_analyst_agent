# v1.7 最终稳定性与证据闭环设计

## 背景

项目的核心功能、安全边界和确定性测试已经具备面试展示基础，但当前版本仍存在四类交付风险：Windows 下全量 pytest 不能稳定结束、SSE 进度回调异常被静默忽略、本地提交尚未经过远端完整 CI、真实模型报告没有绑定当前 HEAD。与此同时，`graph.py` 等模块体积较大，但面试前全面重构会扩大回归面。

本轮采用低风险收口策略：只重构与上述风险直接相关的职责，保持 LangGraph 节点顺序、共享状态、SQL Guard 前置校验和最多 3 次 Repair 约束不变。

## 目标

1. 让 `pytest backend/tests -q` 在 Windows 本地环境中能够稳定结束并输出完整结果。
2. 让同步和异步 SSE 进度回调失败可观测，同时不影响主查询、不记录回调参数或异常原文。
3. 把进度通知职责从 `AgentGraph` 中抽成可独立测试的小模块，降低主图的非业务职责。
4. 在独立分支上推送当前版本，并让 GitHub Actions 对同一 HEAD 完成全部确定性门禁。
5. 对同一 HEAD 运行真实模型手动工作流，核验报告元数据和安全审计 artifact。

## 非目标

- 不全面拆分 `graph.py`、`llm_service.py` 或 `data_permission.py`。
- 不修改 LangGraph 节点、条件边或 Repair 重试语义。
- 不改变 SQL 生成、权限改写或数据库执行结果。
- 不为了本机缺少 Docker 而降低容器、PostgreSQL 或 secure profile 门禁。
- 不把教程任务尚未提交的工作日志内容夹带到本轮提交。

## 方案比较

### 方案 A：低风险收口（采用）

定位测试卡顿根因，抽取进度回调边界，补齐针对性测试，然后通过远端 CI 和真实模型 workflow 完成交付证据。改动面最小，适合面试前冻结版本。

### 方案 B：中等职责拆分

除进度通知外，再拆分 LLM transport、图路由和权限策略执行器。长期维护性更好，但会触及大量 mock、导入路径和集成测试，不利于当前版本快速形成可信证据。

### 方案 C：全面架构重构

重写三个超大模块并重新定义边界。长期收益最高，但当前缺少与风险相匹配的业务需求，不符合本轮 YAGNI 原则。

## 设计

### 1. 全量测试稳定性

先使用可复现的收集顺序、逐文件执行和超时诊断定位第一个不释放资源的测试。重点检查 DuckDB 连接、SQLite session store、后台线程、全局单例和 pytest fixture 终结器。

修复必须落在资源所有者处：连接由创建者关闭，线程或执行器在 fixture 结束时回收，测试数据库按进程隔离。禁止通过永久拆分 CI 测试、跳过测试或简单增加无限等待来掩盖问题。

验收标准：

- `pytest backend/tests -q` 在 Windows 上连续完成，不残留 pytest Python 进程。
- CI 仍使用单条全量命令并保留 75% 覆盖率门槛。
- 修复前能稳定复现或用测试证明对应资源泄漏，修复后有回归测试。

### 2. SSE 进度回调边界

新增独立的进度通知模块，提供同步和异步调用入口。模块负责：

- 回调不存在时立即返回。
- 同时支持普通函数和 awaitable 返回值。
- 回调失败时只记录 `stage` 与异常类型，不记录异常原文、回调参数或查询内容。
- 回调失败不改变 Agent 主流程结果，因为进度通知属于旁路能力。

`AgentGraph` 仅调用该模块，不再自行维护重复的 try/except。新增文件按项目规范添加中文关键步骤注释。

### 3. 维护性边界

本轮只抽取进度通知这一项可独立理解、独立测试且与现有问题直接相关的职责。三个大模块的进一步拆分写入 v1.8 技术债，包括候选边界、依赖方向和触发条件；不在 v1.7 冻结前实施无业务收益的结构变化。

### 4. 分支与远端 CI

所有新增修改位于 `codex/v1.7-final-hardening`。提交按“测试稳定性”“进度通知”“文档/证据”分组，每次提交前执行规格符合性和代码质量自审。

推送后，以分支 HEAD SHA 查询 GitHub Actions。必须确认以下门禁成功：后端全量与覆盖率、PostgreSQL migration round trip、前端测试/lint/build、Playwright、Compose 与镜像 readiness、Python/Node 依赖审计、Secret Scan 和确定性评测。

任何 CI 失败都在同一分支修复并重新验证，不以旧提交的成功记录替代。

### 5. 真实模型证据

基础 CI 全绿后，手动触发 `real-qwen-evaluation.yml`。工作流必须使用分支当前 HEAD，并产出包含以下字段的 artifact：

- HEAD SHA
- Provider 与模型名
- 脱敏端点
- case pack 版本
- UTC 生成时间
- smoke、质量门禁和安全审计报告

如果仓库缺少有效 Secret、模型服务不可用或额度不足，应明确报告外部阻塞，不伪造或复用旧 artifact。成功后使用 `scripts/interview_evidence.py` 生成面试证据清单并核对 SHA。

## 错误处理与隐私

- 测试诊断日志不得输出 API Key、JWT Secret、SQL 原文或用户问题原文。
- 进度回调异常只记录异常类型。
- CI 和真实模型工作流失败必须保留 run URL、失败 job 和当前 SHA，便于复盘。
- 外部服务失败不回滚已经验证通过的本地修复，但不能把目标标记为完全完成。

## 验证矩阵

| 层级 | 验证 |
|---|---|
| 单元 | 进度通知同步、异步、awaitable、失败日志脱敏 |
| 稳定性 | Windows 单命令全量 pytest 完成且无残留进程 |
| 静态 | Ruff、Secret Scan、`git diff --check` |
| 前端 | ESLint、58 个单测、生产构建 |
| 核心路径 | 15 条可执行 Agent 核心路径 |
| 远端 | 当前 HEAD 的完整 GitHub CI 全绿 |
| 真实模型 | 当前 HEAD 的手动 workflow 成功并核验 artifact |

## 完成定义

只有同时满足以下条件才算完成：

1. 本地针对性测试和 Windows 全量测试通过。
2. 规格符合性与代码质量复核通过。
3. 所有本轮改动已提交且工作树只保留用户原有的教程日志改动。
4. 分支已推送，当前 HEAD 的基础 CI 全绿。
5. 当前 HEAD 的真实模型评测成功，或明确记录无法由代码修复的外部阻塞。
