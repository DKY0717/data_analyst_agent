# v0.6 分层分析意图、Schema Grounding 与主动澄清设计

## 1. 目标与研究定位

v0.6 将当前“安全意图检查后加载完整 Schema”的链路升级为可解释、可分层评测的分析意图理解系统。系统先表达用户真正需要的业务概念，再将概念映射到物理 Schema，通过关系图完成表路由，并根据风险决定直接执行、扩大候选或主动澄清。

v0.6 的研究主线是结构化分析意图与 Schema Grounding；表路由和 Join 图推理是下游工程能力；端到端联合模型作为后续对照基线，不作为首版主架构。

严格发布原则：任一核心验收指标未达标，禁止发布 `v0.6.0`。

## 2. 系统边界

主链路采用分层架构：

```text
Safety Intent Guard
  → Analysis Intent Parser
  → Schema Grounder
  → Schema Graph Router
  → Clarification Risk Decision
      → 需要澄清：返回候选并等待用户回答
      → 无需澄清：构建精简 Schema Context
  → SQL Generator
  → SQL Guard
  → SQL Executor / SQL Repair
  → Answer Generator
```

各层边界如下：

- Safety Intent Guard 继续作为唯一安全入口，只判断请求是否允许进入分析流程。
- Analysis Intent Parser 只表达业务需求，不直接选择物理表。
- Schema Grounder 将业务概念映射到字段、指标表达式和物理候选。
- Schema Graph Router 选择覆盖 Grounding 候选的最小连通 Schema 子图。
- Clarification Risk Decision 综合槽位缺失、候选差距、Join 风险和错误代价做决策。
- Schema Context Builder 仅向 SQL Generator 提供必要表、字段、关系和业务语义。
- 每层保存候选、分数、证据和决策原因，支持评测、审计和消融实验。

## 3. 技术路线

采用可解释混合分层方案：

```text
规则解析器 + LLM 意图解析器
→ 候选合并与结构校验
→ Schema Grounding
→ 关系图路由
→ 风险校准与主动澄清
```

规则解析器处理时间范围、时间粒度、排序和 Top-N 等高确定性槽位。LLM 解析器处理复杂业务指标、多意图和隐式表达。合并器不得静默覆盖冲突；冲突必须保留为候选并降低置信度。

方案对照：

- 纯 LLM 分层方案作为主要实验基线。
- 禁用规则解析器、禁用图推理、禁用主动澄清作为消融实验。
- 专用意图模型和端到端联合模型在数据集积累后进入后续版本。

## 4. 结构化分析意图

统一结构化意图示例：

```json
{
  "task_types": ["aggregation", "ranking"],
  "metrics": [
    {
      "concept": "sales_amount",
      "aggregation": "sum",
      "confidence": 0.96,
      "evidence": "销售额"
    }
  ],
  "dimensions": [
    {
      "concept": "region",
      "confidence": 0.94,
      "evidence": "各地区"
    }
  ],
  "filters": [
    {
      "concept": "order_date",
      "operator": "year_equals",
      "value": 2024,
      "confidence": 0.99
    }
  ],
  "ranking": {
    "direction": "desc",
    "limit": 5
  },
  "missing_slots": [],
  "overall_confidence": 0.95
}
```

结构要求：

- `task_types` 支持多标签，包括聚合、趋势、排名、占比、同比、环比、留存、复购、异常检测和归因分析。
- `metrics`、`dimensions`、`filters` 分别保存概念、置信度和证据。
- 时间范围、粒度、排序、Top-N 和比较关系使用独立槽位。
- `missing_slots` 记录执行分析所需但未提供的信息。
- 多轮追问保存合并后的完整结构化意图，不依赖历史文本简单拼接。

## 5. Schema Grounding

Grounding 为每个业务概念保留一个或多个物理候选：

```json
{
  "concept": "sales_amount",
  "candidates": [
    {
      "candidate_id": "sales_by_order_total",
      "expression": "SUM(orders.total_amount)",
      "tables": ["orders"],
      "columns": ["orders.total_amount"],
      "score": 0.86,
      "evidence": ["语义指标定义", "字段别名匹配"]
    },
    {
      "candidate_id": "sales_by_item_amount",
      "expression": "SUM(order_items.quantity * order_items.unit_price)",
      "tables": ["order_items"],
      "columns": ["order_items.quantity", "order_items.unit_price"],
      "score": 0.82,
      "evidence": ["语义指标定义"]
    }
  ]
}
```

约束：

- `candidate_id` 必须稳定，用于澄清、评测、恢复和审计。
- Grounding 分数综合概念别名、字段匹配、语义指标定义、类型兼容性和上下文证据。
- 业务口径冲突不得静默选择；候选接近时提高澄清风险。
- 无候选时不得猜测字段，必须进入澄清或稳定失败。

## 6. Schema 图路由

Router 根据 Grounding 结果选择覆盖全部必需概念的最小连通子图：

```json
{
  "selected_tables": ["orders", "customers", "regions"],
  "join_edges": [
    ["orders.customer_id", "customers.customer_id"],
    ["customers.region_id", "regions.region_id"]
  ],
  "evidence": {
    "orders": ["指标 sales_amount 的 Grounding 表"],
    "regions": ["维度 region 的 Grounding 表"],
    "customers": ["连接指标表与维度表所需的桥接表"]
  },
  "confidence": 0.97
}
```

路由目标函数同时考虑：

- Grounding 概念覆盖率。
- Join 图连通性。
- 路径长度和桥接表数量。
- 无关表惩罚。
- 表与边的业务先验。

Router 必须输出选择证据。无法生成连通子图时不得进入 SQL 生成。

## 7. 风险决策与主动澄清

澄清风险综合以下因素：

```text
clarification_risk =
  Grounding 候选接近程度
  + 业务口径歧义
  + 缺失必要槽位
  + Join 路径不确定性
  + 错误执行代价
```

决策分为三档：

- 低风险：构建 Schema Context 并继续执行。
- 中风险：扩大候选范围并重新推理一次。
- 高风险：创建主动澄清请求。

澄清请求示例：

```json
{
  "clarification_id": "clarify_sales_amount_001",
  "reason": "metric_definition_ambiguity",
  "question": "销售额采用哪种计算口径？",
  "options": [
    {
      "candidate_id": "sales_by_order_total",
      "label": "订单总金额",
      "description": "按 orders.total_amount 汇总"
    },
    {
      "candidate_id": "sales_by_item_amount",
      "label": "商品明细金额",
      "description": "按数量乘商品成交单价汇总"
    }
  ],
  "max_rounds": 2
}
```

交互同时支持候选按钮和自由文本回答。自由文本必须归一化为已有稳定 `candidate_id`。无法归一化时允许再次澄清；最多两轮，超过后返回稳定错误，不猜测执行。

澄清恢复流程：

```text
用户点击候选或输入自由文本
→ 重新执行 Safety Intent Guard
→ 将答案归一化为 candidate_id
→ 合并回冻结的 AnalysisIntent
→ 重新 Grounding 与路由
→ 再次执行风险决策
→ 继续执行或停止
```

## 8. Schema Context 构建

Context Builder 只提供：

- 路由选中的表和必要字段。
- 主键、外键与选中的 Join 路径。
- 当前问题相关的业务指标定义。
- 相关稳定输出别名和必要类型信息。

低风险时使用裁剪 Context。中风险只能扩大一次候选范围。Context Builder 若丢失必需表、字段或 Join 边，必须 fail-closed，不进入 SQL Generator。

SQL Repair 默认复用原裁剪 Context；只有执行错误明确表明 Schema 信息不足时，才允许扩大一次 Context，且修复结果仍必须重新经过 SQL Guard。

## 9. LangGraph 与状态模型

主链路调整为：

```text
check_intent
→ parse_analysis_intent
→ ground_schema
→ route_schema
→ assess_clarification_risk
    ├─ clarify → END
    ├─ expand_candidates → ground_schema
    └─ build_schema_context
→ generate_sql
→ validate_sql
→ execute_sql
→ repair_sql / generate_answer
```

`AgentState` 新增：

```text
analysis_intent
grounding_result
schema_route
routing_confidence
clarification_risk
clarification_request
clarification_round
resolved_candidate_ids
schema_context
```

现有安全约束保持不变：

- Safety Intent Guard 始终为唯一入口。
- SQL 必须经过 SQL Guard 后才能执行。
- SQL Repair 后必须再次经过 SQL Guard。
- SQL Repair 最多三次。

## 10. API 与向后兼容

现有 `/api/chat/query` 保持可用。明确问题仍通过一次请求直接返回结果；只有需要澄清时返回：

```json
{
  "status": "clarification_required",
  "clarification": {
    "clarification_id": "...",
    "question": "...",
    "options": []
  }
}
```

澄清回答请求包含：

```json
{
  "session_id": "...",
  "clarification_id": "...",
  "candidate_id": "...",
  "answer_text": "按订单总金额"
}
```

系统设置特性开关，可回退到当前完整 Schema 流程。v0.5 的 10 条黄金问题必须通过新链路并保持结果正确率 `10/10`。

## 11. 模块划分

新增模块：

```text
backend/app/analysis_intent/
├── models.py
├── rule_parser.py
├── llm_parser.py
├── merger.py
├── clarification.py
└── normalizer.py

backend/app/schema_context/
├── metadata_catalog.py
├── grounder.py
├── graph_router.py
├── context_builder.py
└── risk_calibrator.py
```

职责：

- `models.py`：定义跨层稳定数据契约。
- `rule_parser.py`：解析时间、排序、Top-N 等高确定性槽位。
- `llm_parser.py`：解析复杂指标、多意图与隐式表达。
- `merger.py`：合并候选、保留冲突并计算置信度。
- `clarification.py`：创建澄清请求并恢复冻结意图。
- `normalizer.py`：将按钮或自由文本回答映射到稳定候选 ID。
- `metadata_catalog.py`：构建物理 Schema、业务语义和关系图目录。
- `grounder.py`：完成业务概念到物理候选的映射。
- `graph_router.py`：计算最小连通 Schema 子图。
- `context_builder.py`：生成裁剪后的 SQL Generator Context。
- `risk_calibrator.py`：计算风险并执行三档决策。

## 12. 错误处理与审计

- Analysis Intent 解析失败：返回稳定错误，不进入 SQL 生成。
- LLM 解析失败但规则结果完整：降级使用规则结果并记录事件。
- Grounding 无候选：触发澄清或稳定失败。
- Router 无法生成连通子图：触发澄清或稳定失败。
- 中风险候选：最多扩大一次候选范围。
- 高风险候选：必须澄清。
- 澄清超过两轮：停止任务。
- 所有层记录稳定错误类型、候选摘要、分数和决策原因。
- 审计信息不得记录密钥、完整敏感输入或模型内部推理文本。

## 13. 数据集与标注规范

v0.6 使用项目内研究增强数据集，后续 v0.7 再适配 Spider、BIRD 等公开数据集进行跨领域泛化对比。

数据集划分：

| 集合 | 用途 | 规模 |
|---|---|---:|
| 开发集 | 规则、Prompt 与阈值调试 | 约 200 条 |
| 校准集 | 置信度校准与澄清风险阈值选择 | 约 100 条 |
| 冻结测试集 | 最终严格验收 | 至少 100 条 |

生成与冻结流程：

```text
模型生成候选
→ 自动格式与一致性校验
→ 人工审核、修正并定稿
→ 冻结测试集
→ 计算内容哈希，防止静默修改
```

每条样本至少标注：

- 结构化 Analysis Intent。
- 正确 Grounding candidate ID。
- 必需表和可接受附加表。
- 必需 Join 边。
- 是否需要澄清。
- 可接受澄清候选与标准恢复意图。

冻结测试集覆盖：

- 单表和跨表分析。
- 桥接表与间接 Join。
- 多指标、多维度和多意图。
- 时间范围、粒度、同比、环比、Top-N、排序、占比和复购。
- 指标、维度和过滤条件歧义。
- 不需澄清但表达较模糊的问题。
- 候选按钮和自由文本澄清回答。
- 50 至 100 张干扰表环境。

## 14. 分层评测与消融实验

评测指标：

| 层级 | 核心指标 |
|---|---|
| 分析意图 | Task Macro-F1、Slot-F1、Intent Exact Match |
| Grounding | Recall@1、Recall@3、MRR |
| 表路由 | Table Precision、Recall、F₂ |
| 图推理 | Join Path Exact Match、Edge-F1 |
| 风险校准 | ECE、Brier Score |
| 主动澄清 | 必要澄清召回率、不必要澄清率 |
| 澄清恢复 | 候选归一化准确率、恢复意图 Exact Match |
| Context 构建 | Schema Token 减少率、必需字段召回率 |
| 端到端 | SQL 执行率、结果正确率 |

消融实验：

```text
完整分层系统
对比：移除规则解析器
对比：移除图推理
对比：禁用主动澄清并直接选择最高分候选
```

主动澄清必须证明澄清后的最终结果正确率高于禁用澄清基线。
