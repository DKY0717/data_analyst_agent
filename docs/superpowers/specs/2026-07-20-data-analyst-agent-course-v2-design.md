# Data Analyst Agent 全新学习课程 v2 设计

## 1. 背景

当前项目主要通过 Vibe Coding 完成。代码已经形成较完整的 Data Analyst Agent，但项目所有者缺少对开发顺序、模块职责、关键取舍、失败处理和验证证据的系统理解。

仓库已有 `user_docs/data-analyst-agent-course/` 十九章课程，其最后系统核对时间为 2026-07-11。当前代码基线为提交 `8dffc1d76c514c7efe1b6e642ea1880a81989109`，2026-07-12 至 2026-07-18 又加入了进度通知拆分、答案生成失败隔离、权限和语义修复、真实模型失败证据、评测分片、原子 checkpoint、严格汇总及新的 GitHub Actions 证据流。旧课程不再作为新课程的内容来源或结构约束。

本设计建立一套完全独立的新课程。课程从当前代码反向设计一条适合学习的“从零重建”路径，使学习者最终能够讲清楚项目、独立调试和扩展项目，并从零重写精简版核心 Agent。

## 2. 学习者与约束

### 2.1 学习者基础

- 掌握 Python 基础语法和简单 SQL。
- 尚未系统使用 FastAPI、Vue、LangGraph、NL2SQL 和模型评测。
- 每天可以投入 1～2 小时。
- 项目代码不是学习者逐步手写完成的，不能假设其了解已有模块的产生原因。

### 2.2 学习目标

课程按三个递进层级验收：

1. **能讲清楚**：能够脱离文档画出架构、解释请求链路和主要工程取舍。
2. **能独立修改**：能够定位错误、增加业务指标、修改工作流，并补齐测试和评测。
3. **能从零重写**：能够不复制现有实现，独立完成精简版 NL2SQL Agent 的核心闭环。

### 2.3 时间预算

- 总周期为 8 周。
- 每周建议学习 5 天，每天 1～2 小时。
- 普通知识章节控制在一个学习时段内；综合实验可以跨两个学习时段。
- 学习完成以可执行验收为准，不以阅读页数为准。

## 3. 方案选择

采用“当前代码重建主线”，不采用旧十九章增量修订，也不按全部 Git 提交逐条考古。

旧课程增量修订会继续继承已经发生漂移的章节结构；逐提交考古会把文档调整、反复修复和中间失败全部暴露给初学者，学习主线容易被噪声打断。当前代码重建主线以现有实现为事实来源，按照依赖关系把完整系统还原成从简单到复杂的构建过程，同时在相关章节加入真实历史故障作为案例。

## 4. 输出位置与独立性

新课程写入独立目录：

```text
user_docs/data-analyst-agent-course-v2/
```

旧目录 `user_docs/data-analyst-agent-course/` 不修改、不引用，也不作为新课程的前置阅读材料。

新课程至少包含以下总览文件：

| 文件 | 职责 |
|---|---|
| `README.md` | 课程入口、学习方法、八周路线和完成标准 |
| `CURRENT-CODE-MAP.md` | 当前源码、配置、测试、评测与章节的双向映射 |
| `STUDY-CHECKLIST.md` | 每日任务、运行证据、复述和掌握度记录 |
| `INTERVIEW-QUESTIONS.md` | 分模块面试题、回答要点和项目边界 |
| `TROUBLESHOOTING.md` | 环境、LLM、数据库、测试、前端和 CI 常见故障 |
| `CHANGELOG.md` | 课程基线和后续代码同步记录 |

## 5. 课程结构

课程共七部分、三十二章。章节编号和依赖顺序固定，正文文件按部分分目录保存。

### 5.1 第一部分：学会阅读和运行项目

1. 项目解决什么问题与完整架构
2. Python 项目结构、类型模型、异步与依赖注入
3. SQL、DuckDB 与八张电商业务表
4. 开发环境、配置加载、日志和调试方法

本部分验收：能够启动确定性环境，解释前后端、数据库和 LLM 的边界，并沿入口定位一次请求。

### 5.2 第二部分：从零搭建最小 NL2SQL

5. 数据初始化、连接管理与 Schema Loader
6. FastAPI 请求、响应和异常边界
7. OpenAI-compatible LLM 客户端与结构化输出
8. SQL Generator、QueryRunner 与 Answer Generator

本部分验收：能够使用精简代码完成“自然语言问题 → SQL → DuckDB → 答案”的最小闭环，并说明它为什么还不安全。

### 5.3 第三部分：把流水线升级为 Agent

9. 规则解析、LLM 解析与结构化 Analysis Intent
10. 语义层、元数据目录与业务指标口径
11. Schema Grounding、JOIN 路由与主动澄清
12. AgentState、LangGraph 节点、条件边与回流
13. SQL Repair、Optimizer、SessionStore 与多轮上下文

本部分验收：能够画出状态变化和条件路由，新增一个节点或修改一条边，并用测试证明没有破坏失败路径。

### 5.4 第四部分：安全、权限与可靠性

14. Intent Guard 与 SQLGlot SQL Guard
15. 查询沙箱、只读执行、LIMIT 与超时控制
16. 认证、角色权限、字段阻断与行级 SQL 改写
17. 网络重试、业务重试、答案失败隔离与降级
18. 审计报告、LLM 可观测性和脱敏边界

本部分验收：能够解释危险意图、危险 SQL、越权 SQL 和执行错误分别在哪一层处理；修复后的 SQL 必须重新经过安全和权限检查。

### 5.5 第五部分：把后端能力变成产品

19. Query API、SSE 进度、取消与缓存
20. Vue、Pinia、Axios 和三栏工作台
21. 图表、表格导出、公式注入防护与审计面板
22. Docker、Nginx、readiness 与 demo/secure 部署边界

本部分验收：能够启动完整产品，观察成功、危险、越权和多轮四类路径，并区分前端展示与后端安全判定。

### 5.6 第六部分：测试、评测与 CI 证据

23. pytest、Vitest、Playwright 与测试金字塔
24. NL2SQL、Intent、Grounding 和权限评测
25. SQL Repair 故障注入与黄金结果正确性
26. 真实模型分片、原子 checkpoint 与严格汇总
27. GitHub Actions、质量门禁、安全审计和证据绑定
28. MiMo 空 content、长尾延迟与 75 分钟超时事故复盘

本部分验收：能够区分单元测试、核心路径、结构化评测和真实模型证据；能够解释为什么缺片必须 fail closed，以及为什么一次红色 run 不等于全部业务能力失败。

### 5.7 第七部分：独立开发与面试毕业

29. 独立增加一个业务指标的完整闭环
30. 综合故障定位实验：从现象到根因和回归测试
31. 不复制源码重写精简版 Data Analyst Agent
32. 架构答辩、项目边界、STAR 故事与模拟面试

本部分验收：完成实现、测试、评测、文档和口头答辩五类成果，并能在无外部 LLM 时使用确定性替身证明核心编排。

## 6. 单章教学契约

每章必须使用一致结构，不能只介绍文件或粘贴源码：

1. 本章学习目标。
2. 前置知识和预计时间。
3. 要解决的真实问题。
4. 模块输入、输出和依赖。
5. 执行流程或状态变化。
6. 当前代码地图。
7. 关键代码片段与逐段解释。
8. 最小动手运行。
9. 故意制造一个错误并观察现象。
10. 调试路径和常见误判。
11. 独立编码练习。
12. 测试或评测验证。
13. 面试复述题。
14. 掌握度检查和下一章衔接。

解释性正文使用中文。代码只保留理解知识点所需片段，并标明真实源码路径和符号名称，不整文件复制。命令必须注明工作目录、是否访问网络、是否调用真实模型、是否可能产生费用。

## 7. 学习数据流

课程把每个主题组织为一个可重复学习闭环：

```text
问题场景
  → 建立行为预期
  → 定位入口与测试
  → 阅读当前实现
  → 运行最小验证
  → 注入故障
  → 根据日志定位
  → 完成小改造
  → 补充测试与评测
  → 口头复述
```

阅读顺序优先采用“测试 → 接口 → 实现 → 集成流程”。测试先说明输入和预期，再进入实现细节；复杂模块同时提供调用链和状态字段变化，避免按目录机械阅读。

## 8. 练习与答案边界

练习分为三类：

- **观察题**：运行命令并解释输出，不修改代码。
- **修改题**：在当前项目中完成边界明确的小改造并补测试。
- **重建题**：在独立学习目录中从空文件实现精简版本，不复制生产代码。

正文提供验收条件和提示，不直接紧跟完整答案。参考思路集中放在章末折叠小节或独立答案文件中，学习者先提交自己的解释或实现，再对照参考。

所有真实 LLM 练习必须有不调用外部模型的确定性替代方案。未经明确确认，不自动发送问题、Schema、Prompt 或评测数据到第三方模型。

## 9. 故障与真实性处理

课程必须明确区分以下失败：

| 失败类型 | 教学处理 |
|---|---|
| 环境或工作目录错误 | 给出检查命令和正确入口，不修改业务代码绕过 |
| 数据库或 Schema 错误 | 展示连接、初始化和元数据检查路径 |
| LLM 认证、网络、限流或空响应 | 标记外部调用边界，保留脱敏错误，不归因于 SQL 逻辑 |
| Agent 业务错误 | 沿 state、节点和条件边定位，使用确定性测试复现 |
| SQL 安全或权限阻断 | 根据预期区分正确阻断和误杀 |
| 评测证据缺失 | 标记为未验证，严格汇总 fail closed，不伪造通过率 |

章节引用历史运行结果时，必须同时注明提交、模型、运行时间和证据类型。旧报告不能自动证明当前代码质量。MiMo 事故章节以 run `29634864907` 为案例，说明 8/13 NL2SQL 分片成功、5/13 超时，Repair 3/3 和 Correctness 5/5 成功，最终严格汇总因 NL2SQL 证据不完整而失败。

## 10. 当前代码覆盖机制

`CURRENT-CODE-MAP.md` 同时维护两个方向：

1. 章节到源码、配置、测试、评测和命令的映射。
2. 核心模块到负责解释它的章节映射。

首版必须覆盖以下核心区域：

- `backend/app/agents/`
- `backend/app/analysis_intent/`
- `backend/app/db/`
- `backend/app/security/`
- `backend/app/semantic/`
- `backend/app/schema_context/`
- `backend/app/services/`
- `backend/app/api/`
- `backend/evaluation/`
- `backend/tests/`
- `frontend/src/`
- `frontend/tests/` 与 `frontend/e2e/`
- `database/` 与 `backend/alembic/`
- `.github/workflows/`
- Docker、Compose、Nginx 和项目配置

课程基线记录当前 HEAD、核对日期和主要测试计数。代码基线变化后，先执行覆盖审计，再更新受影响章节和 `CHANGELOG.md`，不能只修改首页版本号。

## 11. 文档验证设计

课程实现阶段补充确定性文档测试，至少验证：

- 三十二章文件完整且目录链接有效。
- 总览文件存在并能互相导航。
- 课程引用的本地源码、配置和测试路径存在。
- 每章包含目标、代码地图、动手验证、故障实验、练习、面试复述和掌握度检查。
- 命令块不包含真实 API Key、Token、密码或 Authorization 值。
- 当前代码核心区域都能在 `CURRENT-CODE-MAP.md` 找到负责章节。
- README、代码地图和章节记录相同基线提交。
- 文档没有把未运行、外部失败或缺失报告表述为已通过。

课程命令按风险分级：确定性命令可以作为常规验收；启动服务、Docker、浏览器和真实 LLM 命令标注环境条件，由学习者主动执行。

## 13. 完成标准

课程完成必须同时满足：

- 不看文档画出完整系统架构和关键失败分支。
- 使用自己的语言解释十二个 Agent 节点及主要 state 字段。
- 独立完成一个新业务指标，从语义定义贯通到前端或报告，并补测试和评测。
- 独立定位一个预设故障，提交根因、修复和回归证据。
- 从零实现精简版 NL2SQL Agent，至少包含生成、只读 Guard、执行、失败返回和确定性测试。
- 完成一份五分钟项目介绍、一份十分钟架构讲解和一轮常见追问。
- 能如实说明项目当前真实模型长尾问题和未经过企业生产验证的边界。

## 14. 非目标

- 不把课程写成通用 Python、SQL、Vue 或 LangGraph 百科。
- 不逐提交复述全部 Git 历史。
- 不复制旧十九章正文或以其为前置依赖。
- 不在正文中粘贴完整生产文件。
- 不要求学习者记住全部 717 个后端测试。
- 不把确定性 Fake LLM 证据冒充真实模型质量。
- 不宣称项目已经经过企业生产规模验证。
- 不在课程生成过程中自动调用或付费使用第三方 LLM。

## 15. 验收条件

- 新课程目录独立存在，旧课程无修改。
- 七部分三十二章及五个总览文件全部完成。
- 每章符合单章教学契约，并至少包含一个可执行的确定性验证或明确的观察任务。
- 当前核心代码、测试、评测、前端和交付配置都在代码覆盖映射中有归属。
- 文档契约测试通过，引用路径和导航链接无失效。
- 八周清单可以逐日勾选，并记录实际命令、结果和未执行原因。
- 第29～32章形成可检查的毕业成果，而不是只提供阅读材料。
- 项目后续变化时，可以通过基线检查、覆盖审计和课程更新日志定位需要修订的章节。
