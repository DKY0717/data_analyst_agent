# v0.5 结果正确性黄金基准设计

## 1. 背景与目标

当前 Data Analyst Agent 已能量化 SQL 是否生成、是否通过安全校验、是否执行成功以及危险请求是否被阻断。但 SQL 成功执行并不代表分析结果正确：错误 JOIN、重复计数、业务口径偏差、排序错误和时间范围错误都可能返回看似合理的结果。

v0.5 第一阶段建立独立的结果正确性黄金基准，用人工审核的参考 SQL 和固定业务断言回答：

> Agent 最终返回的分析结果是否真的正确？

本阶段只建设结果正确性评测，不实现 Schema Context Manager。下一阶段将使用本阶段基线判断 Schema 筛选是否降低正确率。

## 2. 范围

### 2.1 第一版黄金 case

第一版选择 10 条核心单轮业务问题：

1. 统计 2024 年每个月的销售额
2. 找出销售额最高的 5 个商品
3. 统计各地区的客户数量
4. 按地区统计 2024 年销售额
5. 统计各商品类别的销售额
6. 分析各商品类别的退款率
7. 计算 2024 年的客单价
8. 分析用户复购率
9. 按支付方式统计销售额
10. 统计每个月订单数

覆盖范围包括普通聚合、时间趋势、Top-N、多表 JOIN、比率和核心业务指标。

### 2.2 不在本阶段实现

- 不实现 Schema Context Manager。
- 不评估多轮会话正确性。
- 不直接支持 PostgreSQL 方言参考 SQL。
- 不把真实 Qwen 正确率直接设为普通 PR 的强制门禁。
- 不使用 LLM 判断两个结果是否语义等价。

## 3. 黄金 case 数据结构

新增独立文件：

```text
backend/evaluation/cases/golden_result_cases.yaml
```

每条 case 至少包含：

```yaml
- id: monthly_sales_2024
  question: 统计 2024 年每个月的销售额
  category: time_series
  reference_sql: |
    SELECT ...
  comparison:
    mode: ordered
    required_columns: [month, sales_amount]
    order_by: [month]
    absolute_tolerance: 0.001
  fixed_assertions:
    row_count: 12
```

字段定义：

- `id`：稳定唯一标识。
- `question`：发送给 AgentGraph 的自然语言问题。
- `category`：`aggregation`、`time_series`、`top_n`、`dimension` 或 `business_metric`。
- `reference_sql`：人工审核的 DuckDB 参考 SQL。
- `comparison.mode`：比较模式。
- `comparison.required_columns`：正确性评测使用的标准列名。
- `comparison.order_by`：需要验证顺序时使用的列。
- `comparison.absolute_tolerance`：数字绝对误差，默认 `0.001`。
- `fixed_assertions`：防止参考 SQL 和种子数据同时发生意外变化的固定断言。

所有参考 SQL 必须显式使用稳定别名，保证比较器不需要猜测列含义。

## 4. 比较语义

### 4.1 比较模式

支持四种模式：

| 模式 | 用途 | 规则 |
|---|---|---|
| `unordered` | 普通维度聚合 | 忽略行顺序，列和值必须匹配 |
| `ordered` | 时间趋势 | 列、值和行顺序必须匹配 |
| `top_n` | Top-N | 行数、顺序、维度值和指标值必须匹配 |
| `scalar` | 客单价、复购率等单值指标 | 必须返回一行，并在容差内匹配 |

### 4.2 值归一化

比较前统一处理：

- `Decimal` 转为稳定数字表示。
- 日期和时间转为 ISO 字符串。
- `None` 保持为空值。
- 列名做大小写不敏感比较，但报告保留原始列名。
- 数字按 `absolute_tolerance` 比较。
- 字符串不做模糊匹配。

不允许通过四舍五入、大范围容差或 LLM 判断隐藏业务口径错误。

### 4.3 失败类型

单条 case 可能产生以下失败原因：

- `agent_execution_failed`
- `reference_guard_blocked`
- `reference_execution_failed`
- `column_mismatch`
- `row_count_mismatch`
- `value_mismatch`
- `order_mismatch`
- `fixed_assertion_failed`
- `unexpected_error`

报告必须指出具体失败类型，并提供有限数量的差异样本。

## 5. 组件设计

### 5.1 ReferenceQueryRunner

职责：

1. 使用现有 SQL Guard 校验 `reference_sql`。
2. 只执行已通过 Guard 的参考 SQL。
3. 返回标准化 columns、rows 和错误信息。

参考 SQL 不允许绕过现有安全治理。Guard 阻断或执行失败都视为黄金 case 配置错误。

### 5.2 ResultComparator

职责：

1. 归一化 Agent 结果和参考结果。
2. 按比较模式验证结构、数值和顺序。
3. 执行固定断言。
4. 返回稳定的比较结果和有限差异样本。

ResultComparator 不访问数据库、不调用 Qwen，能够使用单元测试完全验证。

### 5.3 ResultCorrectnessEvaluator

数据流：

```text
golden_result_cases.yaml
  -> AgentGraph 执行用户问题
  -> ReferenceQueryRunner 执行 reference_sql
  -> ResultComparator 比较结果
  -> 汇总正确性指标
```

行为要求：

- 支持注入 Agent runner、ReferenceQueryRunner 和 case 文件。
- 单条 case 失败后继续执行其他 case。
- Agent 执行失败直接计为该 case 不正确。
- 不修改现有 `EvaluationRunner` 的流程稳定性指标。

### 5.4 CorrectnessReportWriter

同时输出中文 Markdown 和 JSON：

```text
result-correctness-evaluation-<timestamp>.md
result-correctness-evaluation-<timestamp>.json
```

Markdown 包含：

- 总体结果正确率
- 列结构匹配率
- 结果值匹配率
- 排序匹配率
- 核心业务指标命中率
- 参考 SQL 与固定断言状态
- 每条失败 case 的失败类型和有限差异样本

报告不得输出完整大结果集，单项差异样本默认最多 5 条。

## 6. 指标定义

单条 case 至少输出：

```text
case_id
category
agent_execution_success
reference_guard_passed
reference_execution_success
columns_matched
row_count_matched
values_matched
order_matched
fixed_assertions_matched
result_correct
failure_types
diff_samples
```

汇总指标：

| 指标 | 定义 |
|---|---|
| `result_correctness_rate` | `result_correct=true` 的 case 比例 |
| `column_match_rate` | 列结构匹配比例 |
| `value_match_rate` | 结果值匹配比例 |
| `order_match_rate` | 需要顺序比较的 case 中顺序匹配比例 |
| `business_metric_accuracy` | `business_metric` case 的正确比例 |
| `reference_guard_pass_rate` | 参考 SQL 通过 SQL Guard 的比例 |
| `reference_execution_success_rate` | 参考 SQL 执行成功比例 |
| `fixed_assertion_pass_rate` | 固定断言通过比例 |

空分母使用稳定数值 `0.0`，不返回 `NaN`。

## 7. 固定断言

固定断言用于发现参考 SQL、种子数据或业务口径被意外修改。第一版至少为以下 case 配置固定断言：

- 2024 年每月销售额：月份数量。
- 销售额最高的 5 个商品：行数为 5。
- 各地区客户数量：地区数量与客户总数。
- 各商品类别销售额：类别数量。
- 各商品类别退款率：类别数量。
- 2024 年客单价：单值期望和容差。
- 用户复购率：单值期望和容差。
- 每月订单数量：月份数量与订单总数。

固定值必须由当前固定随机种子的种子数据库计算并人工审核后写入 YAML。

## 8. 错误处理与安全

- 参考 SQL 必须通过 SQL Guard。
- ReferenceQueryRunner 不接受多个语句或 DDL/DML。
- 单条异常记录为 case 失败，不中断批量评测。
- 日志不输出完整查询结果。
- Markdown 限制差异样本数量。
- JSON 可保存结构化差异，但同样不保存完整大结果集。
- 不提交 `.env`、本地 DuckDB 文件或临时报告。

## 9. CI 与真实 Qwen 策略

普通 CI：

- 运行黄金 case 文件结构测试。
- 运行 ReferenceQueryRunner、ResultComparator、Evaluator 和 ReportWriter 的确定性测试。
- 使用测试数据库验证 10 条参考 SQL、固定断言和安全校验全部通过。
- 不调用真实 Qwen。

手动真实 Qwen workflow：

- 增加 Result Correctness Evaluation。
- 与 NL2SQL、Intent 和 SQL Repair 报告一起上传 artifact。
- 第一版记录基线，不将真实正确率作为普通 CI 强制门禁。

后续 Schema Context Manager 必须满足：

- 结果正确率不得低于 v0.5 基线。
- 同时证明 Schema Token、平均 LLM Token 或耗时下降。

## 10. 验收标准

### 10.1 确定性工程门禁

必须全部达到 100%：

- 10 条黄金 case 均具有合法参考 SQL和明确比较策略。
- 参考 SQL Guard 通过率。
- 参考 SQL 执行成功率。
- 固定断言通过率。
- 比较器单元测试通过。
- 中文 Markdown/JSON 报告生成成功。
- Secret Scan、后端测试和前端构建通过。

### 10.2 首轮真实 Qwen 最低目标

- 结果正确率 `>= 80%`
- 列结构匹配率 `>= 90%`
- 核心业务指标命中率 `>= 75%`
- 参考 SQL与固定断言正确率 `100%`

若真实结果未达到目标，不降低比较标准。应分析失败 case，改进语义层、提示词或 SQL 生成策略后重新评测。

## 11. 后续路线

完成本阶段后进入 Schema Context Manager：

1. 按问题召回候选表和业务指标。
2. 补全 JOIN 路径所需表。
3. 只向 Qwen 发送相关字段与语义定义。
4. 使用本阶段结果正确性基线证明准确率不下降。
5. 对比 Token、耗时和表选择准确率。
