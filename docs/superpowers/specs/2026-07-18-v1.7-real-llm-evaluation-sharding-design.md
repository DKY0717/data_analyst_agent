# v1.7 真实 LLM 评测分片与证据恢复设计

## 1. 背景

当前真实模型工作流把确定性测试、4 条 smoke、65 条 NL2SQL、6 条 SQL Repair、10 条结果正确性评测和最终质量门禁放在一个 45 分钟 job 中串行执行。MiMo 实测 4 条 smoke 约需 36 分钟，工作流随后在 NL2SQL 阶段被 job 超时取消。由于 NL2SQL 报告只在整批结束后写入，已完成用例也没有形成可复用证据。

本设计解决的是评测编排和证据完整性问题，不调整业务 Agent、Prompt、Schema、模型参数或质量阈值。

## 2. 目标与成功标准

### 2.1 目标

- 把长耗时真实评测拆成可隔离、可定位、可独立重跑的固定分片。
- 将整个 workflow 的 MiMo 最大并发限制为 2，优先保证稳定性和供应商友好性。
- 每完成一条用例就保存一次原子 checkpoint，分片失败时保留已经完成的脱敏结果。
- 最终质量门禁只消费经过完整性校验的全集报告。
- 分片缺失、重复、来源不一致或 case 集变化时 fail closed，同时留下机器可读的诊断证据。

### 2.2 成功标准

- 65 条 NL2SQL、6 条 Repair、10 条 Correctness 用例都能被确定性分配且恰好覆盖一次。
- 同一类真实评测任意时刻最多运行 2 个分片；三类真实评测不并行重叠。
- 分片步骤超时后，后续 artifact 上传步骤仍有时间上传最近一次 checkpoint。
- 汇总器拒绝缺片、重复 case、未知 case、错误分片编号以及不同提交、provider、model 或 case 文件哈希的报告。
- 只有三类完整报告都生成后，质量门禁才可以判定通过。
- 基础 CI 不调用真实模型，新增逻辑可以通过确定性单元测试验证。

## 3. 方案选择

采用 GitHub Actions 矩阵分片，不采用单进程异步并发或多个手工 workflow。

矩阵分片把故障限制在单个 runner，天然支持 `max-parallel` 和分片 artifact；单进程并发仍保留单 job 的超时爆炸半径，并可能污染数据库连接、缓存和调用追踪；多个手工 workflow 则增加错 SHA、漏 artifact 和人工汇总风险。

## 4. 总体架构

工作流按以下阶段运行：

1. `preflight`：准备数据库，运行确定性后端测试、真实模型 smoke 和确定性 Intent 评测。
2. `nl2sql-shards`：13 个固定分片，矩阵 `max-parallel: 2`。
3. `repair-shards`：3 个固定分片，等待全部 NL2SQL 分片结束后启动，矩阵 `max-parallel: 2`。
4. `correctness-shards`：5 个固定分片，等待全部 Repair 分片结束后启动，矩阵 `max-parallel: 2`。
5. `quality-gate`：下载所有 artifact，严格合并三类真实报告，再运行确定性 Grounding、Permission、质量门禁和严格安全审计。

如果某个真实分片失败，后续真实评测类型仍继续运行，以获得尽可能完整的诊断证据；最终汇总 job 使用 `always()` 语义启动，并根据完整性校验决定质量门禁是否可以执行。预检失败时不继续消耗真实模型额度，但仍上传预检和审计证据。

## 5. 分片规则

### 5.1 固定分片

每类用例按 YAML 中的原始顺序编号，并使用 `position % shard_count == shard_index` 做 round-robin 分片。该规则满足：

- 同一 case 文件和 shard 数得到稳定结果；
- 不依赖 Python 随机哈希；
- 相邻用例分散到不同分片，降低局部慢 case 集中风险；
- 合并时可以按原始 case 顺序恢复最终报告。

分片数量固定为：

| 评测类型 | 用例数 | 分片数 | 每片用例数 |
|---|---:|---:|---:|
| NL2SQL | 65 | 13 | 5 |
| SQL Repair | 6 | 3 | 2 |
| Result Correctness | 10 | 5 | 2 |

分片索引从 0 开始。CLI 必须拒绝 `shard_count <= 0`、负索引或 `shard_index >= shard_count`。

### 5.2 并发与时间预算

- 三个矩阵分别设置 `max-parallel: 2`。
- 三类矩阵通过依赖关系串行衔接，保证 workflow 级真实模型并发不超过 2。
- 单个分片 job 上限为 90 分钟。
- 真实评测步骤上限为 75 分钟，至少预留 15 分钟执行 `always()` checkpoint/artifact 步骤。
- 不在 Python 进程内并发 case；每个分片内部顺序执行，避免共享状态污染。

## 6. 共享分片与 checkpoint 组件

新增一个与具体评测类型无关的分片组件，职责限制为：

- 校验分片参数并选出当前分片的 case。
- 计算 case 文件 SHA-256。
- 顺序调用评测器已有的 `evaluate_case`。
- 每完成一条 case 后重新计算当前分片 summary，并原子写入 checkpoint。
- 评测完成时把 `complete` 标记为 `true`。

原子写入使用同目录临时文件、flush 后 `Path.replace`，避免进程被取消时留下半个 JSON。checkpoint 输出路径由 CLI 显式传入，workflow 使用稳定文件名，不依赖时间戳查找。

三类评测器保留现有的无参数完整运行方式，同时增加统一参数：

- `--shard-index`
- `--shard-count`
- `--checkpoint-output`
- `--case-file`（现有 Correctness 参数保留并统一行为）

分片模式只改变选择哪些 case 和何时写报告，不改变单 case 的执行、脱敏和指标定义。

## 7. 分片报告契约

checkpoint JSON 顶层继续保留 `summary` 和 `results`，新增 `shard` 元数据：

```json
{
  "shard": {
    "schema_version": 1,
    "suite": "nl2sql",
    "head_sha": "<git sha>",
    "provider": "mimo",
    "model": "mimo-v2.5-pro",
    "case_file_sha256": "<sha256>",
    "shard_index": 0,
    "shard_count": 13,
    "expected_case_ids": ["monthly_sales_2024"],
    "completed_case_ids": ["monthly_sales_2024"],
    "complete": false
  },
  "summary": {},
  "results": []
}
```

`EVALUATION_HEAD_SHA`、`LLM_PROVIDER` 和 `QWEN_MODEL` 从现有 workflow 环境读取。分片报告不会增加 API Key、原始供应商响应、`.env` 内容或完整数据库结果。现有报告已经允许的问题、SQL、有限差异和脱敏错误类型保持不变。

## 8. 严格合并

新增汇总命令分别处理 NL2SQL、Repair 和 Correctness 分片。合并过程先完成全部校验，再写最终报告：

1. 从当前 case YAML 读取权威 case ID 顺序并计算文件哈希。
2. 要求分片编号集合正好等于 `0..shard_count-1`。
3. 要求所有分片 `complete=true`，且 `completed_case_ids` 等于本分片的 `expected_case_ids`。
4. 要求全部分片具有一致且符合 workflow 期望的 SHA、provider、model、suite、schema version、shard count 和 case 文件哈希。
5. 拒绝重复、未知、缺失或顺序不一致的 case 结果。
6. 按权威 YAML 顺序恢复结果，并调用该评测类型既有 summary 计算逻辑重新计算全集指标。
7. 使用既有正式报告写入器生成质量门禁消费的 JSON 和 Markdown。

任何校验失败都写出独立的 `merge-diagnostics.json`，列出稳定错误代码和缺失/重复 ID，然后以非零状态退出。诊断信息不包含凭据或原始供应商错误文本。

## 9. GitHub Actions 证据流

每个 job 都重新 checkout、安装后端依赖并在需要数据库时准备独立评测库。真实模型 job 使用现有 provider credential 选择逻辑，密钥只写入被 GitHub 掩码的环境变量。

Artifact 采用稳定命名：

- `real-llm-preflight-<run_id>`
- `real-llm-nl2sql-<run_id>-shard-<index>`
- `real-llm-repair-<run_id>-shard-<index>`
- `real-llm-correctness-<run_id>-shard-<index>`
- `real-llm-quality-gate-<run_id>`

上传步骤使用 `if: always()`。最终 job 下载当前 run 的上述 artifact，不从其他 workflow run 或提交复用结果。汇总时显式传入当前 `github.sha`、provider 和 model，防止混用证据。

## 10. 失败处理

- 单 case 异常：沿用现有结构化失败结果并继续当前分片。
- 分片步骤超时：进程被终止，最近一次完整 checkpoint 保留；上传步骤继续执行。
- checkpoint 不存在：artifact 可以为空或只含运行元数据，最终汇总报告缺片。
- 任一分片不完整：对应正式全集报告不生成，质量门禁不得运行或通过。
- 质量门禁输入缺失：严格安全审计继续运行并记录缺失真实报告。
- 供应商认证、欠费、限流或网络错误：继续只保留既有清洗后的状态、code/type 或异常类型，不保存原始 message。

## 11. 测试设计

### 11.1 单元测试

- 分片参数边界和 round-robin 稳定性。
- 全部分片恰好覆盖一次且合并恢复 YAML 原始顺序。
- 每完成一个 case 都写 checkpoint，最后一次标记完整。
- 写入失败或中断不会破坏上一个有效 JSON。
- 三类评测器的分片 CLI 使用相同契约。
- 汇总成功时重新计算全集 summary，而不是信任分片 summary。
- 缺片、重复 case、未知 case、错误分片编号、未完成分片和元数据不一致均失败并生成诊断。

### 11.2 工作流契约测试

- 预检、三个矩阵和最终汇总 job 均存在，依赖顺序正确。
- 三个矩阵都设置 `max-parallel: 2`。
- 分片 job/step 的 90/75 分钟预算存在。
- checkpoint 和 artifact 使用稳定路径及 `always()` 上传。
- 最终汇总使用当前 SHA/provider/model 做严格校验。
- 严格安全审计即使上游失败也执行。

### 11.3 回归验证

- 分片与汇总专项 pytest。
- 评测报告、质量门禁、安全审计和 workflow 契约测试。
- 后端全量 pytest、Ruff、Secret Scan 和 `git diff --check`。
- 基础 GitHub CI 必须全绿。

真实 MiMo workflow 会发送固定测试问题、Schema 和语义上下文给第三方模型，因此代码完成后必须再次取得用户明确授权，才能在新 SHA 上触发。

## 12. 非目标

- 不缩减或抽样 65/6/10 的固定用例集。
- 不改变 Prompt、模型温度、重试次数、请求超时或质量阈值。
- 不在真实评测中引入跨分片缓存或共享数据库。
- 不自动重试失败分片，避免不可控重复计费；失败后由同一 workflow 的完整性证据指导人工重跑。
- 不把仅增加单 job 超时作为解决方案。

## 13. 验收条件

- 新增确定性测试全部通过，后端原有测试无回归。
- workflow 静态契约证明最大并发、依赖、时间预算、上传和严格汇总符合本设计。
- 本地构造的完整分片集能生成三份正式全集报告并进入质量门禁。
- 本地删除、篡改或重复任一分片时，汇总稳定失败并生成可审计诊断。
- 新提交基础 CI 全绿。
- 获得单独授权后，新 SHA 的真实 MiMo workflow 能保留每片证据，并且只有完整 81 条真实评测结果才可能通过最终质量门禁。
