# v1.0 Permission Policy Externalization and Row-Level Filter Design

## Summary

v1.0 upgrades the current Data Permission Guard from hard-coded role policies to an external, validated policy file and adds deterministic row-level filtering before SQL execution.

The current project already has a strong execution chain:

```text
SQL Generator -> SQL Guard -> Data Permission Guard -> Query Runner
```

v1.0 keeps that security boundary. The LLM still never decides permissions. The Data Permission Guard continues to inspect the final SQL AST after SQL Guard sanitization, then either blocks the query or returns an authorized SQL string. If a role has row-level filters, the guard rewrites the SQL with those filters before `QueryRunner` executes it.

## Goals

- Move table and column role policies out of `backend/app/security/data_permission.py`.
- Add a default YAML policy file tracked in the repo.
- Allow operators and tests to override the policy path through `DATA_PERMISSION_POLICY_PATH`.
- Preserve current admin, analyst, support table and column behavior.
- Add row-level filters that are applied deterministically by backend SQL AST rewriting.
- Record row-filter evidence in the existing audit report without leaking the full policy.
- Keep permission failures fail-closed: invalid policy, ambiguous SQL, unsupported rewrite, and unauthorized access must block execution.

## Non-Goals

- No frontend policy editor in this version.
- No database-backed policy management table.
- No dynamic IAM integration.
- No LLM-generated policy logic.
- No broad ABAC engine. v1.0 only supports role-based table, column, and row-filter rules.

## Policy File

Default file:

```text
backend/app/security/data_permissions.yaml
```

Environment override:

```text
DATA_PERMISSION_POLICY_PATH=/absolute/or/project/relative/path/to/data_permissions.yaml
```

If the env var is not set, the loader uses the default file. If the configured file is missing, malformed, or semantically invalid, policy loading fails and authorization fails closed with `block_permission_policy_error`.

### YAML Shape

```yaml
version: 1
roles:
  admin:
    tables:
      "*":
        columns: ["*"]

  analyst:
    tables:
      regions:
        columns: ["*"]
      customers:
        columns: ["customer_id", "gender", "age", "region_id", "register_date"]
      categories:
        columns: ["*"]
      products:
        columns: ["*"]
      orders:
        columns: ["*"]
        row_filter:
          expression: "customer_id IN (SELECT customer_id FROM customers WHERE region_id IN (1, 2))"
          rule_id: "row_filter_region_scope"
      order_items:
        columns: ["*"]
      payments:
        columns: ["payment_id", "order_id", "payment_method", "payment_status", "paid_at"]
      refunds:
        columns: ["refund_id", "order_id", "refund_reason", "refund_date"]

  support:
    tables:
      regions:
        columns: ["*"]
      categories:
        columns: ["*"]
      products:
        columns: ["*"]
      orders:
        columns: ["*"]
      order_items:
        columns: ["*"]
```

Policy semantics:

- `tables["*"].columns: ["*"]` grants all table and column access.
- Table-level absence denies the table.
- Column-level absence denies the column.
- Multiple roles merge by union for table and column access.
- Row filters merge by conjunction when more than one applicable filter targets the same query.
- Row filters are only applied to tables referenced by the final SQL.
- Row filter expressions may reference the target table's local columns and may use read-only subqueries for schema relationships that are not denormalized on the target table. In the e-commerce schema, `orders` is region-scoped through `orders.customer_id -> customers.region_id`.

## Components

### `PermissionPolicyLoader`

New module:

```text
backend/app/security/permission_policy.py
```

Responsibilities:

- Load YAML using PyYAML, already present in `backend/requirements.txt`.
- Resolve default path from `settings.BASE_DIR / "backend" / "app" / "security" / "data_permissions.yaml"` or use `DATA_PERMISSION_POLICY_PATH`.
- Validate policy structure into small dataclasses:
  - `PermissionPolicy`
  - `RolePolicy`
  - `TablePolicy`
  - `RowFilterPolicy`
- Normalize role, table, and column names to lowercase.
- Reject empty policies, unknown top-level versions, invalid column lists, invalid row filter expressions, and row filters without a `rule_id`.
- Return a stable policy object to `DataPermissionGuard`.

### `DataPermissionGuard`

Existing module:

```text
backend/app/security/data_permission.py
```

Changes:

- Replace hard-coded `ROLE_POLICIES` with policy loader output.
- Keep current table and column extraction behavior.
- Return an expanded `DataPermissionResult`:

```python
authorized_sql: str
row_filters_applied: list[dict[str, Any]]
```

- If no row filter applies, `authorized_sql` equals the SQL passed into `authorize`.
- If filters apply, `authorized_sql` becomes the rewritten SQL.
- Audit events include `row_filters_applied`, `authorized_sql_changed`, referenced tables, and checked columns.

### AgentGraph

Existing module:

```text
backend/app/agents/graph.py
```

Changes:

- `_authorize_sql` must use `result.authorized_sql`.
- If permission succeeds, `validated_sql` is updated to the authorized SQL before execution.
- Audit report `final_sql` therefore reflects the SQL that was actually executed.

## Row-Level Filter Rewriting

Row filtering happens after table and column authorization passes.

The guard parses both:

- the final SQL statement
- each policy `row_filter.expression`

with `sqlglot` using the DuckDB dialect.

### Supported Queries

v1.0 supports row-filter injection for:

- `SELECT ... FROM table`
- `SELECT ... FROM table alias`
- `SELECT ... FROM table JOIN ...`
- existing `WHERE` clauses, by combining with `AND`
- CTEs, as long as the physical table reference can be found in the parsed tree

### Unsupported Queries

The guard blocks with `block_row_filter_unsupported` if a row filter should apply but the SQL cannot be rewritten safely. Examples:

- table references that cannot be mapped to a physical table
- ambiguous aliases for a filtered table
- row filter expressions that cannot be parsed as a predicate
- subqueries inside row filters that are not read-only `SELECT` statements
- row filter expressions that reference unknown physical tables or columns
- SQLGlot parse or serialization failures during rewrite

### Rewrite Rule

For every referenced physical table with an applicable filter:

1. Resolve the table alias used in the query.
2. Parse the row filter expression.
3. Qualify unqualified columns that belong to the target table with the table alias when needed.
4. Preserve safe read-only subqueries inside the configured filter expression.
5. Combine the filter with the nearest SELECT scope that reads the table.
6. Preserve existing SQL clauses, including `LIMIT` injected by SQL Guard.

Example:

```sql
SELECT SUM(total_amount) AS sales
FROM orders
WHERE order_date >= DATE '2024-01-01'
LIMIT 1000
```

For analyst:

```sql
SELECT SUM(total_amount) AS sales
FROM orders
WHERE order_date >= DATE '2024-01-01'
  AND customer_id IN (
    SELECT customer_id
    FROM customers
    WHERE region_id IN (1, 2)
  )
LIMIT 1000
```

The implementation should prefer `sqlglot` AST APIs over string concatenation. String concatenation is only acceptable inside policy fixture text and expected SQL assertions.

## Audit Behavior

Allowed query with no row filters:

- `stage`: `authorization`
- `status`: `success`
- `rule_id`: `null`
- details include user identity, roles, referenced tables, checked columns, and an empty `row_filters_applied`.

Allowed query with row filters:

- `stage`: `authorization`
- `status`: `success`
- `rule_id`: `row_filter_applied`
- details include:
  - `row_filters_applied`: list of `{table, rule_id}`
  - `authorized_sql_changed`: `true`
  - referenced tables and checked columns

Blocked query:

- existing rule ids remain stable where possible:
  - `block_unauthorized_table`
  - `block_unauthorized_column`
  - `block_ambiguous_column`
- new policy and row filter failures use:
  - `block_permission_policy_error`
  - `block_row_filter_unsupported`

The audit event must not dump the full policy or raw configured policy file.

- Analyst order queries are rewritten with row-level filters before execution.

## Risks and Mitigations

- **Risk:** SQL rewrite accidentally changes query semantics.
  - Mitigation: restrict v1.0 rewrite support to SELECT scopes that sqlglot can parse and serialize safely; fail closed otherwise.
- **Risk:** Row filters expose policy internals in audit details.
  - Mitigation: audit only table and rule id, not full expressions or policy file content.
- **Risk:** Policy override breaks local development.
  - Mitigation: default tracked policy mirrors current hard-coded behavior, and override path is optional.
- **Risk:** Multi-role row filters become confusing.
  - Mitigation: v1.0 uses conjunction for filters on the same table and documents that behavior explicitly.
