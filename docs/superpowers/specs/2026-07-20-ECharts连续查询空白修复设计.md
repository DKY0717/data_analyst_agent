# ECharts 连续查询空白修复设计

## 问题

用户首次得到可视化结果后再次发起查询，前端会先把 `result` 设为 `null`。`ChartPanel` 的 `v-if` 因此销毁当前图表容器，但已有 ECharts 实例只执行 `clear()`，仍绑定到已经脱离文档的旧 DOM。

新结果返回时，新图表容器已经创建，但 `chartInstance` 仍为真值，组件跳过 `echarts.init(newDom)`，随后把配置写入旧实例。页面能显示图表类型按钮，却看不到图形；切换图表类型也无法恢复。

## 方案比较

### 方案一：销毁失效实例并按当前 DOM 重建（采用）

当图表配置或容器不存在时，对已有实例执行 `dispose()` 并将引用设为 `null`。渲染时额外确认实例绑定的 DOM 是否等于当前 `chartRef`；不一致则销毁旧实例并重新初始化。

优点是生命周期边界清晰，可以处理连续查询、条件渲染和未来的容器替换。代价只是结果切换时重新初始化一次 ECharts，影响可忽略。

### 方案二：将 `v-if` 改为 `v-show`

始终保留同一个图表容器，只切换显示状态，可以避免 DOM 被替换。但隐藏期间容器尺寸可能为零，恢复时仍需要额外 resize 处理，而且空状态与画布会同时长期存在。

### 方案三：通过 `key` 强制重建整个 `ChartPanel`

每次结果变化都重新挂载组件。实现表面简单，但会销毁主题监听、窗口监听和动态模块状态，重建范围大于问题本身。

## 实现设计

在 `ChartPanel.vue` 中增加一个集中释放函数：

- 若实例存在，调用 `dispose()`。
- 将 `chartInstance` 置为 `null`，防止后续误用旧实例。
- 图表配置或容器为空时调用该函数。
- 当前实例绑定 DOM 与 `chartRef` 不一致时，先释放再初始化。
- 组件卸载时复用该函数，保持唯一的资源释放路径。

不修改图表类型识别、数据转换、颜色、布局或后端响应格式。

## 错误边界

本次只解决确定的旧 DOM 生命周期缺陷。动态模块加载失败、浏览器 Canvas 不可用等异常仍由现有前端错误机制处理，不在本次改动中新增 UI 错误面板。

## 测试

新增 `ChartPanel` 组件测试并模拟 ECharts：

1. 第一次传入有效结果时初始化并绘图。
2. 将结果切换为 `null` 时销毁旧实例。
3. 再传入有效结果时在新 DOM 上创建新实例并绘图。
4. 切换图表类型时向当前实例更新配置。
5. 组件卸载时释放实例。

测试不加载真实 Canvas，也不请求后端或真实模型。

## 验收标准

- 连续两次分析后第二次图表正常渲染。
- 图表按钮切换作用于当前可见画布。
- 旧 ECharts 实例不会继续持有已销毁 DOM。
- 新组件测试、前端全量单元测试和生产构建通过。
