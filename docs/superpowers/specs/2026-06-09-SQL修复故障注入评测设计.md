# SQL Repair 故障注入评测设计

## 1. 背景

现有 32 条 NL2SQL 真实评测能够量化自然语言生成、正常执行与危险请求阻断能力，但其中 5 条“修复类提示题”均被 Qwen Plus 在第一次生成时直接转换为可执行 DuckDB SQL，没有稳定触发 SQL Repair。

因此，现有报告中的 SQL 修复成功率不能代表真实修复能力。需要新增一套独立、确定性的故障注入评测，直接把固定错误 SQL、真实数据库错误和 Schema 交给 SQL Repair，并验证修复后的 SQL 是否安全、可执行且保持原查询意图。

## 2. 目标

1. 使用固定错误 SQL 稳定触发 SQL Repair。
2. 使用 DuckDB 实际执行错误 SQL，获取真实数据库错误信息。
3. 使用真实 Qwen Plus 修复错误 SQL。
4. 修复结果必须通过 SQL Guard 后才能执行。
5. 独立统计 SQL Repair 的输出、Guard、执行和意图保持能力。
6. 输出 JSON 与中文 Markdown 报告，便于版本比较和面试展示。

## 3. 非目标

1. 不修改正常 AgentGraph 流程。
2. 不为了触发修复而故意降低 SQL Generator 质量。
3. 不把危险 SQL 交给 Repair。
4. 不使用 Mock 结果作为真实 Repair 基线。
5. 第一版不引入 LLM Judge 判断语义等价性。

## 4. 评测架构

```text
固定 Repair Case
  → SQL Guard 校验错误 SQL
  → DuckDB 执行错误 SQL并获取真实错误
  → Qwen Plus SQL Repair
  → SQL Guard 校验修复 SQL
  → DuckDB 执行修复 SQL
  → 规则化意图检查
  → Repair 报告
```

错误 SQL 必须是安全的 `SELECT` 或 `WITH`，且能够通过 Guard。只有数据库执行失败后，才允许调用 SQL Repair。

修复后的 SQL 必须再次经过 Guard。Guard 拒绝时立即记录失败，禁止执行。

## 5. 组件设计

### 5.1 Repair Case 文件

路径：

`backend/evaluation/cases/sql_repair_cases.yaml`

每条 case 包含：

```yaml
- id: wrong_order_amount_column
  description: 修复不存在的订单金额字段
  original_sql: SELECT SUM(revenue) AS sales_amount FROM orders
  expected_tables: [orders]
  required_sql_fragments: [total_amount]
  forbidden_sql_fragments: [revenue]
```

第一版包含 6 条确定性错误：

1. 不存在字段：`orders.revenue`。
2. 不存在表：`order_detail`。
3. 缺失 JOIN：从 `orders` 直接查询 `customer_name`。
4. 错误日期函数：`DATE_FORMAT`。
5. 错误季度格式：`strftime(..., '%q')`。
6. 错误地区字段：`regions.province_name`。

所有错误 SQL 都是安全查询，不包含 DDL、DML、文件访问、系统表或危险函数。

### 5.2 RepairEvaluationRunner

路径：

`backend/evaluation/repair_evaluator.py`

职责：

1. 加载 Repair case。
2. Guard 校验原始错误 SQL。
3. 执行错误 SQL并确认确实失败。
4. 调用 `sql_repair_agent.repair()`。
5. Guard 校验修复 SQL。
6. 执行修复 SQL。
7. 根据固定片段规则检查意图保持。
8. 汇总指标并调用 Repair 报告写入器。

单条结果结构：

```python
{
    "case_id": str,
    "description": str,
    "original_sql": str,
    "original_guard_passed": bool,
    "failure_injected": bool,
    "original_error": str | None,
    "repair_output_success": bool,
    "repaired_sql": str,
    "repair_reason": str,
    "repaired_guard_passed": bool,
    "execution_success": bool,
    "intent_preserved": bool,
    "required_fragments_met": bool,
    "forbidden_fragments_absent": bool,
    "execution_time_ms": int,
    "error": str | None,
}
```

### 5.3 意图保持检查

第一版使用确定性规则，不使用第二次 LLM 调用：

- `expected_tables`：修复 SQL 必须包含预期表。
- `required_sql_fragments`：修复 SQL 必须包含这些片段。
- `forbidden_sql_fragments`：修复 SQL 不得包含这些片段。

所有 SQL 文本比较均忽略大小写。

`intent_preserved` 仅在以上三类规则全部满足时为真。

### 5.4 Repair 指标

Repair 汇总报告包含：

- `total_cases`
- `failure_injection_rate`
- `repair_output_success_rate`
- `repaired_guard_pass_rate`
- `repair_execution_success_rate`
- `intent_preservation_rate`
- `end_to_end_repair_success_rate`
- `average_execution_time_ms`

端到端修复成功要求同时满足：

1. 错误注入成功。
2. Repair 返回结构化结果。
3. 修复 SQL 通过 Guard。
4. 修复 SQL 执行成功。
5. 意图保持检查通过。

### 5.5 Repair 报告

路径：

`backend/evaluation/repair_report_writer.py`

输出目录：

`backend/evaluation/reports/`

文件命名：

- `sql-repair-evaluation-<timestamp>.json`
- `sql-repair-evaluation-<timestamp>.md`

Markdown 报告包含：

1. 汇总指标。
2. Case 结果表。
3. 原始 SQL、真实错误、修复 SQL、修复原因与最终错误摘录。

## 6. 错误处理

1. 原始 SQL 未通过 Guard：记录 `original_guard_passed=false`，不执行、不调用 Repair。
2. 原始 SQL 意外执行成功：记录 `failure_injected=false`，不调用 Repair。
3. Repair 调用异常：记录 `repair_output_success=false`，继续下一条 case。
4. 修复 SQL 未通过 Guard：记录失败，不执行。
5. 修复 SQL 执行失败：保留数据库错误，继续下一条 case。
6. 任意单条 case 失败不得中断整批评测。

## 7. 测试策略

### 单元测试

使用 fake Repair Agent、fake Guard 和 fake Query Runner，覆盖：

1. Case 加载。
2. 原始 SQL 未通过 Guard时跳过。
3. 原始 SQL 意外成功时跳过。
4. Repair 输出成功。
5. 修复 SQL Guard 拒绝。
6. 修复 SQL 执行失败。
7. 意图保持规则。
8. 汇总指标。
9. Markdown 与 JSON 报告生成。

### 真实验收

使用 Qwen Plus 运行全部 6 条固定 Repair case：

```powershell
$env:PYTHONPATH='E:\Code\PythonCode\data_analyst_agent\backend'
python -m evaluation.repair_evaluator
```

真实基线报告必须包含每条 case 的原始数据库错误和最终修复结果。

## 8. 安全约束

1. 原始错误 SQL 必须先通过 Guard。
2. Repair 只接受数据库执行失败的安全 SQL。
3. 修复 SQL 必须重新通过 Guard。
4. 评测 case 文件禁止加入 DDL、DML、系统表、文件函数和外部数据库操作。
5. API Key 仅从本地 `.env` 加载，报告不得包含密钥或请求头。

## 9. 验收标准

1. 新增 6 条固定 Repair case。
2. 单元测试覆盖主要分支并通过。
3. 完整后端测试通过。
4. 真实 Qwen Plus Repair 评测能够完整运行，单条失败不终止整批。
5. 生成 JSON 与中文 Markdown 报告。
6. 报告明确展示端到端修复成功率与意图保持率。
7. 不修改正常 AgentGraph 的生产行为。
